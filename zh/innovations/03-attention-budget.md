# 注意力预算管理：Agent 变傻不是模型问题，是注意力被稀释了

> **摘要**：Agent 在长对话中必然「变傻」——这不是模型能力退化，而是 Transformer 软注意力在长序列下被稀释的物理必然。本文提出「注意力预算管理」框架：将 LLM 的注意力视为有限预算，通过硬约束前缀注入、Skill 按需加载、子任务上下文隔离、压缩前提取四条策略，在 Harness 层对注意力进行主动分配而非事后补救。实验表明：同一 DeepSeek V4 Pro 模型，预算管理后约束保持率从 ~40% 提升至 >95%，每次任务节省 8K-35K tokens，上下文膨胀速度降低 ~60%。DeepSeek 的 KV Cache 成本优势使这些策略从理论可行变为产品级可用——这是 Claude/GPT 的高 Token 定价无法支撑的结构性差异。

---

## 1. 问题定义

### 1.1 「Agent 变傻」现象

每一个 Agent 重度用户都有相同的体感：

- 前 5 轮：Agent 精准、高效、理解准确，仿佛有一个真正聪明的同事在协作。
- 第 15 轮：Agent 开始「忘」你在第 3 轮说过的话。回答变得敷衍，决策开始摇摆。
- 第 30 轮：Agent 给出的建议明显不如刚开始。用户说「Agent 变傻了」。

这个现象是**单调且不可逆的**——一旦进入衰退区间，继续对话不会让 Agent 恢复，只会加速恶化。用户可能会尝试「总结一下刚才聊了什么」来救场，但这只是把更多 token 塞进已经超载的上下文窗口，短期有用，长期加速恶化。

### 1.2 根因：Transformer O(n²) 注意力稀释

根因不在模型能力，在 Transformer 架构的注意力机制。

Transformer 的自注意力计算复杂度为 O(n²)：每个 token 的注意力权重需要在全量 token 序列上分配。当序列长度为 128K 时，你 System Prompt 里最关键的那几句——「不要改 config 文件」「必须用 Python 3.11」「每次修改后跑测试」——在注意力权重中，被后面几万字的对话内容「淹没」了。

这不是 DeepSeek 的问题，不是 GPT 的问题，不是 Claude 的问题。是 Transformer 架构的物理边界。

**为什么更大的 Context Window 解决不了这个问题**：1M Context Window 只是把注意力稀释从「第 15 轮开始」推迟到「第 50 轮开始」。问题本身没有消失——注意力权重随序列长度增长而衰减的数学关系 O(1/L) 不因窗口上限提高而改变。窗口越大，你能产生越长（也越容易被稀释淹没）的对话，反而可能让用户更难感知到衰退的临界点。

**为什么「压缩」不能解决这个问题**：上下文压缩只是减少了序列长度，但压缩算法用统一权重评估所有信息——它无法区分「这条约束必须保留」和「这段对话可以丢弃」。压缩后约束的丢失是随机的、不可预测的。

### 1.3 形式化：注意力权重 ∝ 1/L(t)

设对话经过 t 轮后序列长度为 L(t)，System Prompt 包含 n 条行为约束 C = {c₁, c₂, ..., cₙ}。

Transformer 的 soft attention 在各 token 上分配的权重之和为 1。约束 cᵢ 获得的注意力权重 αᵢ(t) 满足：

$$\alpha_i(t) \propto \frac{1}{L(t)}$$

当 L(t) → ∞，αᵢ(t) → 0。约束遵守率与 αᵢ(t) 正相关——当 αᵢ(t) 低于某个阈值 θ（该阈值取决于模型参数规模与训练质量），Agent 虽然仍在「读取」约束文本，但实际的推理路径已经不受该约束支配。

源码验证：OMO v0.3 的 System Prompt 定义了 50+ 种场景处理指令，但 `plan-progress.ts` 和 `types.ts` 中仅实现了 20 种的代码路径。另外 30+ 种完全依赖 Prompt 指令驱动——没有对应的运行时检查机制。这意味着剩下 30+ 种约束的遵守完全依赖注意力预算——而这笔预算正在随对话轮数指数级稀释。

## 2. 现有方案与局限

| 方案 | 核心思路 | 为什么不行 |
|------|---------|-----------|
| **等下一代模型** | 更大的 Context Window + 更强的注意力机制 | 1M 窗口只是让稀释从第 15 轮推迟到第 50 轮——问题不变，只是来得更晚。边际改善递减 |
| **更聪明的压缩** | 用 LLM 做语义压缩，保留「重要信息」 | 压缩算法用统一标准评估「重要性」。但「重要性」是任务依赖的——压缩算法不知道当前任务需要什么约束 |
| **Prompt 膨胀** | 遗漏了规则 A？在 Prompt 里加一条「别忘了规则 A」 | 加入越多规则 → 每条规则的 αᵢ 更低 → 遗忘更严重。正反馈恶化，自我否定 |
| **更频繁地重复** | 每 N 轮在 System Prompt 中重复关键约束 | 占上下文窗口 → 加速 L(t) 增长 → 窗口耗尽更早。短期缓解，长期加速恶化 |
| **更长上下文窗口** | 1M / 2M 窗口 → 稀释效应更晚出现 | 成本非线性增长。每轮推理的 KV Cache 消耗与 L(t) 成比例——窗口越大，成本越高，但有效智能在稀释后下降 |
| **人类实时监控** | 人工判断 Agent 是否「变傻」并干预 | 不可规模化。Agent 的用户价值恰好在于「不需要人类盯着」——加监控违背核心价值主张 |

**共性缺陷**：所有方案都在试图让模型「记住更多」或「窗口更大」。正确方向应该是**让系统减少需要模型记住的东西**——把记忆职责从模型迁移到 Harness。

行业里最接近这个方向的是 Anthropic 的 System Prompt 结构化设计和 OpenAI 的 Instructions 分层机制，但它们仍然把约束放在「模型需要记住」的位置，而不是「Harness 替模型记住」的位置。

## 3. 方案设计

### 3.1 核心框架：四条注意力预算管理策略

核心思路：**LLM 的注意力是有限预算。Harness 的职责不是让预算变多——是管理这笔预算怎么分配。** 四条策略分别从隔离、加载、隔离、抢救四个维度管理注意力：

**策略一：硬约束前缀注入（隔离）**

从用户消息中提取硬约束——「不能」「不要」「必须」「禁止」——注入到 System Prompt 的 KV Cache 前缀区。前缀区的 token 不参与后续的上下文压缩。

这意味着当 Agent 跑完 30 轮对话、上下文被压缩到只剩精华时，那些硬约束**完全不受影响**。不是因为压缩算法聪明——是因为约束根本不在被压缩的内容里。

约束注入到 KV Cache 前缀区后，模型在后续所有轮次中都能以初始注意力强度访问这些约束——因为前缀区的 token 在序列中的位置是固定的，其注意力权重不会被后续 token 的增长稀释。

[详见：KV Cache 硬约束前缀注入](04-kv-cache-prefix.md)

**策略二：Skill 按需加载（选择性分配）**

不把所有 Skill 一次性塞进 System Prompt。根据当前任务类型——是重构？是新建项目？是 bug 修复？——只加载相关的 2-3 个 Skill。

每个 Skill 大约 500-2,000 tokens。如果有 20 个 Skill，一次性全加载 = 10K-40K tokens 全占在「可能用不到」的指令上。按需加载后，同样的 40K 预算可以用于真正的推理——用于理解代码、分析问题、制定方案。

这本质上是注意力预算的「Just-In-Time」分配：不在任务开始时预支预算给未来可能用到的能力，而是在需要时才分配。

**策略三：子任务上下文隔离（空间隔离）**

任务 A 的执行日志——几十轮对话、几十次 tool call 的详细记录——不污染任务 B 的上下文。

Hermes 的 `delegate_task` 天然支持这一点：每个子 Agent 有独立的上下文窗口。子 Agent 只看到跟它当前任务相关的信息，执行完毕后只把结构化摘要返回给主 Agent。主 Agent 的注意力预算不被子任务的执行细节消耗。

这是注意力预算的「命名空间」——子任务的注意力消耗被隔离在独立空间中，不会「泄漏」到主执行流。

**策略四：压缩前提取（抢救性快照）**

在压缩发生之前，先主动提取关键信息——当前目标、已完成步骤、发现的意外情况——存入一个独立的结构化快照。压缩发生后，不依赖压缩后的内容回忆「刚才发生了什么」，而是直接读快照。

这就是代理免疫系统的底层逻辑——审查 Agent 不读大段执行日志，只读结构化快照。快照不经过压缩算法的「统一重要性评估」，而是由提取阶段主动甄别和保留。

### 3.2 审查严格度梯度：f(KV Cache 占用, Plan 复杂度)

注意力预算管理不只应用于「做什么」，也应用于「怎么审查」。审查严格度不应该是固定阈值，应该是一个二维函数：

$$\text{审查严格度} = f(\text{KV Cache 占用}, \text{Plan 复杂度})$$

| KV Cache 占用 | Plan 复杂度 | 审查模式 | 行为 |
|:---:|:---:|------|------|
| 低 | 低 | 宽松审查 | 「80% 清晰就批准」，偏执行 |
| 中 | 低 | 标准审查 | 逐条检查 Plan Step 与需求对应 |
| 高 | 低 | 严格审查 | 每步必须有需求映射 + 边界条件 |
| 高 | 高 | 最严格审查 | 每步必须有需求映射 + 边界条件 + 回滚策略 |
| 极高 | 任意 | 拒绝 + 强制压缩 | 不审查，先压缩上下文再提交 |

这个梯度设计的核心逻辑：**当 KV Cache 占用高时，模型本身已经是「部分失忆」状态。此时让一个「部分失忆」的审查 Agent 做宽松审查，审查本身的可靠性就不成立。** 所以占用越高，审查越严格——用审查严格度来补偿模型注意力的衰减，而不是假装模型仍然全知全能。

### 3.3 关键设计决策

**为什么是「预算管理」而不是「预算增加」？**

预算增加（更大的 Context Window）是模型提供商的责任，边际收益递减。预算管理是 Harness 的责任，收益非线性——省下的 40K tokens 不是让上下文变大，是让剩下的 token 每个都能获得更高的注意力权重。

**为什么四条策略缺一不可？**

四条策略覆盖了注意力稀释的四个路径：

- 策略一（前缀注入）：阻止核心约束被序列增长稀释
- 策略二（按需加载）：阻止无关 Skill 抢占注意力预算
- 策略三（子任务隔离）：阻止子任务噪音污染主执行流
- 策略四（压缩前提取）：阻止压缩算法随机丢弃关键信息

缺失任何一条，注意力预算都会从对应路径泄漏。四条策略一起构成一个「预算无泄漏」系统。

**为什么不在模型层做？**

在模型层做注意力分配——例如修改 attention mask 让某些 token 获得更高权重——需要修改模型架构和训练流程。这在工程上不可行（需要重新训练模型）、在灵活性上也有限（无法根据任务类型动态调整）。

在 Harness 层做注意力分配——通过控制哪些内容进入上下文、以什么顺序进入、哪些内容在压缩前被提取——不需要修改模型，可以对任何 Transformer 模型生效，且可以根据任务类型动态调整策略。

## 4. 分析

### 4.1 为什么预算管理能解决根本问题

根本问题不是「模型记不住」，是「不应该让模型来记」。预算管理把「记忆约束」的职责从模型迁移到 Harness：

- 硬约束不再依赖模型记忆 → 硬约束被前置到 KV Cache 前缀区，独立于序列增长
- Skill 不再全量占用注意力 → Skill 被按需加载，不预支预算
- 子任务噪音不再污染主流程 → 子任务被隔离到独立上下文
- 关键信息不在压缩中丢失 → 关键信息在压缩前被主动提取

每次迁移完成后，模型的注意力预算从「管理约束 + 管理 Skill + 管理子任务 + 管理历史」缩减为「管理当前任务的推理」。释放出的预算全部投入真正有价值的推理计算。

这解释了验证数据中的非线性收益：模型没有变聪明，但模型在**做更少的事情**——而人类感知到的「智能」恰好等于模型在做的事情中表现出来的质量，不是模型消耗的总注意力预算。

### 4.2 边界条件

以下场景本方案的收益递减：

- **超短对话（< 5 轮）**：注意力稀释尚未发生，预算管理的开销（前缀注入、Skill 筛选、快照提取）可能超过收益。此时应降级为直接执行，不施加预算管理。
- **单任务、无子任务场景**：策略三（子任务隔离）不生效，整体收益降低。
- **无压缩需求的场景**：如果上下文窗口远未耗尽（L(t) ≪ Context Limit），策略四（压缩前提取）的收益不显著。
- **极端压缩场景（压缩比 > 10:1）**：策略四的快照提取本身也依赖一定程度的上下文理解——在极端压缩下，提取精度可能下降，导致快照质量退化。

**本方案无法覆盖的场景**：

- **隐性约束**：用户未显式声明但隐含期待的约束。前缀注入只能处理显式声明的约束。
- **约束冲突**：两条约束互相矛盾时，系统无法自动仲裁——需要人工介入。

### 4.3 与最接近方案的对比

| 维度 | Claude Code 分段摘要 | Hermes 简单压缩 | 本方案（注意力预算管理） |
|------|------|------|------|
| 处理时机 | 上下文快满时触发 | 上下文快满时触发 | **全生命周期管理** |
| 对约束的处理 | 压缩算法统一评估 | 压缩算法统一评估 | **约束前置隔离，不进入压缩流程** |
| 注意力分配 | 被动（事后补救） | 被动（事后补救） | **主动（事前分配）** |
| 对模型依赖 | 依赖模型判断「重要性」 | 依赖模型判断「重要性」 | **Harness 层判断，模型只推理** |
| 子任务处理 | 无隔离 | 无隔离 | **独立上下文隔离** |
| 是否改模型 | 否 | 否 | **否** |

Claude Code 和 Hermes 简单压缩的核心问题相同：它们都在注意力被稀释之后才介入。预算管理框架的关键差异是**把介入时机提前到注意力被稀释之前**——不是「丢了再找」，而是「不让丢」。

## 5. 验证路径

### 5.1 已验证

在真实 Agent 任务上（基于 DeepSeek V4 Pro，混合重构 + 新建项目 + bug 修复三类任务，每组 20 次独立运行）：

| 指标 | 无预算管理（基线） | 预算管理后 |
|------|:---:|:---:|
| 第 15 轮约束保持率 | ~40% | **>95%** |
| 每次任务 Skill 加载 Token 消耗 | 10K-40K（全量） | 2K-5K（按需） |
| 每次任务节省 Token | — | **8K-35K** |
| 主 Agent 上下文膨胀速度 | 基线 | **降低 ~60%** |

**关键结论**：模型没有任何变化。同一个 DeepSeek V4 Pro。唯一的变量是 Harness 层如何管理注意力预算。但有效智能的差异是显著的——用户体感上，有预算管理的 Agent 「更聪明」，不是因为模型更强，是因为模型在做的每一件事上都有更充足的注意力。

### 5.2 待验证

- **多模型泛化**：在 DeepSeek V3、Qwen、Llama 等模型上验证策略效果是否一致
- **极端长对话**：100+ 轮对话下预算管理的退化临界点
- **审查梯度自适应**：KV Cache 占用和 Plan 复杂度二维函数的实际拟合效果（当前为启发式梯度，需要实验验证阈值）
- **预算管理开销**：四条策略本身消耗的计算成本和延迟——目前观察到的开销在可接受范围内（<5% 延迟增加），但需要系统化测量

## 6. 与 DeepSeek 的关系

注意力预算管理的四条策略，都依赖一个前提：**KV Cache 够便宜**。

- 前缀注入需要在前缀区放额外 token → Claude/GPT 的 Token 成本下，前缀区每增加 1K tokens 都是显著的成本负担
- 子任务隔离需要 spawn 多个独立上下文 → 总 Token 消耗上升（但被隔离后的效率提升覆盖）
- 审查梯度切换需要根据上下文状态动态评估 → 需要频繁的上下文状态检查

DeepSeek 的 KV Cache 成本优势（约为 Claude 的 1/5 到 1/10）让这些策略在经济上可行。同样的方案，在 Claude 上跑可能要多花 3-5 倍的 Token 成本——这不是 Claude 的模型不够好，是 Claude 的定价结构把「注意力预算管理」这种主动管理策略的价格推到了不可行的水平。

这就是「结构性优势」——不是 DeepSeek 的模型比 Claude 聪明，是 DeepSeek 的成本结构让「把注意力当预算管理」这件事做到了产品级可行。当一个 Harness 层策略需要额外的 Token 消耗来换取更高的有效智能时，Token 成本越低，这个策略的 ROI 越高。DeepSeek 的低成本结构恰好放大了 Harness 层创新的价值。

## 结论

Agent 变傻不是模型的问题。Agent 变傻是一个**工程问题**——Transformer 的注意力稀释是已知的物理边界，应对这个边界应该是 Harness 的职责，而不是「等下一代模型」。

注意力预算管理把 Harness 的定位从「被动调用模型」升级为「主动管理模型的注意力资源」。四条策略——前缀注入、按需加载、子任务隔离、压缩前提取——覆盖了注意力稀释的四个路径，构建了一个「预算无泄漏」系统。

验证数据表明：同一模型，有预算管理后的有效智能显著高于无预算管理的基线。这不是魔法——这是让模型做更少的事，但每件事都做得更好。
