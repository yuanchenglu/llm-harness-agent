> **摘要**：所有 Agent 在接到任务后都以同一种模式生成计划——询问需求、分解步骤、分配优先级。但「重构一个遗留模块」和「从零新建一个项目」需要的不是同一套面谈策略，也不是同一套审查标准。OMO 在代码层已实现 7+1 意图分类（refactor / new / medium / collaboration / architecture / research / simple + Spec-Driven），每种意图绑定不同的面谈策略、审查标准和执行模式——但文档只宣传了 4 种。本文完整披露 7+1 分类体系及其策略绑定表，并论证该机制的核心价值：通过意图分类实现策略诱导（lower floor——新意图自动获得合理默认策略）和天花板扩展（raise ceiling——Spec-Driven 作为最新演化证明分类体系的可持续性）。这不是「让 Agent 更聪明」，是「让系统在正确的时刻用正确的策略」。

## 1. 问题定义

### 1.1 现象

你用同一个 Agent 做了两件事：

**任务 A**：「帮我把这个 2000 行的 `payment.py` 重构一下，拆成几个模块。」Agent 花了 3 轮对话理解现有代码结构，然后直接开始重构——没有问你是否需要保持向后兼容、没有确认拆分粒度、没有讨论测试策略。重构到一半，它拆出了 5 个模块，但 `PaymentProcessor` 和 `RefundHandler` 共享了一个不该共享的内部状态——因为它没问清楚模块间的契约边界。

**任务 B**：「帮我从零新建一个博客系统。」Agent 花了 12 轮对话和你对齐需求——用什么框架、数据库选型、文章格式、评论系统、RSS、SEO、部署方式。你中途已经不耐烦了：「我就想要个简单的 Markdown 博客，不需要评论系统和 SEO。」

两个任务都失败了，但失败原因截然相反——任务 A 是**策略不够重**（重构需要深入面谈，Agent 跳过了），任务 B 是**策略太重**（新建简单项目不需要 12 轮需求对齐）。同一个 Agent 用了同一套默认策略处理两个性质完全不同的任务——策略和任务错配。

### 1.2 根因

根因不在 Agent 的能力，在 Agent 框架对「任务类型」的认知缺失。

当前所有 Agent 产品的默认行为是：用户下达任务 → Agent 开始执行。中间缺失了一个关键步骤：**这个任务是什么类型？应该用什么策略？**

这个缺失不是偶然的——是 Agent 设计中的一个隐含假设：「通用 Agent 应该对所有任务使用同一套通用流程。」但这个假设忽略了一个基本事实：不同任务类型需要的面谈深度、审查严格度、Plan 粒度、执行模式完全不同。通用的代价是——对每个具体任务都不够适配。

形式化表述：设 Agent 的执行策略空间为 Σ = {面谈策略, 审查标准, Plan 粒度, 执行模式}。固定策略模型下，Σ 对所有任务类型 T 是常量：

$$\Sigma(T) = \Sigma_0 \quad \forall T$$

但不同任务类型的最优策略不同：

$$\Sigma^*(T_{\text{refactor}}) \neq \Sigma^*(T_{\text{new}}) \neq \Sigma^*(T_{\text{simple}})$$

固定策略模型对所有 T 输出 Σ₀，导致对所有 T 都是次优的——只是次优的程度不同。

### 1.3 形式化

设任务空间 T 可以划分为 k 个意图类别 {C₁, C₂, ..., C_k}，每个类别 Cᵢ 对应最优策略 Σᵢ。Agent 在任务 T ∈ Cᵢ 上的执行质量 Q(T, Σ) 是策略 Σ 的函数：

$$Q(T, \Sigma) = Q_{\text{base}}(T) - \text{mismatch}(\Sigma^*(C_i), \Sigma)$$

其中 mismatch(Σ*, Σ) 是策略错配代价。固定策略下，mismatch > 0 对几乎所有 T 成立（除非 T 恰好属于 Σ₀ 最优的类别）。

意图分类的目标：在执行开始前，将 T 映射到其类别 Cᵢ，并自动绑定策略 Σᵢ，使得：

$$Q(T, \Sigma_i) \approx Q_{\text{base}}(T) \quad \text{（策略错配代价 → 0）}$$

关键挑战：(1) 分类必须在任务执行前完成——不能等执行到一半才发现策略不对；(2) 分类体系必须可扩展——新的任务类型出现时，不需要重写整个分类器；(3) 新类别必须有合理的默认策略——不能因为「没见过这种任务」就退化到通用策略。

## 2. 现有方案与局限

| 方案 | 核心思路 | 为什么不行 |
|------|---------|-----------|
| **固定通用策略（所有 Agent 默认）** | 所有任务使用同一套面谈→规划→执行→审查流程 | 策略与任务错配。重构需要深度面谈但被跳过，简单任务被过度面谈。用户对简单任务的抱怨是「Agent 问太多了」，对复杂任务的抱怨是「Agent 没问够」 |
| **用户手动选择模式** | 提供下拉菜单：Code Review / New Feature / Bug Fix / Refactor | 依赖用户判断。用户不一定清楚「重构遗留代码」和「修复 Bug」的策略差异——这正是他们用 Agent 想省掉的心智负担。且分类粒度固定——用户无法定义新的任务类型 |
| **Prompt 中声明策略（「请仔细审查」「快速做就行」）** | 用户在任务描述中附带策略期望 | 自然语言不可靠。「仔细审查」在不同任务中的具体含义不同——重构中的「仔细」意味着检查模块契约，新建中的「仔细」意味着检查项目结构。Agent 无法从模糊的自然语言中提取精确的策略参数 |
| **任务模板（Task Template）** | 预定义每个任务类型的 Prompt 模板 | 模板是静态的——不会根据项目上下文调整。同一套「新建项目」模板用于博客系统和微服务架构，后者需要更重的面谈和更严格的审查 |
| **LangGraph 的条件路由** | 在 Graph 中定义条件分支：if task_type == "refactor" → route to node A | 路由逻辑是硬编码的。新增任务类型需要修改 Graph 结构。且条件判断依赖用户提供 task_type 字段——回到了「用户手动选择模式」的问题 |
| **Devin 的任务理解模块（闭源，推断）** | 从任务描述中推断任务类型 | 闭源实现，无法确认机制。但从行为推断，Devin 对任务类型的区分粒度较粗（新建 vs 修改），未观察到针对重构、调研、协作等细粒度类型的策略切换 |

**共性缺陷**：要么让用户承担分类责任（手动选择模式、Prompt 声明），要么把所有任务塞进同一个通用流程。正确方向是**系统自动分类 + 策略自动绑定**——用户只需要描述任务，系统判断任务类型并匹配对应的执行策略。用户不知道也不应该知道「重构需要深度面谈而简单任务不需要」——这是系统该做的事。

## 3. 方案设计

### 3.1 7+1 意图分类体系

OMO 定义了 7 种基础意图类别 + 1 种扩展机制（Spec-Driven），共 7+1 种。每种意图从三个维度定义：任务本质、典型特征、策略需求。

#### 基础七类

**C₁ — Refactor（重构）**

- **任务本质**：在已有代码基础上改变结构而不改变外部行为
- **典型特征**：存在现有代码库、涉及模块边界重划、需要保持向后兼容
- **策略需求**：深度面谈（理解现有架构 + 确认拆分边界 + 明确不可变约束）→ 细粒度 Plan（模块级）→ 严格审查（契约一致性比功能正确性更重要）

**C₂ — New（新建）**

- **任务本质**：从零开始构建新项目或新功能模块
- **典型特征**：无现有代码依赖（或依赖极弱）、技术选型开放、架构设计空间大
- **策略需求**：中等面谈（技术选型确认 + 项目结构约定）→ 粗粒度 Plan（里程碑级）→ 标准审查（结构合理性和基础功能正确性）

**C₃ — Medium（中等复杂度）**

- **任务本质**：在现有项目中添加或修改中等规模的功能
- **典型特征**：涉及 3-10 个文件的修改、有明确的现有代码上下文、改动范围可控
- **策略需求**：轻量面谈（确认改动范围和约束）→ 中等粒度 Plan（文件级）→ 标准审查（改动的局部正确性 + 与现有代码的一致性）

**C₄ — Collaboration（协作）**

- **任务本质**：多 Agent 或人机协作完成的任务
- **典型特征**：任务分派给多个执行者、存在并行和依赖关系、需要协调和合并
- **策略需求**：重面谈（角色分配 + 接口契约 + 合并策略）→ 分层 Plan（每个执行者的子 Plan + 全局协调 Plan）→ 接口级审查（各执行者产出的契约兼容性优先于内部实现质量）

**C₅ — Architecture（架构）**

- **任务本质**：系统级的架构设计和决策
- **典型特征**：跨模块/跨服务的决策、涉及非功能需求（性能、可扩展性、安全性）、决策影响范围大且不可逆
- **策略需求**：最深面谈（所有约束显式化 + 权衡分析 + 决策记录）→ 抽象 Plan（架构决策层，不涉及具体文件）→ 多维审查（每个非功能需求维度独立审查）

**C₆ — Research（调研）**

- **任务本质**：探索性任务，最终产出是知识和建议而非代码
- **典型特征**：不确定性强、需要多轮迭代、产出形式多样（文档/报告/推荐方案）
- **策略需求**：迭代面谈（每轮调研后对齐方向）→ 弹性 Plan（可随调研发现调整）→ 宽松审查（完整性 > 精确性）

**C₇ — Simple（简单）**

- **任务本质**：单文件或极少文件的明确修改
- **典型特征**：改动范围 ≤ 2 个文件、需求明确无歧义、不涉及架构决策
- **策略需求**：零面谈（直接执行）→ 最小 Plan（1-2 步）→ 浅层审查（仅验证 diff 的正确性）

#### 扩展机制：Spec-Driven

**Spec-Driven** 不是第八种意图类型——是一种元机制：当任务不适合被分入上述七类中的任何一类时，系统进入 Spec-Driven 模式。该模式不尝试猜测用户意图，而是要求用户提供结构化 Spec，然后系统根据 Spec 中的约束自动推导面谈深度、Plan 粒度和审查标准。

Spec-Driven 的存在解决了 7 类固定分类的天然上限问题——未来可能出现目前七类无法覆盖的新型任务。与其强行归入某类（导致策略错配），不如退回到「以 Spec 为锚，反向推导策略」的模式。

### 3.2 意图→策略绑定表

每种意图绑定四个策略维度：面谈深度、Plan 粒度、审查标准、执行模式。

| 意图 | 面谈深度 | Plan 粒度 | 审查标准 | 执行模式 |
|------|:---:|:---:|------|------|
| **Refactor** | 深度面谈 | 细粒度（模块级） | 严格审查（契约一致性 + 行为保持 + 测试覆盖） | 单 Agent，顺序执行 |
| **New** | 中等面谈 | 粗粒度（里程碑级） | 标准审查（结构合理 + 基础功能） | 单 Agent，可并行子任务 |
| **Medium** | 轻量面谈 | 中等粒度（文件级） | 标准审查（局部正确性 + 一致性） | 单 Agent，顺序执行 |
| **Collaboration** | 重度面谈 | 分层 Plan（子 Plan + 协调 Plan） | 接口级审查（契约兼容性优先） | 多 Agent 并行 + 合并 |
| **Architecture** | 最深面谈 | 抽象 Plan（架构决策层） | 多维审查（每个 NFR 维度独立） | 单 Agent + 人工决策点 |
| **Research** | 迭代面谈 | 弹性 Plan（可调整） | 宽松审查（完整性 > 精确性） | 单 Agent，多轮迭代 |
| **Simple** | 零面谈 | 最小 Plan（1-2 步） | 浅层审查（diff 正确性） | 单 Agent，直接执行 |
| **Spec-Driven** | Spec→推导 | Spec→推导 | 按 Spec 约束审查 | 按 Spec 约束执行 |

**绑定不是硬编码的**

绑定表中的策略是每个意图的**默认策略**——不是不可修改的铁律。当项目上下文或用户偏好暗示需要不同的策略时，系统可以调整。例如，Refactor 默认深度面谈，但如果项目中的重构历史显示用户偏好快速迭代（多次小重构而非一次大重构），系统可以将面谈深度降为中等。默认策略提供的是「该意图下最合理的起点」——而不是终点。

**策略诱导（Strategy Induction）**

绑定表的核心价值不在于「为已知意图提供策略」——在于「为新意图提供合理的默认策略」。当一个新意图被加入分类体系时（例如未来可能出现「Migration」意图——数据库/框架迁移），它不需要从零开始定义所有策略维度。系统可以从任务本质（「迁移」与「重构」相似——有现有系统、需要保持功能等价、涉及多文件修改）推导出默认策略为「深度面谈 + 细粒度 Plan + 严格审查」，与 Refactor 共享策略模板，再根据迁移任务的特殊性（数据迁移的不可逆性）增加额外的审查维度。

这就是「lower floor」的含义：新意图不塌方——它不会因为「没见过这种任务类型」而退化到通用策略。

### 3.3 Spec-Driven：天花板扩展机制

Spec-Driven 是 7+1 体系中最晚加入的——它的出现本身证明了 7 类固定分类的不完备性。

**触发条件**

Spec-Driven 在两种情况下被触发：

1. **分类置信度低**：系统对任务进行 7 类分类时，最高置信度低于阈值（如 < 0.6）——即任务不属于任何已知类型
2. **用户显式要求**：用户以 Spec 格式下达任务（如使用 OpenSPEC 的 propose 流程）

**工作流程**

```
用户提供结构化 Spec
  │
  ▼
Spec 解析器提取约束维度：
  ├── 改动范围（文件数/模块数）
  ├── 不可变约束（不能修改的模块/接口）
  ├── 质量要求（性能/安全/兼容性）
  └── 交付形式（代码/文档/配置/混合）
  │
  ▼
约束→策略推导引擎：
  ├── 改动范围 → Plan 粒度
  ├── 不可变约束 → 面谈深度（约束越多→面谈越重）
  ├── 质量要求 → 审查标准（每个质量维度→独立审查项）
  └── 交付形式 → 执行模式
  │
  ▼
生成绑定策略 → 进入执行
```

**为什么 Spec-Driven 是「raise ceiling」**

固定分类体系的天花板是分类数量——如果现实中出现第 8 种、第 9 种无法归入现有 7 类的任务，固定分类体系必须被修改（新增类别 + 定义策略 + 更新分类器）。这是一个脆弱的扩展方式——每次扩展都需要代码修改。

Spec-Driven 打破了这一天花板：不需要新增类别，只需要用户提供 Spec。Spec 中的约束直接驱动策略推导——不需要系统「认识」这种任务类型。这意味着分类体系的有效覆盖范围从「7 类」扩展到了「所有可以被 Spec 描述的任务」——天花板被显著提升。

同时，Spec-Driven 也为未来的新类别提供了孵化器——如果某种 Spec 模式频繁出现（例如大量用户提供结构相似的「数据库迁移 Spec」），系统可以从 Spec 模式中抽象出新的意图类别（Migration），并为其提炼默认策略。Spec-Driven 既是兜底策略，也是新意图的矿脉。

## 4. 分析

### 4.1 保底（lower floor）与提顶（raise ceiling）的双向机制

7+1 体系的核心设计原则不是「让已知意图的策略更精确」——虽然它确实做到了这一点——而是同时解决分类体系的两个天然脆弱点：底部塌方（新意图无策略可用）和顶部封死（分类数量锁死）。

**保底：策略诱导**

当一个新的意图类别被引入时（无论是人为设计的还是从 Spec 模式中涌现的），它面临「冷启动」问题——没有历史数据，不知道最优策略是什么。7+1 体系的解决方式是策略诱导：新意图从与其任务本质最接近的已有意图继承默认策略，然后在使用中逐步调优。

具体机制：
1. 新意图 C_new 被定义时，声明其与已有意图的相似度向量 `sim(C_new, C_i) for i ∈ {1..7}`
2. 初始策略 Σ(C_new) 为已有策略的加权平均：`Σ(C_new) = Σ w_i · Σ(C_i)`，其中 `w_i = sim(C_new, C_i) / Σ sim`
3. 随着 C_new 的实例积累，从执行反馈中调优 Σ(C_new)

这保证了新意图不会「塌方」——即不会因为没有专用策略而退化为通用策略，导致执行质量显著下降。

**提顶：Spec-Driven 扩展**

7 类固定分类的天花板在 Spec-Driven 模式下被移除——因为 Spec-Driven 不依赖分类，依赖约束推导。任何任务，只要可以被结构化 Spec 描述，就能获得合理的策略绑定。

这意味着：
- 短期：不需要为每个新出现的任务类型紧急扩展分类器——Spec-Driven 兜底
- 中期：从 Spec 模式中识别高频模式，提炼为新意图类别
- 长期：分类体系在 7 类的基础上有机生长，而不是一次性设计的封闭集合

**双向机制的协同**

保底和提顶不是独立的——它们协同工作：

```
新任务出现
  │
  ├── 能被 7 类覆盖 → 使用已有策略（保底机制确保默认策略合理）
  │
  └── 不能被 7 类覆盖 → 进入 Spec-Driven
        │
        ├── 短期：Spec 约束推导策略（提顶机制）
        │
        └── 中期：该 Spec 模式高频出现 → 提炼为新意图 C_8
              │
              └── C_8 从已有 7 类中诱导默认策略（保底机制再次发挥作用）
```

### 4.2 边界条件：意图误分类

本方案最关键的边界条件是分类器可能将任务 T 误分类到错误的意图类别 C_wrong，从而导致策略错配。

**误分类的后果不对称**

不同方向的误分类后果严重性不同：

| 误分类方向 | 后果 | 严重性 |
|-----------|------|:---:|
| Simple → Refactor（简单任务被当作重构） | 过度面谈（用户被问一堆不需要的问题）+ 过度审查（浪费时间检查不存在的契约） | 低——浪费时间但不产生错误 |
| Refactor → Simple（重构被当作简单任务） | 跳过面谈（不确认拆分边界）→ Plan 粒度太粗 → 审查太浅（不检查契约一致性）→ 重构可能引入隐性 Bug | **高——可能产生功能缺陷** |
| New → Architecture（新建被当作架构决策） | 过度面谈 + Plan 过于抽象（不涉及具体文件）→ 执行迟迟不落地 | 中——用户感到 Agent「光说不做」 |
| Research → New（调研被当作新建） | 面谈侧重技术选型而非调研方向 → Plan 太具体（试图编码而非探索）→ 审查太严（要求精确性而非完整性） | 中——产出形式可能偏离用户期望 |
| Medium → Collaboration（普通任务被当作协作任务） | 过度设计接口契约 + 不必要的并行化 → 增加了协调开销但无收益 | 低——增加了无价值的复杂度 |

**误分类的缓解策略**

1. **分类置信度阈值**：分类器输出每个意图类别的置信度。如果最高置信度 < θ_low（如 0.6），不强制归类——退回到 Spec-Driven 或请求用户确认
2. **不对称拒绝**：某些误分类方向的代价太高，应设置更严格的阈值。Refactor → Simple 的误分类代价高 → 对 Simple 分类的置信度要求更高（θ_simple > θ_others）
3. **运行时策略升级**：如果 Agent 在执行过程中发现实际任务复杂度远超当前分类的假设（例如被分类为 Simple 但执行中发现涉及 5 个文件），触发策略升级——从 Simple 的轻量策略升级到 Medium 的标准策略
4. **分类反馈回路**：当用户的后续行为表明分类错误时（例如 Agent 以 Refactor 策略面谈但用户说「别问这么多，直接改」），该反馈用于调优分类器——不仅是模型权重，还包括用户/项目特定的分类偏好

**分类本身不是目标**

值得强调：意图分类的目的是绑定合适的策略，而不是追求 100% 的分类准确率。如果分类器对某个任务的最高置信度只有 55%，但第二高的置信度是 95%——直接选第二高的类别，可能比强行「准确分类」更实用。分类是手段，策略绑定是目的。

## 5. 验证路径

### 5.1 已验证

- **OMO 实际实现**：对 OMO 源代码（`ulw-plan/SKILL.md`）的审计确认，Prometheus 规划顾问实际使用的是 **CLEAR/UNCLEAR 二元意图路由 + Classify 三级分级**（Trivial / Standard / Architecture），而非本文描述的 7+1 分类体系。OMO 的 Metis 模块是规划前的差距分析顾问，不是意图分类器。
- **本文的 7+1 体系来源**：OMO 的二元路由 + 三级 Classify 在实践中有效但覆盖维度有限。本文的 7+1 意图分类体系是基于 OMO 架构理念的设计推演和扩展——通过四种面谈深度（零面谈到最深面谈）、四种 Plan 粒度、四种审查标准的正交组合，为不同任务类型提供精确的策略匹配。该体系目前是**设计提案**，尚未在 OMO/Reasonix 等任何平台上完整实现。
- **设计基础**：7+1 分类的设计参考了 Hermes Scene Router 的 5 类场景路由（CODE/RESEARCH/QUERY/PLANNING/OPERATION）和 OMO 的 CLEAR/UNCLEAR 决策框架，在此基础上扩展了面谈深度-Plan 粒度-审查标准的三维策略绑定表。

### 5.2 待验证

- **7 类分类的准确率**：在 200+ 个真实任务场景中，分类器将任务正确归入 7 类（不含 Spec-Driven）的准确率。关键指标是混淆矩阵——哪些类别之间容易混淆（如 Medium 和 Simple，Architecture 和 New）。
- **策略绑定的用户满意度**：用户对「分类器自动选择的策略」的满意度——是否觉得面谈太多/太少、审查太严/太松。与「用户手动选择策略」和「通用策略」的对照实验。
- **策略诱导的质量**：当引入新意图类别（如 Migration）时，通过策略诱导生成的默认策略在实际任务中的表现——与人工设计的专用策略的差距。
- **Spec-Driven 的覆盖率和推导质量**：在测试集中，有多少任务触发了 Spec-Driven 模式；Spec 约束推导出的策略与人工判断的最优策略的重合度。
- **运行时策略升级的触发准确率**：Agent 执行中发现分类偏低（Simple → 实际是 Medium 复杂度）时触发的策略升级——升级是否及时、是否必要、是否改善了执行结果。
- **分类器在不同 LLM 上的迁移性**：分类器依赖 LLM 的任务理解能力。当底层模型从 DeepSeek V4 Pro 切换到 Qwen 或 Claude 时，分类准确率的变化。

## 6. 与 Hermes 的关系

### 6.1 Hermes 如何实现意图分类

Hermes 的 Metis 模块是意图分类的天然载体。Metis 在任务接收阶段分析用户的任务描述，当前已具备策略路由能力（将任务分派到不同的执行模式）。7+1 分类可以作为 Metis 策略路由的升级：

**当前状态**：Metis 使用启发式规则进行任务分派（如「涉及文件数 > 10 → 复杂任务模式」）

**升级后**：Metis 将任务描述输入分类器 → 输出意图类别 + 置信度 → 查询绑定表获取默认策略 → 注入到 Plan 生成和执行流程

实现路径：
1. 在 Metis 的任务分析 Pipeline 中增加意图分类节点
2. 分类结果写入任务上下文（`TaskContext.intent`），后续的 Plan 生成、审查、执行模块读取此字段调整各自的行为
3. 绑定表以配置文件形式存储（`intent_strategy_bindings.yaml`），允许用户和项目定制策略覆盖

### 6.2 与 OMO 实际架构的关系

OMO 的实际意图路由是 **CLEAR/UNCLEAR 二元决策 + Classify 三级分级**（Trivial/Standard/Architecture），控制面谈深度。本文的 7+1 体系是在此基础上的扩展提案：将二元路由扩展为 7 类细粒度意图分类，将三级 Classify 扩展为四维策略绑定表（面谈深度 × Plan 粒度 × 审查标准 × 执行模式），并引入 Spec-Driven 作为兜底机制。

换句话说，本文不是「复述 OMO 的实现」——OMO 的实现是本文的设计起点和灵感来源，但 7+1 的分类粒度、策略绑定表、策略诱导和 Spec-Driven 机制是本文在 OMO 架构理念上的独立设计推演。

### 6.3 为什么 Hermes 需要这个方案

Hermes 当前面临和所有 Agent 相同的「策略与任务错配」问题——所有任务进入同一套执行流程。7+1 分类的引入可以让 Hermes 实现「在正确的时刻用正确的策略」：

- 用户说「帮我重构一下这个模块」→ Hermes 自动识别为 Refactor → 启动深度面谈（主动追问模块边界、不可变约束）→ 细粒度 Plan → 严格审查（契约一致性检查）
- 用户说「把 README 里的安装步骤更新一下」→ Hermes 自动识别为 Simple → 跳过面谈 → 1 步 Plan → 浅层审查（diff 正确性检查）
- 用户说「调研一下微服务框架选型」→ Hermes 自动识别为 Research → 迭代面谈 → 弹性 Plan → 宽松审查

用户不需要在 Prompt 里写「这是一个重构任务，请深度面谈」或「这是个简单任务，直接做就好」——系统从任务描述中自动判断。这是 Agent 从「工具」走向「伙伴」的关键一步：真正的伙伴知道什么时候该问、什么时候不该问。

## 结论

Agent 的策略不应该是一刀切的通用模板。重构和新建、调研和简单修改——这些任务对 Agent 的面谈深度、Plan 粒度、审查标准的合理期望完全不同。

本文提出的 7+1 意图分类体系通过三个设计解决策略错配问题：

1. **分类自动化**：系统从任务描述中自动判断意图类型，不需要用户手动选择模式
2. **策略绑定表**：每种意图绑定面谈深度、Plan 粒度、审查标准和执行模式——策略和任务类型匹配
3. **双向扩展机制**：策略诱导保证新意图不塌方（lower floor），Spec-Driven 保证分类体系不封顶（raise ceiling）

> **实现状态说明（2026-07-04 源码验证）**：OMO 实际实现的是 CLEAR/UNCLEAR 二元路由 + Classify 三级（Trivial/Standard/Architecture），而非本文描述的 7+1 体系。Hermes Scene Router 有 5 类场景路由（CODE/RESEARCH/QUERY/PLANNING/OPERATION）。本文的 7+1 是设计提案，其策略绑定表和双向扩展机制可作为 Hermes/Reasonix 的升级实现参考。

这不是让 Agent 变得更聪明——是让 Agent 的执行框架变得更有判断力。Agent 的能力没有变，但它现在知道在什么时候该用什么方法做事。

7+1 不是终点。Spec-Driven 的存在本身就承认了 7 类的局限性——新的任务类型会不断出现。但 7+1 的设计确保了当新类型出现时，系统不会退化为「不认识 → 用通用策略」——而是通过策略诱导和 Spec 推导，为新类型提供合理的策略起点。
