> **摘要**：长任务 Agent 必须压缩上下文以控制窗口膨胀，但压缩算法用统一权重评估所有信息——它不知道「config 文件绝对不能动」比「第 7 轮那个 warning」重要 100 倍。本文提出 KV Cache 前缀区硬约束注入方案：把不可丢失的约束从可压缩区抽离，放入前缀区（KV Cache）实现物理隔离。约束不丢不是因为压缩算法聪明——是因为约束根本不在被压缩的内容里。该方案在 15 轮对话后将约束保持率从 ~40% 提升至 >95%，且其经济可行性依赖于 DeepSeek 的低 KV Cache 成本——同样的方案，Claude 和 GPT 做不起。

---

## 1. 问题定义

### 1.1 压缩的「公平性暴力」

每一个长任务 Agent 都必须做上下文压缩。不压缩 = 上下文窗口爆炸 = Agent 跑不动。压缩本身不是问题——问题是：**你不知道压缩后什么信息丢了。**

典型场景：你在第 3 轮对话中反复强调：「config 文件绝对不能动，动了整个服务会挂。」Agent 回应：「明白。」

20 轮对话后，上下文被压缩了。那个在第 3 轮被你反复强调的约束——还在吗？

压缩算法不知道「config 文件绝对不能动」比「第 7 轮报的那个 warning 没啥关系」重要 100 倍。它用统一的权重评估所有信息。重要的被淹没，不重要的被保留。这就是压缩的「公平性暴力」——在数学上公平，在语义上愚蠢。

### 1.2 根因：约束与对话共享同一个压缩域

问题的根源不是压缩算法不够好——是**架构层面的失误**：硬约束（不可丢）和对话历史（可压缩）被放在了同一个区域，接受同一个压缩算法的统一处理。

```
传统方案：
  [System Prompt 包括约束A/B/C][对话1][对话2][对话3]...[对话N]
    ↑ 全在同一压缩域内 → 压缩后约束A可能被随机丢弃
```

约束和对话在序列中的位置不同、重要性不同、生命周期不同，但被一视同仁地送入同一个压缩器。这不是压缩器的问题——是「把所有东西混在一起再压缩」这个前提本身有问题。

### 1.3 形式化

设 Agent 的长对话序列 S = C ∥ D，其中 C = {c₁, c₂, ..., cₖ} 为硬约束集合（不可丢失），D = {d₁, d₂, ..., dₜ} 为对话历史（可压缩）。

设压缩函数 f 将序列长度从 L 压缩至 L'（L' < L），压缩比 r = L'/L。f 在语义上是不忠实的——对任意信息片段 x ∈ S，f 保留 x 的概率 P(保留 | x, f) 与 x 的语义重要性无关。

当 t → ∞，序列长度 L(t) ∝ t，压缩比 r 趋向一个固定上界（由窗口大小决定），因此对任意 cᵢ：

P(cᵢ ∈ f(S)) → r · (1/|S|) · |S| = r

即每条约束的期望留存率收敛于压缩比 r。当 r = 0.4（典型 Agent 长任务的压缩比），约束留存率约为 40%。

**关键洞察**：如果 C 和 D 被送入同一个 f，C 的留存率由 r 决定，不可控。但如果 C 被物理隔离——不经过 f——则 P(cᵢ 保留) = 1。

## 2. 现有方案与局限

| 方案 | 核心思路 | 为什么不行 |
|------|---------|-----------|
| **更聪明的压缩算法** | 优化压缩策略，用重要性评分代替统一权重 | 仍然是事后补救。重要性评估本身依赖模型判断——模型在长上下文中同样会遗忘，重要性评估的准确性随对话轮数下降 |
| **重要性感知压缩** | 标记关键信息为「不可压缩」 | 标记行为本身依赖模型在长上下文中的判断。轮数越深，模型越可能漏标。本质是将同一个模型的注意力问题外包给同一个模型 |
| **更长的压缩间隔** | 多攒几轮再压缩，压缩频率更低 | 延迟问题。窗口耗尽只是来得更晚——从第 15 轮推迟到第 30 轮，但问题性质不变 |
| **分段摘要 + 累计追踪**（Claude Code） | 对历史对话分段摘要，维护文件修改的累计追踪 | 业界最强，但摘要仍是压缩——摘要中约束的表达被改写后，忠实度下降。约束从「绝对不能动 config 文件」变成「注意 config 文件」 |
| **约束重复注入** | 每 N 轮在对话中重新注入关键约束 | 占上下文窗口 → 加速 L(t) 增长 → 更早触发压缩 → 加速约束稀释。正反馈恶化 |

**共性缺陷**：所有方案都在「压缩后保留最重要的信息」这个命题上卷。正确的命题应该是：**什么信息根本不应该被压缩？**

## 3. 方案设计

### 3.1 前缀区隔离原理

KV Cache 的物理结构提供了一个天然的隔离区：**前缀区**。

前缀区的 token 在生成每个新 token 时都会被计算（参与自注意力），但它们位于序列的最前端，**不参与后续的对话累积和压缩流程**。前缀区 = 物理上的「压缩免死金牌」。

核心方案：将硬约束从可压缩区抽离，放入系统 prompt 的稳定前缀（Agent 启动时组装一次后冻结，利用推理引擎的 Prefix Caching 机制使其不参与后续对话的压缩流程）。

```
传统方案：
  [System Prompt 包括约束A/B/C][对话1][对话2][对话3]...
    ↑ 全在可压缩区 → 压缩后约束A可能被丢掉

前缀注入方案：
  [约束A][约束B][约束C]  ← 前缀区（KV Cache，从不被压缩）
  ═══════════════════════
  [System Prompt 其余部分][对话1][对话2][对话3]... ← 可压缩区
```

核心逻辑：不需要更聪明的压缩算法。只需要在物理上把「不能丢的」和「可以丢的」分开。

> **约束不丢不是因为压缩算法聪明。是因为约束根本不在被压缩的内容里。**

### 3.2 硬约束的定义与提取

不是所有指令都应该注入前缀区。前缀区是稀缺资源——每多放一个 token，KV Cache 的固定计算成本就增加一点。

**硬约束的定义**：违反就导致任务不可接受的失败。

| 约束类型 | 示例 | 是否硬约束 | 理由 |
|---------|------|:---:|------|
| 安全红线 | 「不能删除生产数据库」 | ✅ | 违反 = 灾难 |
| 环境限制 | 「必须用 Python 3.11」 | ✅ | 违反 = 代码不可运行 |
| 操作边界 | 「不要改 /etc/ 下的文件」 | ✅ | 违反 = 系统级破坏 |
| 格式要求 | 「输出用 Markdown 表格」 | ✅ | 违反 = 产出物不可用 |
| 风格偏好 | 「代码注释用英文」 | ⚠️ 视场景 | 不遵守不导致失败，但可能在特定场景（如国际化团队）升格为硬约束 |
| 命名习惯 | 「变量名用驼峰式」 | ❌ | 不必要时不放——前缀区资源有限 |

**提取机制**：硬约束的提取应当自动化——从 System Prompt 或用户消息中通过模式匹配提取「不得/不要/必须/禁止/严禁」等关键词，自动注入前缀区。不需要用户手动标注，也不需要模型在长上下文中判断——关键词匹配是确定性的，不受对话轮数影响。

### 3.3 实现：System Prompt 预处理 Skill

在 Hermes 架构中，本方案作为一个 **System Prompt 预处理 Skill** 实现，充分利用 Hermes 的三层注入结构（模板注入 + Skills 注入 + Memory 注入）：

1. 用户发送任务时，预处理 Skill 扫描 System Prompt 和用户消息
2. 通过关键词匹配提取所有硬约束（「不能/不要/必须/禁止/严禁」）
3. 将提取的约束列表注入前缀区（KV Cache 层）
4. 正常执行任务——System Prompt 的其余部分和对话历史在可压缩区

这个 Skill 不需要修改 Hermes 的核心代码。它是 Skills 机制的一个优雅用例——利用 Harness 层的可扩展性，在不触碰核心架构的前提下实现约束的物理隔离。

## 4. 分析

### 4.1 为什么这个方案能解决根本问题

根本问题不是「压缩算法不够聪明」——是**约束和对话共享同一个压缩域**。

所有现有方案都在试图改善 f 的忠实度（让压缩后保留更重要的信息）。但 f 的忠实度有硬上限——模型的注意力随序列长度稀释是物理规律，不是工程问题。

前缀注入方案不改善 f。它**绕过 f**：

- 硬约束 C 进入前缀区 → 不经 f → P(保留) = 1（物理保证）
- 对话历史 D 留在可压缩区 → 经 f → 接受压缩（可接受，因为 D 本身就是为了被压缩而设计的）

这和 Claude Code 的 compaction 有本质区别：Claude Code 的 compaction 是在 **f 内部**做优化（更好的摘要策略、累计追踪）。前缀注入是在 **f 外部**做架构——让不该被 f 处理的东西根本不过 f。

比喻：Claude Code 在造更好的榨汁机（让渣更少、汁更纯）。前缀注入在说——先别榨，把重要的果子拿出来放旁边。

### 4.2 边界条件

以下场景前缀注入**无法**覆盖：

- **隐性期望**：用户隐含的、未在 Prompt 中明确表达的约束。关键词提取只能抓显式声明——「代码应该写得优雅」这种模糊期待无法被识别为硬约束，也无法注入前缀区。它在压缩区，会被压缩。

- **情境依赖约束**：约束的有效性依赖当前上下文判断。例如「如果是紧急修复，可以跳过代码审查」——将这条原样注入前缀区后，模型在每一轮都会看到它，但「紧急修复」的判断在对话区。对话区被压缩后，模型可能无法判断当前是否属于紧急修复场景。

- **约束膨胀问题**：如果 System Prompt 中 80% 的语句都包含「必须/不得」关键词，前缀区将被硬约束填满。前缀区每多 1K tokens，每个新 token 的生成成本相应上升。极端情况下，过大的前缀区会反噬推理速度——这是一个需要在「约束覆盖率」和「推理成本」之间权衡的工程决策。

- **前缀区注意力稀释**：虽然约束不在压缩范围内（P(保留) = 1），但如果前缀区本身很大（比如注入 5K tokens 的约束），模型在极长对话中仍可能对前缀区后部的约束注意力权重不足。前缀注入解决了「约束不丢失」的问题，但不能完全解决「约束被遵守」的问题。后者需要配合[注意力预算管理](03-attention-budget.md)。

### 4.3 与 Claude Code Compaction 的对比

| 维度 | Claude Code Compaction | 前缀注入方案 |
|------|----------------------|:---:|
| 优化对象 | 压缩算法 f 的内部质量 | 被压缩对象的物理分布 |
| 策略性质 | 被动——压缩后再恢复 | 主动——压缩前先隔离 |
| 约束保留机制 | 依赖摘要的语义忠实度 | 物理隔离，不经过压缩 |
| 对模型依赖 | 需要模型在长上下文中保持判断力 | 不需要——关键词匹配是确定性的 |
| 保留率 | 渐进改善，有硬上限 | >95%（物理保证） |
| 额外成本 | 每次压缩消耗 token 做摘要 | 前缀区每多 1K tokens → 每个新 token 生成成本微量上升 |

Claude Code 的 compaction 是业界最成熟的方案，但它的本质仍然是在「压缩后保留最重要信息」上做优化。前缀注入换了一个命题——**不压缩**最重要信息。

## 5. 验证路径

### 5.1 已验证

| 方案 | 15 轮对话后约束保持率 | 备注 |
|------|:---:|------|
| 无前缀注入（约束在可压缩区） | ~40% | 约束随对话内容一起被压缩——留存率收敛于压缩比 |
| **前缀注入（约束隔离）** | **> 95%** | 约束在物理隔离区内，压缩触碰不到 |

> **数据说明**：自测数据，与 article 03 同组实验的前缀注入维度，非独立实验。需独立复现。

保持率 > 95% 不是 100% 的原因：不是约束被压缩丢了——是在极长上下文中，模型对前缀区远端约束的注意力权重可能被稀释。前缀注入确保了「物理存在」，但「注意力分配」仍受 Transformer 的 O(n²) 注意力分布限制。

### 5.2 待验证

- **成本-收益曲线**：前缀区每增加 1K tokens，对话吞吐量的边际下降是多少？在什么约束量级下，前缀注入的成本超过收益？
- **前缀区最优大小**：不同任务类型下，前缀区 token 数的 sweet spot 在哪里？是否存在「约束挤占推理空间」的临界点？
- **关键词匹配的精确率/召回率**：自动提取硬约束的误提取率（把不需要注入的规则也注入）和漏提取率（漏掉了真正的硬约束）
- **与注意力预算管理的联合效果**：前缀注入（物理保障）+ 注意力预算管理（注意力分配优化）联合运行时，15 轮后的约束**遵守率**（不只是保持率）

## 6. 为什么只有 DeepSeek 能这样玩

这不是技术壁垒——是**经济壁垒**。

KV Cache 不是免费的。前缀区放的约束越多，每个新 token 生成时要计算的 KV 对越多 → Token 生成成本上升。标准自注意力下每多 1K tokens 在前缀区 → 每个后续 token 的推理 FLOPs 约按 d_model × n_layers 量级上升。但 DeepSeek V4 通过 CSA（Compressed Sparse Attention）+ MQA（1 KV head）大幅压缩了 KV cache 和推理开销（官方数据：10% KV cache vs V3.2），意味着前缀区可以放更多约束而不显著增加成本。

- **Claude / GPT**：Token 本身贵（GPT-4 输出 $60/1M tokens）。前缀区每多 1K tokens，每个新 token 成本上升 → 边际成本显著。一个长任务可能生成 50K tokens → 额外成本足以让商业上不可行。
- **DeepSeek**：KV Cache 便宜（输出 ~$2/1M tokens）。前缀区可以放更多约束 → 边际成本可忽略。同样的 50K token 任务，额外成本在一顿饭钱以内。

同样的方案，从工程上 Claude 和 GPT 也能实现，但从商业上做不起。这不是「别人做不到」——是「别人做不起」。

这就是为什么 DeepSeek 的 KV Cache 优势是**产品护城河**。Harness Engineering 的本质，就是利用这种结构性成本优势，设计出别人在经济上追不上的产品方案。当你的竞争优势来自对手的成本结构而非技术能力时，对手的追赶窗口不是在「研发一个新方案」——是在「等自己的 Token 降价」。

## 结论

上下文压缩的「公平性暴力」不是压缩算法的问题——是架构层面的失误：把不该被压缩的东西放进了压缩域。前缀注入方案不优化压缩算法，而是把硬约束从压缩域中物理隔离出去。约束不丢不是因为算法聪明——是因为它根本不在被压缩的内容里。而这项方案的经济可行性，依赖于 DeepSeek 的低 KV Cache 成本——这是 Harness Engineering 利用结构性成本优势的典型案例。
