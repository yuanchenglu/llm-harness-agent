> **摘要**：Momus 作为 Hermes 的审查 Agent，在短对话中表现良好——「80% 清晰就批准」是一个合理的高效启发式。然而随着对话增长、KV Cache 占用攀升，Momus 的审查判断力随注意力稀释同步退化。在 128K 上下文中以「80% 清晰」为标准的审查，实际有效审查质量可能已降至 40%——而 Momus 自己不知道自己变傻了。本文指出所有 Agent 框架共享同一个隐藏假设——审查深度是固定阈值——并论证该假设在长上下文下构成系统性自欺。提出审查深度的二维动态函数 f(KV Cache 占用, Plan 复杂度)，定义四个审查区间的深度梯度（<8K / 8-32K / 32-128K / >128K），使审查深度随上下文状态自动切换。该方案是注意力预算管理框架（article 03）中审查维度的专项深化。

## 1. 问题定义

### 1.1 Momus 的自欺陷阱

Momus 是 Hermes 的审查 Agent，负责在任务执行过程中检查产出物是否符合需求。它的核心工作模式是：

1. 读取当前 Plan Step 的 `key`（可验证完成标准）
2. 读取 Agent 的产出物（代码、文档、配置）
3. 判断是否「足够清晰」→ 通过 or 驳回

在短对话中（< 10 轮），这个模式工作得很好。Momus 的标准审查深度——「逐条检查 Plan Step 与需求的对应关系」——在注意力充足的条件下是有效的。

问题出现在长对话中。

当主 Agent 已经执行了 30+ 轮对话，上下文膨胀到 64K+ tokens 时，Momus 也会以该膨胀的上下文作为审查输入。Momus 读取产出物时，它的注意力已经被前面数万 tokens 的历史稀释——如同一个疲劳的审查员在连续工作 8 小时后审阅第 47 份文件。

但 Momus 使用的仍然是同一个审查标准：「80% 清晰就批准」。

这里的自欺链条是：

1. Momus **以为自己**在 80% 清晰度标准下审查
2. 但实际上，由于注意力稀释，它对「清晰度」的判断本身已不准确——它可能漏掉了关键不一致性
3. 它仍然给出「批准」——因为它没有意识到自己的判断力已经下降
4. 用户收到的是「审查已通过」的结果——但审查质量已经显著退化

这就是「固定审查深度的自欺」——审查标准没有变，但执行审查的 Agent 能力已经变了，而审查框架没有感知到这个变化。

### 1.2 根因：审查深度是一个隐藏静态参数

在今天的 Agent 生态中，审查深度（review depth）不是一个显式的、可配置的参数。它被隐式地嵌入在 System Prompt 中：

```
"仔细检查产出物是否满足需求"
"逐条验证 Plan Step 的 key"
"确保代码风格一致"
```

这些指令描述的是审查行为的目标，而不是审查行为的「深度预算」。它们假设审查 Agent 在任何上下文中都具有同等的判断力——但凡一个审查员没有 100% 的注意力，他逐条检查的细致度就和精力充沛时不同。

但 System Prompt 不知道 KV Cache 已满。System Prompt 不知道 Plan 有 15 个步骤。System Prompt 不知道当前上下文里有 3 段废弃的对话和 2 个已完成子任务的执行日志。System Prompt 只会重复同一句话：「仔细检查」。

这不是 Prompt Engineering 的问题——是审查深度的**决策框架缺失**。审查深度不应该是静态指令，应该是上下文状态的函数。

### 1.3 形式化

设审查 Agent M 在上下文 S(t) 下执行审查，审查深度为 d。d 决定了审查的细致程度：检查多少维度、验证多少约束、对边界条件的严格程度。

固定审查深度模型：

```
d = d₀（常量）
审查质量 Q = g(M 的能力, d₀, S(t))
```

当 S(t) 增长，M 的有效注意力 α_M(t) ∝ 1/|S(t)|，M 的实际判断力下降。但 d₀ 不变——M 用衰减的判断力执行不变的工作量。结果：

```
当 |S(t)| ≫ 初始 |S₀|：
  g(M 的能力, d₀, S(t)) ≪ g(M 的能力, d₀, S₀)
  但 d 不变 → 审查输出仍标记为「通过」→ 用户不知情
```

动态审查深度模型：

```
d = f(K, P)，其中：
  K = KV Cache 占用率 = |S(t)| / Context Window
  P = Plan 复杂度 = Plan Step 数量 + 步骤间的依赖关系密度
审查质量 Q = g(M 的能力, f(K, P), S(t))
```

当 K 升高时，f(K, P) 自动提高审查深度，用「更严格的审查标准」补偿「更低的判断力基线」。这不是模型变聪明了——是用更高的审查要求对冲更低的审查能力，维持净审查质量的稳定。

## 2. 现有方案与局限

### 2.1 固定审查深度的普遍性

| 方案 | 审查机制 | 审查深度如何定义 | 是否随上下文变化 |
|------|---------|----------------|:---:|
| **Claude Code** | 在生成后执行 `code_review` 工具 | System Prompt 指令：「Review the code for correctness, bugs, and style」 | 否 — 固定指令，不感知上下文长度 |
| **CodeWhale** | `plan_review` tool | 固定检查项列表：逻辑正确性、边界条件、错误处理 | 否 — 无论上下文 8K 还是 128K，检查项不变 |
| **Devin** | 内置审查流程（闭源） | 未公开，但从行为推断为固定标准 | 否 — 审查行为在长任务和短任务中表现一致 |
| **Hermes (当前)** | Momus 审查 Agent | 基于 Plan key 的逐条对比 | 否 — 审查深度由 Momus 的 System Prompt 静态定义 |
| **LangGraph Supervisor** | 监督 Agent 审查 Worker Agent 产出 | 固定审查维度（格式、内容、约束满足） | 否 — 审查维度列表在初始化时确定 |

**共性缺陷**：所有方案的审查深度都是一个**静态值**。它要么是 System Prompt 里的一段自然语言描述，要么是一个固定的检查项列表。审查深度和审查时的上下文状态之间没有任何关联——一个 128K 上下文的审查和一个 8K 上下文的审查，在审查框架看来是同一件事。

### 2.2 「让模型自己判断审查深度」为什么不行

一个自然的反驳：「让 LLM 自己判断何时需要更严格审查不就行了？在 Prompt 里加一句 `如果上下文很长，请更仔细地审查`。」

这不行。原因与 attention dilution 的根因相同：

LLM 判断「上下文是否很长」本身就需要注意力——而这个注意力也已经被长上下文稀释了。一个注意力已被稀释的模型，在 Prompt 中被要求「更仔细」，它的实际行为往往是在输出中生成更多文字（看起来很仔细），但判断的准确度并未提升。

换句话说：**让注意力被稀释的模型意识到自己注意力被稀释，这本身就是一个悖论。** 就像让一个醉酒的人判断自己是否还能开车——他的判断力已经被酒精损害了，他无法准确评估自己的清醒程度。

审查深度的切换决策必须在 Harness 层完成——不由模型判断，由 Harness 根据 KV Cache 的硬指标（实际占用 token 数、Plan 的步骤数和依赖密度）计算。

### 2.3 固定深度的三种失效模式

**模式一：短上下文过度审查**。上下文只有 8K，Plan 只有 2 步。Agent 的注意力充足，判断力处于峰值。此时仍执行「逐条验证 + 边界条件 + 回滚策略」的完全审查，是对注意力预算的浪费——用十分注意力做一个需要三分注意力就够的判断。

**模式二：长上下文审查不足**。上下文 128K，Plan 12 步，依赖关系稠密。Agent 的注意力已被稀释至基线水平。此时执行「80% 清晰就批准」的宽松审查，实际审查质量可能连「40% 清晰都判断不了」——因为它根本没看到那 40% 的不清晰处。

**模式三：审查质量不可审计**。固定审查深度导致审查质量随上下文长度单调衰减，但衰减曲线对用户完全不可见。用户看到的是「审查通过/不通过」的二元结果，不知道通过这个结果的审查质量是 90% 还是 40%。审查结果失去了可审计性。

## 3. 方案设计

### 3.1 核心函数：审查深度 = f(KV Cache 占用, Plan 复杂度)

审查深度不是一个常数——它是一个二维函数：

$$\text{ReviewDepth} = f(K, P)$$

其中：

- **K = KV Cache 占用率**：当前上下文实际 token 数 / 模型 Context Window 上限。直接反映注意力稀释程度。K ∈ [0, 1]，K 越高 → 注意力越低 → 需要更严格的审查标准来补偿。
- **P = Plan 复杂度**：当前 Plan 的步骤数 n 与步骤间依赖关系的稠密度 ρ。P = n · (1 + ρ)，其中 ρ = |E| / |V|（依赖图中的边数/节点数）。P 越高 → 出错概率越大 → 需要更严格的审查标准。

**为什么是这两个维度？**

- K 反映的是 **审查者能力衰减**——注意力被稀释后，同样的审查指令执行出来的有效质量更低。
- P 反映的是 **审查对象难度**——Plan 越复杂，步骤间相互影响越多，越容易出错。

两个维度独立但协同：当 Plan 简单（P 低）且上下文短（K 低），审查可以宽松——因为审查者状态良好且出错概率低。当 Plan 复杂（P 高）且上下文长（K 高），审查必须最严格——因为审查者状态差且出错概率高。

**为什么不是更多维度？**

更复杂的函数（加入任务类型、用户声誉、历史审查通过率等）理论上可以更精确，但成本更高——每次审查决策都需要计算和查找。二维 f(K, P) 是「足够好」与「足够简单」的平衡点：K 和 P 都是 Harness 已经持有且实时更新的指标，不需要额外计算。

### 3.2 审查深度梯度表

基于 K 和 P 定义四个审查区间，每个区间对应不同的审查深度和具体审查行为：

| K（KV Cache 占用） | P（Plan 复杂度） | 审查区间 | 审查深度 | 具体行为 |
|:---|:---|------|------|------|
| < 8K tokens<br>（K < 0.06, 128K 窗口） | 任意 | **宽松审查** | 浅层 | 验证产出物的 Key 是否满足。不检查 Step 间一致性。不检查边界条件。「80% 清晰度 → 批准」 |
| 8K - 32K<br>（K ∈ [0.06, 0.25]） | P ≤ 3 | **标准审查** | 中层 | 逐条验证 Key + 检查当前 Step 与上一 Step 的产出一致性 + 关键边界条件（输入为空、权限不足、超时） |
| 8K - 32K<br>（K ∈ [0.06, 0.25]） | P > 3 | **严格审查** | 深层 | 逐条验证 Key + 检查当前 Step 与所有依赖 Step 的产出一致性 + 完整边界条件 + 错误恢复路径 |
| 32K - 128K<br>（K ∈ [0.25, 1.0]） | P ≤ 3 | **严格审查** | 深层 | 同严格审查 + 标记「上下文已膨胀，建议压缩」 |
| 32K - 128K<br>（K ∈ [0.25, 1.0]） | P > 3 | **最严格审查** | 最深层 | 同严格审查 + 每步必须有需求映射 + 边界条件全枚举 + 回滚策略 + 强制输出「审查置信度」分数（0-1） |
| > 128K<br>（接近窗口上限） | 任意 | **拒绝审查** | — | 不审查。强制触发上下文压缩，压缩后再审查。因为 K 接近 1 时审查质量已不可靠，审查本身失去意义 |

**梯度设计的四个原则**：

**原则一：K 驱动审查深度提升，而不是降低。** 直觉上，「上下文很长 → 审查简化 → 快速通过」似乎合理——因为全面审查消耗 Token 太多。但这是错误的方向。上下文很长意味着 Agent 已经在前面的步骤中产生了大量产出——如果前面的产出有错，后面的步骤会基于错误继续构建。此时的审查不是「可以宽松」——是「必须更严格」。Token 应该花在刀刃上，而长上下文下的审查就是刀刃。

**原则二：P 放大了 K 的影响。** 同样的 K 下，Plan 复杂度 P 决定了审查深度的进一步分层。一个简单的 3 步 Plan 在 64K 上下文中的审查风险可控；一个 15 步稠密依赖的 Plan 在同样的 64K 上下文中，任何一步的遗漏都可能扩散到整个依赖图。

**原则三：超过 128K 时拒绝审查。** 这不是一个可调参数——是一个硬边界。当上下文接近窗口上限时，审查 Agent 的有效注意力已降至不可接受的基线。此时的审查结果——无论是「通过」还是「驳回」——都不可靠。正确的行为不是「降低审查深度凑合审」，而是「先压缩上下文，腾出注意力预算，再审查」。

**原则四：审查置信度是必要条件。** 在最严格审查区间，Momus 必须输出一个 0-1 的置信度分数，表示它对本次审查判断的信心水平。这不是为了让用户看——是一个反馈信号：如果 Momus 多次在 K 极高的场景下输出高置信度（>0.9），说明它没有意识到自己注意力不足——这是一个需要排查的审查系统失效信号。

### 3.3 实现路径

审查深度切换在 Harness 层的实现分为三个组件：

**组件一：上下文状态监控器（Context Monitor）**

```typescript
interface ContextState {
  kvCacheTokens: number;       // 当前实际 KV Cache token 数
  contextWindow: number;       // 模型上下文窗口上限（如 128K）
  planComplexity: number;      // Plan 步骤数 × (1 + 依赖密度)
  reviewDepth: ReviewDepth;    // 当前建议的审查深度
}

// 每次审查前调用
function getReviewDepth(state: ContextState): ReviewDepth {
  const K = state.kvCacheTokens / state.contextWindow;
  const P = state.planComplexity;
  
  if (state.kvCacheTokens > CONTEXT_WINDOW_HARD_LIMIT) return "reject";
  if (K < 0.06) return "shallow";
  if (K < 0.25 && P <= 3) return "medium";
  if (K < 0.25 && P > 3) return "deep";
  if (K >= 0.25 && P <= 3) return "deep";
  return "deepest";
}
```

**组件二：审查深度→System Prompt 的编译（Prompt Compiler）**

不同审查深度对应不同的审查 System Prompt。不是简单的「在 Prompt 里加一句更严格」——是完整的 Prompt 替换：

- `shallow` → Momus 负载最轻的 Prompt（仅 Key 验证，~200 tokens）
- `medium` → Momus 标准 Prompt（Key + 单步一致性 + 关键边界，~600 tokens）
- `deep` → Momus 严格 Prompt（Key + 全依赖一致性 + 完整边界，~1200 tokens）
- `deepest` → Momus 最严格 Prompt（严格 Prompt + 需求映射 + 回滚策略 + 置信度，~2000 tokens）
- `reject` → 不调用 Momus，触发上下文压缩流程

关键设计决策：**审查深度越高，Momus 的 System Prompt 越长。** 这看起来矛盾——在注意力已经被稀释的上下文中增加 System Prompt 长度？但审查 Prompt 不是「更多内容」，是「更具体的检查项」。一个 2000 token 的审查 Prompt 中包含了结构化的检查清单，Momus 可以逐项执行——这比一个模糊的「仔细检查」指令（200 tokens）让注意力不足的模型表现更好。结构化检查清单是注意力稀释下的最佳审查格式。

**组件三：审查深度审计日志（Review Audit Trail）**

每次审查记录以下数据：

```typescript
interface ReviewAuditEntry {
  timestamp: number;
  reviewDepth: ReviewDepth;     // 使用的审查深度
  K: number;                     // 审查时的 KV Cache 占用率
  P: number;                     // 审查时的 Plan 复杂度
  result: "pass" | "reject";     // 审查结果
  confidence?: number;           // 置信度（仅 deepest 模式）
  tokensUsed: number;            // 本次审查消耗的 Token 数
}
```

审计日志的作用：让审查质量的衰减曲线变得**可见**。当 K 从 0.05 增长到 0.8 时，审查通过率和置信度的变化趋势可以在审计日志中被追踪——用户不再面对「审查通过/不通过」的黑盒。

## 4. 分析

### 4.1 为什么动态切换能解决固定深度的三个失效模式

**对应模式一（短上下文过度审查）**：K < 8K 时进入宽松审查。不检查 Step 间一致性和边界条件——因为在注意力充足的短上下文中，Agent 自己犯这些错误的概率本来就低。宽松审查节省的 Token（约节省 300-800 tokens/次）投入到实际任务执行中。

**对应模式二（长上下文审查不足）**：K ≥ 32K 时进入严格/最严格审查。用结构化的多维度检查清单对冲注意力稀释。审查本身消耗的 Token 增加了（从 ~600 到 ~2000 tokens），但这是有必要的——因为上下文已经膨胀到「不严格审查就可能漏错」的程度。

**对应模式三（审查质量不可审计）**：审计日志记录了每次审查的 K、P、depth、result、confidence。用户可以事后分析在哪些上下文状态下审查通过率异常高（→ 可能审查过于宽松），哪些状态下置信度异常低（→ 可能需要触发压缩）。

### 4.2 为什么「动态审查深度」不是「增加更多审查」

一个容易混淆的点：动态审查深度听起来像是「在长上下文中做更全面的审查」——这不就是在长上下文中消耗更多 Token 吗？

是的。但关键差别是**什么时候消耗这些 Token**：

- 固定审查深度：在短上下文和长上下文中消耗**同样的审查 Token**（~600 tokens/次），但长上下文中的 **Token 有效产出**更低（因为注意力稀释）
- 动态审查深度：在短上下文中消耗**更少**的审查 Token（~200 tokens/次），在长上下文中消耗**更多**的审查 Token（~2000 tokens/次），但长上下文中的**每 Token 有效产出**由结构化检查清单保证

动态审查的总 Token 消耗不一定比固定审查更低——可能更高。但它的**有效审查质量**是稳定的（或至少不会随 K 单调衰减），而固定审查的质量随 K 单调衰减。Token 效率不是「绝对值」，是「单位审查质量的 Token 消耗」。

### 4.3 边界条件

以下场景本方案**无法**覆盖：

- **KV Cache token 数不可精确获取**：部分模型 API（如 GPT-4 的 Chat Completions API）不直接暴露 KV Cache token 数，只暴露 Prompt tokens + Completion tokens。此时需要用 `prompt_tokens` 作为 K 的近似代理——在大多数情况下足够准确，但在有 prefix caching 的场景下可能低估实际 K。
- **Plan 复杂度 P 的准确性问题**：P 的计算依赖 Plan 的依赖图质量。如果 Plan 的 `dependency_ids` 标注不全（如 article 06 中讨论的），P 会被低估 → 审查深度不够。不过，P 被低估的后果是「审查偏松」而非「审查偏严」——在安全敏感的场景下是一个需要关注的风险。
- **K 恰好处于阈值附近时的频繁切换**：如果 KV Cache token 数在 8K 附近波动（±500 tokens），审查深度可能在 shallow 和 medium 之间反复切换，导致 Momus 的审查行为不一致。解决方案：引入阈值滞后（hysteresis）——从 shallow 切换到 medium 的阈值是 K > 0.0625（8K tokens），但从 medium 切换回 shallow 的阈值是 K < 0.05（6.4K tokens），避免边界抖动。
- **不同模型的 Context Window 不同**：K 是一个相对比例。但 32K / 128K 和 32K / 1M 的 K 相同（0.25），实际的注意力稀释程度是否相同？对于 1M 窗口的模型，32K 可能仍然在「注意力充足」区间。这需要针对不同模型窗口大小校准 K 的阈值，而不是使用统一的绝对 token 阈值。

## 5. 验证路径

### 5.1 已验证

- **问题存在性（间接验证）**：article 03 的注意力预算实验已验证，同一 DeepSeek V4 Pro 模型在第 15 轮对话时约束保持率从 ~100% 降至 ~40%。审查 Agent 的注意力衰减遵循相同的物理规律——Momus 在长上下文中的审查判断力衰减和主 Agent 的任务执行能力衰减是同源的。
- **审查梯度的可行性（理论基础）**：Attention Budget Management 框架（article 03 §3.2）已提出 f(K, P) 的二维函数概念和初步梯度表。本文在此基础上将审查维度的梯度表细化到四个区间 + 具体的审查行为定义。
- **Harness 可获取的指标**：Hermes 的 KV Cache 监控已产出 `kv_cache_tokens` 和 `context_window` 两个指标；OKR PlanStep（article 06）已产出 `dependency_ids` 和步骤计数——P 的计算所需数据已就绪。

### 5.2 待验证

- **K 阈值的经验校准**：当前阈值（<8K / 8-32K / 32-128K / >128K）是基于 128K Context Window 的启发式设定。需要在实测中验证：(a) 各阈值下 Momus 的实际审查准确率变化曲线；(b) 不同任务类型（重构/新建/调试）下阈值是否需要调整。
- **审查深度切换对净审查质量的影响**：核心假设是「更深度的审查 Prompt 可以补偿注意力稀释」。需要对照实验：固定审查深度 vs 动态审查深度，在 K 从 0.05 到 0.8 的梯度上，测量两种方案下的审查召回率（漏报数）和精确率（误报数）。
- **审查 Token 消耗的经济性**：动态审查在 K 高时消耗更多审查 Token。需要测量：(a) 动态审查的全任务 Token 总消耗 vs 固定审查的 Token 总消耗；(b) 单位审查质量的 Token 消耗对比。
- **置信度分数的校准**：Momus 在最严格审查区间输出的置信度分数（0-1）是否与其实际审查准确率相关？如果置信度和准确率解耦（高置信度 + 低准确率），说明 Momus 存在「过度自信」偏差——需要进一步校准或引入第二审查 Agent。
- **跨模型泛化**：DeepSeek V4 Pro 的注意力稀释曲线是否适用于 Qwen、Llama、Claude 等模型？不同架构的 Transformer 在相同 K 下的注意力分布可能不同 → 阈值需要模型特定的校准。

## 6. 与 Hermes 的关系

### 6.1 Momus 为什么需要这个方案

Momus 的当前实现是一个「诚实但盲目」的审查者——它尽职地执行审核，但从未问过自己：「我现在的判断力还够吗？」

动态审查深度切换对 Momus 的改造分为两步：

**第一步：嵌入 Context Monitor。** Momus 在被调用审查前，Harness 先查询 Context Monitor → 获取当前 (K, P) → 计算审查深度 → 编译对应的审查 Prompt → 传入 Momus。Momus 不需要知道审查深度切换的存在——它只需要收到一个更具体或更宽松的 Prompt。

**第二步：审计反馈回路。** Momus 的审查结果（通过/驳回 + 置信度）写入审计日志。Harness 定期分析审计数据，检测异常模式（如高 K 下通过率异常高）→ 触发审查深度阈值的自动调参或通知用户。

### 6.2 与 article 03 的分工

article 03（注意力预算管理）定义了四条注意力管理策略，其中审查维度（§3.2）是四条策略之一。本文是审查维度的**专项深化**：

- article 03 做的事：提出 f(K, P) 概念 + 初步 5 行梯度表
- 本文做的事：将梯度表扩展为 6 区间的完整审查行为定义 + 审查深度→Prompt 的编译机制 + 审计日志 + 失效模式分析 + 边界条件（滞后、跨模型校准）

两者关系：article 03 是框架，本文是框架中审查模块的详细设计文档。

### 6.3 与 article 06 的协同

article 06（OKR PlanStep + 级联修正引擎）定义了 P 的计算基础——`dependency_ids` 和 `association_strength` 让 Plan 复杂度 P 不再是简单的步骤计数 n，而是 n · (1 + ρ)，其中 ρ 考虑了依赖图结构。

两者的协同关系：

- article 06 的依赖图 → article 07 的 P 维度
- article 07 的审查深度 → article 06 的级联修正中的审查环节（当节点 marked as `pending_review` 时，Momus 的审查深度应由 f(K, P) 决定）
- 当级联修正引擎标记了 5 个 `pending_review` 节点时，每个节点的审查深度不是统一的——而是根据审查时的 (K, P) 独立计算

## 结论

Momus 的「80% 清晰就批准」在短对话中是高效启发式，在长对话中是系统性自欺。不是因为 Momus 不努力——是因为审查深度是静态的，而审查者的判断力是动态衰减的。

审查深度 = f(KV Cache 占用, Plan 复杂度) 将审查从一个「固定标准」升级为一个「状态感知系统」。当上下文膨胀、注意力稀释时，审查不放松——审得更深。用结构化的多维度检查清单对冲注意力的衰减，用审计日志让审查质量从黑盒变为可见曲线。

这不是让 Momus 变聪明——是让 Momus 知道什么时候自己不够聪明，并用更强的审查框架来补偿。
