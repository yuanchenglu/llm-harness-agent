> **摘要**：Agent 产出的文档随迭代不断膨胀，每次新 session 读取时全文塞入上下文——Token 消耗线性增长，但核心结论仅占全文的 ~30%。本文提出将 KV Cache 前缀复用原理应用到文档结构上：把「每次都要读的」核心结论固化在前缀区（YAML 元数据 + TL;DR + 核心论点），把「偶尔才需要的」讨论过程放在尾部变化区。追加新内容时前缀区的 KV Cache 全命中，仅重新计算尾部 Token。理论 Token 节省率 > 80%，不损失信息完整度——改变的只是排列顺序。

---

## 1. 问题定义

### 1.1 现象

你用 Agent 写了一篇 15KB 的架构分析文档。三天后开新 session，让 Agent 参考这篇文档继续工作——Agent 把 15KB 全文塞进上下文。你追加了 10KB 讨论附录，文档变成 25KB。下次 Agent 再读，又是 25KB 全进去。循环往复。

每次读取的 Token 消耗随文档大小线性增长。但文档里对当前任务真正有用的信息——核心结论、关键决策——通常只占 30%。另外 70% 是讨论过程、边界探讨、Q&A 附录——有价值但不直接相关。

这是 Agent 开发中最隐蔽的成本漏斗：**文档膨胀 → 上下文浪费 → 有效智能下降**。不是模型变差了，是喂给模型的信号被噪声稀释了。

### 1.2 根因

根因不在 Agent 的读取策略，在文档的书写规范。

当前所有文档规范都是「给人看的」——标题、引言、正文、子标题、代码块、总结。结构服务于人类阅读习惯：先铺垫背景，再展开论述，最后总结。但今天的文档主要读者已经不是人了——是 Agent。Agent 重新读取自己产出的文档时，它在乎的不是排版好不好看，而是**前 500 tokens 能不能告诉我这篇文章的核心结论是什么**。

传统「前言→正文→总结」结构下，Agent 要读 2KB 才能触达核心结论。前言里「随着 AI 技术的快速发展，Agent 架构变得越来越重要……」这类句子对 Agent 是纯粹的 Token 浪费。

更深层的根因：**KV Cache 的缓存粒度是「前缀连续匹配」。** 传统文档结构中，最稳定的信息（标题、元数据）和最不稳定的信息（讨论记录、Q&A）混排在一起。每次追加讨论，前缀就可能被打断，已有的 KV Cache 全部失效——Agent 必须重新编码全文。

### 1.3 形式化

设文档大小 S，其中核心结论区大小为 U（通常 U ≈ 0.3S），讨论/附录区大小为 D = S - U。Agent 第 i 次读取时消耗 Token 数 Tᵢ：

**传统结构**：Tᵢ = S（每次全量编码）。N 次读取总消耗 C = N·S。

**优化结构**（稳定前缀 + 变化尾部）：首次读取 T₁ = S（冷启动）。后续读取时，前缀区 U 的 KV Cache 命中——仅编码尾部变化区 D。Tᵢ ≈ D（i ≥ 2）。N 次读取总消耗 C' = S + (N-1)·D。

Token 节省率 = (C - C') / C = (N-1)·(S-D) / (N·S) = (N-1)·U / (N·S) ≈ U/S ≈ 70%（当 N ≥ 5 时趋近）。

关键洞察：**节省率的来源不是压缩——文档内容完整度不变。来源是排列顺序**：把稳定的放前面，变化的放后面。

## 2. 现有方案与局限

| 方案 | 核心思路 | 为什么不行 |
|------|---------|-----------|
| **人类阅读规范** | 前言→正文→总结，为人类阅读优化 | Agent 是主要读者。前言是噪音，核心结论被埋在正文中段 |
| **全量重读** | 每次 session 完整读取文档 | Token 消耗随文档膨胀线性增长，无上限 |
| **文档摘要作为 System Prompt** | 每次读取前先让 Agent 生成摘要，只把摘要注入上下文 | 摘要生成本身消耗 Token。且摘要和原文之间的信息损失不可控 |
| **RAG 检索式读取** | 按 query 检索文档片段而非全文 | 增加了检索系统复杂度。检索不准时遗漏关键信息。且「Agent 需要知道整篇文档说了什么」的场景无法用片段满足 |
| **上下文压缩** | 压缩历史对话腾出空间 | 压缩算法用统一权重评估所有内容。核心结论和讨论过程混在一起——压缩后丢了什么不可知 |

**共性缺陷**：所有方案都在试图「读得更少」。正确方向应该是「让少读的内容恰好就是需要的内容」——不是压缩，是重排。

## 3. 方案设计

### 3.1 核心原则：稳定在前，变化在后

KV Cache 的工作机制：Transformer 推理时，已计算过的前缀 Token 的 Key-Value 向量被缓存。后续生成时，如果前缀不变，直接从缓存读取——不消耗额外计算。只有变化区的 Token 需要重新计算。

将这个原理平移到文档结构上：

```
传统文档结构：                     KV Cache 优化结构：
  [标题]                             [YAML 元数据]       ← 前缀区（永久不变，全命中）
  [前言/背景]                         [TL;DR 结论]        ← 核心区（极少变更，高命中）
  [正文/分析]                         [核心论点]           ← 核心区
  [讨论过程]                          ═══════════════
  [附录/Q&A]                         [详细论证]           ← 正文区
                                      [讨论过程]           ← 变化区（追加只影响尾部）
                                      [Q&A 附录]          ← 变化区
```

**前缀区**（YAML 元数据 + TL;DR + 核心论点）：首次创建时写入，后续不修改。Agent 每次读取时 KV Cache 全命中——零计算。

**正文区**（详细论证）：偶尔修改，频率远低于讨论区。KV Cache 高命中。

**变化区**（讨论 + Q&A）：每次对话都可能追加新内容。但它们在文件末尾——追加时只重新计算尾部 Token，前缀不受影响。

这不是压缩，是重排。文件大小不变，信息完整度不变。唯一的改变是：**把每次都要读的放在前面，偶尔才需要读的放在后面。**

### 3.2 层级定义

文档被划分为三个逻辑层，按稳定度排序：

| 层级 | 内容 | 变更频率 | KV Cache 命中率 | 典型大小占比 |
|------|------|:---:|:---:|:---:|
| **L1 前缀层** | YAML 元数据、TL;DR、核心结论列表 | 几乎为零 | ~100% | ~5% |
| **L2 正文层** | 详细分析、论证过程、数据支撑 | 偶尔（修改/补充） | ~90% | ~45% |
| **L3 变化层** | 讨论记录、Q&A 附录、迭代笔记 | 频繁（每次对话追加） | ~0%（每次新增尾部） | ~50% |

L1 前缀层是 KV Cache 优化的关键。它必须满足两个约束：

1. **不可变性**：写定后不再修改。如果核心结论真的变了——不是追加，是重构全文档。
2. **自描述性**：Agent 读完前缀层就应该知道「这篇文章讲什么、核心结论是什么、和哪些创新点相关」——不需要继续往下翻。

YAML 元数据示例：

```yaml
---
title: Plan 架构深度分析
created: 2026-05-25
innovation_points: [I-01, I-03, I-04]
kv_cache_stable: true
---
```

`kv_cache_stable: true` 是给 Agent 的信号：「这个文档的前缀区是稳定的，放心缓存，核心结论在读前 500 tokens 就能获取。」

### 3.3 真实例子

以「Plan 架构深度分析」文档为例，展示重构前后的结构差异。

**重构前**（传统结构，~40KB）：

```markdown
# Plan 架构深度分析
> 背景：随着 Agent 技术的快速发展，Plan 作为任务规划模块的重要性日益凸显。
> 本文将从多个维度对 OMO 的 Plan 架构进行深度分析……

## 1. 引言
...500 字段落，铺垫 Plan 在 Agent 系统中的定位...

## 2. Plan vs Reason：核心差异
...核心结论被埋在第二节，Agent 需要读 ~2KB 才能到达...

## 3. 依赖图分析
...

## 4. 讨论与 Q&A
...讨论记录逐次追加...
```

**重构后**（优化结构，同为 ~40KB）：

```yaml
---
title: Plan 架构深度分析
created: 2026-05-25
innovation_points: [I-01, I-03, I-04]
kv_cache_stable: true
---
```

```markdown
# Plan 架构深度分析

> **TL;DR** — Plan 是发散性问题，Reason 是收敛性问题。
> First Principles 拆解是连接两者的桥梁。

## 核心结论
1. OMO 的 Plan 是扁平 checklist，无依赖图
2. 级联修正引擎必须基于有向依赖图
3. 审查严格度应 = f(KV Cache 占用, Plan 复杂度)

---

## 详细分析
[→ 传统正文内容，完整保留]

## 讨论附录
[→ Q&A 讨论记录，按时间追加]
```

效果对比：

| | 传统结构 | 优化结构 |
|---|---|---|
| 文件大小 | 40KB | 40KB（相同） |
| Agent 首次读取 | 40KB | 40KB |
| Agent 再次读取 | 40KB（全量重编） | ~5KB（前缀层命中） |
| 追加讨论后再读 | 42KB（全量重编） | ~7KB（前缀命中 + 新增讨论） |
| Token 节省率 | — | **> 80%** |
| 信息完整度 | 100% | 100%（未删减任何内容） |

Agent 读前 500 tokens 就已经知道：这篇文章讲什么、核心结论是什么、和哪些创新点相关。如果当前任务只需要核心结论——直接跳到执行，不需要把 40KB 全读完。

## 4. 分析

### 4.1 为什么这个方案能解决根本问题

根本问题不是「Agent 读得太慢」，是「Agent 在为不变的内容重复支付 Token 成本」。

本方案把文档从「无差别的文本流」改造为「按稳定度分层的 KV Cache 友好结构」。核心机制：

1. **前缀层固化**：L1（元数据 + TL;DR + 核心结论）写定后不再修改。Agent 系统读取时，前缀 Token 的 KV 向量被缓存——这是 Transformer 推理引擎的底层行为，不需要额外代码。
2. **尾部追加隔离**：所有增量内容（讨论、Q&A）只追加到文件末尾。追加不改变前缀——已缓存的 KV 向量继续有效。
3. **信号前置**：Agent 在读取的前 500 tokens 内就获取全部核心信息。不需要「翻到第二节」才能找到结论。

本质是一个**信息排列优化问题**，不是压缩问题。排列顺序的改变让 KV Cache 机制自然生效，不引入任何新系统。

### 4.2 边界条件

以下场景本方案**无法**覆盖或需要额外注意：

- **核心结论变更**：如果文档的核心结论被推翻重写，前缀层必须修改。这会导致 KV Cache 全量失效——本次读取退化到传统结构。但这种情况的频率远低于讨论追加，平均收益仍然显著。
- **文档结构整体重构**：如果文档需要从「分析报告」变为「操作手册」——不再是追加讨论而是全量重写——前缀层自然失效。这不是本方案的设计目标场景。
- **单次读取场景**：Agent 只读一次就不再读的文档，优化结构无收益（首次读取消耗相同）。收益在「多次跨 session 读取」场景中体现。
- **Agent 系统不支持 KV Cache**：如果底层推理引擎完全不缓存 KV 向量（如某些无状态 API 部署），前缀优化无效——但 Agent 仍然受益于「信号前置」带来的阅读效率提升。
- **多人协作文档**：多人同时追加讨论时，尾部可能产生冲突。需要引入文件锁或每人独立追加区的协作策略。

### 4.3 与最接近方案的对比

| 维度 | RAG 检索 | 文档摘要 | 上下文压缩 | 本方案 |
|------|:---:|:---:|:---:|:---:|
| 原理 | 按 query 取片段 | 生成短摘要替代原文 | 统一压缩 | 按稳定度重排 |
| 信息损失 | 检索不准时丢 | 摘要必然丢 | 不可控 | 零 |
| 额外系统 | 需要向量库+索引 | 需要额外 LLM 调用 | 需要压缩器 | 无 |
| 读取效率 | 取决于检索质量 | 摘要 Token 低 | 压缩后 Token 低 | 前缀 Token 低 |
| 适用场景 | 「找」特定信息 | 「了解」大概内容 | 「节省」上下文 | 「重复读」文档 |

本方案不替代 RAG 或摘要——它们是不同场景的工具。本方案针对的场景是：**Agent 反复读取自己产出的长文档，每次只需要核心结论，但偶尔需要深入细节。** 这个场景下，重排比压缩更合适。

## 5. 验证路径

### 5.1 已验证

- **KV Cache 前缀复用原理**：Transformer 推理引擎的前缀缓存机制已被业界广泛验证（vLLM、SGLang 等推理框架的 Prefix Caching 特性）。
- **问题存在性**：实际 Agent 使用中，文档随对话迭代从 15KB 膨胀到 40KB+ 是常态。每次 session 的 Token 浪费可量化。

### 5.2 待验证

- **Token 节省率实测**：在支持 Prefix Caching 的推理环境中（如 vLLM），对比传统结构和优化结构在 10 次跨 session 读取中的实际 Token 消耗差异。预期节省率 > 80%。
- **信号前置有效性**：Agent 仅读取前缀层（前 500 tokens）vs 读取全文——在「基于文档继续工作」任务上的任务完成质量对比。预期差异不大（因为前缀层已包含核心结论）。
- **开发者采纳门槛**：文档作者需要改变书写习惯——先写核心结论，再写详细论证。这种「倒金字塔」写法对人来说是否可接受、是否会增加书写负担。
- **自动化工具链**：能否开发一个 lint 工具，自动检查 Agent 产出文档是否满足 KV Cache 优化结构（前缀层存在性、核心结论前置、变化区在尾部）。
- **跨框架兼容性**：不同推理框架（vLLM、SGLang、TensorRT-LLM）对前缀缓存的支持粒度不同——需要验证本方案在各框架下的实际收益。

## 6. 与 Hermes 的关系

Hermes 的 System Prompt 已经实现了「稳定在前、变化在后」的设计：Base System Prompt（永久稳定）→ Skills（按需加载）→ Memory（动态注入）。这是一个三层 KV Cache 友好架构——每次推理时，Base System Prompt 的 KV 向量被完全缓存，仅尾部 Memory 需要重新编码。

本方案把这个理念从 System Prompt 层面扩展到**文档产出层面**。Hermes 产出的所有长文档——无论是分析报告、设计方案、还是讨论总结——都遵循「前缀 YAML + TL;DR + 核心结论」的结构。

这形成了一个完整的 KV Cache 优化闭环：

```
System Prompt 层：Base（稳定）→ Skills（半稳定）→ Memory（动态）
        ↓ 同一设计理念
文档产出层：   L1 前缀（稳定）→ L2 正文（半稳定）→ L3 变化（动态）
        ↓ 相互叠加
KV Cache 收益：前缀全命中 × 两层 = 从「怎么读」到「怎么写」全链路优化
```

Hermes 不需要任何架构变更来支持本方案——只需要在文档生成指令中加入「按 KV Cache 优化结构输出」的约束。收益自动体现在后续 session 的上下文消耗中。

---

## 结论

Agent 文档的优化方向不是「写得更短」，是「把不变的信息放在前面」。KV Cache 前缀复用的原理不只在推理引擎里有效——把它应用到文档产出格式上，Agent 的生态就能自我优化。从 System Prompt 到文档产出，同一个设计理念贯穿全链路：**稳定在前，变化在后**。
