## 2. 设计模式

| 模式 | 采用方式 | 证据 |
|---|---|---|
| Evidence Ledger | P0 必须实现 | Stage 2/2.5 trace 与 E4 rows 证明没有证据会误报 |
| Cache-first Context Compiler | P0 采用，可观测而非硬依赖 | E3 Flash/Pro cache hit rate 约 97%–99% |
| Permission + Sandbox + Diff + Rollback | P0 采用 | 本仓库 security tests 与 E4 adapter 使用 ChangeManager |
| Deterministic Verifier | P0 采用 | E4 显示 exact following 不稳定，必须外部验收 |
| Runtime Adapter Boundary | P0 采用 | current/OpenCode 可通过统一接口比较 |
| Multi-agent Team | P1 延后 | Stage 6 单 agent 仍未 100%，先不扩大变量 |
| Skill/Memory Evolution | P1 延后 | 缺负反馈与遗忘策略 |

## 3. 反模式

| 反模式 | 决策 |
|---|---|
| 只看自然语言“完成了” | 拒绝；必须 verifier 通过 |
| 复制某个竞品整体架构 | 拒绝；DeepSeek 协议与 cache 行为需要 native adapter |
| 先 fork 大 runtime 再补协议 | 拒绝；会混淆 provider bug 与 runtime bug |
| 把 cache hit 当作质量提升 | 拒绝；cache 是成本/延迟优化，不是 correctness |
| 把 mock 20/20 写成模型成绩 | 拒绝；mock 仅验证 fixture/verifier |

## 4. 借鉴 / 改造 / 拒绝矩阵

| 能力 | 决策 | 原因 |
|---|---|---|
| OpenCode 多端 runtime | 改造/候选 | installed CLI smoke 可加载 provider，但固定源码 live probe 缺 checkout |
| Codex permission/sandbox | 借鉴 | 与本仓库安全层一致 |
| Claude compaction | 借鉴 | 长上下文必需，但需保留 DeepSeek cache stable prefix |
| OMO intent gate | 改造 | 可用于 routing，但不能破坏 prefix 稳定 |
| OMC team state | 延后 | 多 agent 会放大失败面 |
| OpenSpec artifact flow | 借鉴 | 与 Evidence Ledger/PRD trace 对齐 |
| Superpowers skill discipline | 借鉴 | 作为工程流程而非 runtime 核心 |
