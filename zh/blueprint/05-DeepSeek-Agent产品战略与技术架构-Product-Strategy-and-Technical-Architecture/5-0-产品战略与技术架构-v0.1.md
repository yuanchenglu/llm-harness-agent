# Stage 4：DeepSeek Agent 产品战略与技术架构 v0.1

> 证据状态：假设性架构 v0.1；ADR 是当前最小风险决策，不代表已通过真实任务 ROI 验证。

## 产品定位

面向需要成本可控、行为可审计的代码任务，提供 DeepSeek-native Agent Runtime。核心差异不是“更多 Agent”，而是**协议正确、缓存可观测、工具有边界、结论有证据**。

## 目标用户与价值

- 个人开发者：使用 Flash 完成低成本仓库阅读与简单任务；
- 团队：获得统一 trace、权限和验收证据；
- Harness 研究者：可重复验证 V4 协议与缓存假设。

## 分层架构

```text
CLI / future UI
  -> Orchestrator (step limit, checkpoint, routing later)
  -> Tool & Safety Runtime (workspace boundary, permissions later)
  -> Context Compiler (stable prefix, history, reasoning policy)
  -> DeepSeek Provider Adapter (capabilities, raw HTTP, usage)
  -> Evidence Ledger (redacted traces, benchmark summaries, ADRs)
```

## ADR

1. **ADR-001 Raw HTTP first**：避免 SDK 静默改写影响协议实验；后续 SDK 仅作为兼容层。
2. **ADR-002 Reasoning policy**：tool-loop 默认 replay；普通跨 user turn允许 drop；按 model/mode capability 覆盖。
3. **ADR-003 Cache correctness first**：稳定前缀是优化，不得阻止必要上下文更新。
4. **ADR-004 Read-only MVP**：在审批、sandbox 和 rollback 完成前，不开放模型驱动写文件/执行 shell。
5. **ADR-005 No immediate fork**：先以小型自研 runtime 明确 DeepSeek-specific contract，再评估 OpenCode adapter/fork。

## 成功指标

P0：协议错误率、任务成功率、测试通过率、trace 完整率；P1：cache hit tokens、成本、延迟；P2：人工干预与用户体验。成本优化不能以降低成功率为代价。
