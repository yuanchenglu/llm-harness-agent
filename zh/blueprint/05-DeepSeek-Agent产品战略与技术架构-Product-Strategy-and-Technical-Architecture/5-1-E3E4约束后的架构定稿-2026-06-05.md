# Stage 4：E3/E4 约束后的架构定稿

> 状态：v1.0。该版本定稿研究 MVP 与生产 MVP 的技术边界；生产发布仍受 Stage 6 release gate 约束。

## 1. 最终产品定位

DeepSeek Agent 是 **DeepSeek-native evidence-first coding runtime**，不是通用聊天壳，也不是多 agent 数量竞赛。核心承诺：

1. 协议正确：thinking、tool loop、streaming、invalid schema、reasoning replay/drop 有实证矩阵；
2. Cache 可观测：context compiler 输出稳定前缀与 hit/miss 证据；
3. 副作用可控：所有写操作通过 permission、sandbox、diff preview、rollback；
4. 完成可验证：任务通过 deterministic verifier，而不是模型自述。

## 3. Flash / Pro 策略

E3 显示两者协议主体均可用，invalid schema 均稳定 400，cache 均可观测；E4 tiny-fixture 显示 Flash 18/20、Pro 17/20。结论：

- 默认首轮使用 Flash；
- Pro 不应被假设为更 deterministic；
- 对新增文件、多文件和 validation 类任务，必须使用 verifier feedback retry；
- Pro review 可作为 P1，而不是 P0 release 前提。

## 4. Runtime 底座 ADR

当前选择：**小型自研 runtime + OpenCode adapter 兼容层**。

理由：

- 自研 runtime 已覆盖 DeepSeek-specific E3、security、session、E4 adapter；
- OpenCode installed CLI smoke 证明 `deepseek-direct` provider 可加载，但固定源码 live probe 当前缺完整 checkout，不能作为最终底座证据；
- 直接 fork 会引入 provider/config/session/UI 多重变量，不利于科研归因。

触发重新评估的条件：

- fixed-source OpenCode checkout 可复现 live probe；
- OpenCode adapter 在同一 E4 任务集上达到或超过 current runtime；
- 维护成本和 DeepSeek protocol patch 面小于自研 runtime。

## 5. Release 边界

研究 MVP 可以继续推进；生产发布不得开放 unrestricted shell、网络、git mutation 或无 rollback 写操作。Stage 6 release gate 需要更大任务集、多次运行、兼容矩阵和密钥轮换后才能关闭。
