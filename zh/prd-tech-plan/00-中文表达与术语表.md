# 00. 中文表达与术语表

## 1. 使用目的

本文档用于统一 DeepSeekAgent 0.1.x 产品化阶段的中文表达。

目标不是把科研文档写成营销文案，而是在不破坏事实、证据和边界的前提下，让产品、工程、研究和早期开源试用者都能看懂：

- 当前项目处于什么阶段。
- 下一步为什么要做。
- 一个术语在项目推进里意味着什么。
- 哪些是事实，哪些只是目标、假设或待验证项。

## 2. 中文表达规则

确定：

- 项目中文文档默认使用简体中文说明问题、判断和下一步。
- 技术词可以保留，但第一次出现时要解释它在项目推进里的含义。
- 能用清楚中文表达的，不用堆叠英文术语。
- 涉及命令、文件名、分支名、API 字段、模型名和 release artifact 时，保留原词。
- 结论必须区分 `确定`、`大概率(%)`、`不确定`。
- 不把“目标”写成“已完成事实”。

推荐写法：

```text
release artifact：发布产物，也就是别人可以拿来安装或验证的文件。
```

不推荐写法：

```text
我们需要先补齐 artifact、manifest、checksum、smoke、diagnostics。
```

如果必须使用多个术语，应先说明一句中文结论：

```text
下一步不是加功能，而是补齐公开试用前的发布材料：安装文件、文件指纹、构建清单、最小验证和故障诊断说明。
```

## 3. 产品化阶段一句话说明

当前项目不是“重新找方向”，也不是“继续堆功能”。当前重点是把已经成立的 DeepSeek-native runtime foundation 包装成别人能安装、能验证、能诊断、能复现的 0.1.x 公开试用形态。

更短地说：

```text
先证明别人能拿到发布产物并跑通核心链路，再谈继续扩展功能。
```

## 4. 0.1.x 产品化术语表

| 术语 | 通俗解释 | 为什么重要 |
| --- | --- | --- |
| `develop` | 当前开发分支，可以理解为正在集成新工作的主工作线。 | 后续改动默认放在这里，避免分支混乱。 |
| `origin/develop` | 远端服务器上的 `develop`。 | 用来判断本地代码有没有同步到远端。 |
| `ahead 3` | 本地比远端多 3 个 commit。 | 说明代码已提交到本地，但还没 push。 |
| `commit` | 一次代码或文档变更快照。 | 方便回看、回滚、审查每一步改了什么。 |
| `production gate` | 发布前必须通过的总验收关卡。 | 它不是单个功能点，而是“能不能公开试用”的准入标准。 |
| `gate still open` | 发布关卡还没关闭。 | 说明还不能说 0.1.x 已经发布就绪。 |
| `release artifact` | 可交付的发布产物，例如安装包、wheel 文件或压缩包。 | 用户或测试者需要拿它来安装，而不是直接用源码。 |
| `checksum` | 文件指纹，常见是 `sha256`。 | 用来证明发布产物没有损坏或被篡改。 |
| `build manifest` | 发布产物清单。 | 记录文件名、版本、checksum、构建时间、来源 commit 等。 |
| `permission` | 权限控制。 | Agent 做危险操作前必须知道什么能做、什么不能做。 |
| `diff` | 文件改动差异。 | 用户需要先看到“会改什么”，再决定是否允许。 |
| `rollback` | 回滚。 | 如果改坏了，能恢复到改动前状态。 |
| `session resume` | 会话恢复。 | 中断后可以继续任务，不必从头再来，也不能重复已完成副作用。 |
| `route` | 模型路由。 | 说明这次为什么选 Flash 或 Pro。 |
| `cache` | 缓存命中情况。 | DeepSeek 的成本和速度受 cache 影响很大，但 cache 不能代替正确性。 |
| `usage` | token 使用量。 | 用来估算消耗和分析效率。 |
| `cost` | 成本。 | 产品必须让用户知道大概花了多少钱，尤其是每成功任务成本。 |
| `smoke test` | 冒烟测试，也就是最小可用性检查。 | 不证明全部没问题，只证明核心链路没断。 |
| `public docs` | 面向公开用户的文档。 | 用户需要知道怎么安装、配置、诊断、报错。 |
| `security` | 安全说明。 | 解释密钥、文件、权限、隐私怎么处理。 |
| `troubleshooting` | 故障排查文档。 | 用户遇到问题时能自己定位，或提交有效 issue。 |
| `runtime` | Agent 的核心执行引擎。 | CLI、桌面端和自动化入口都只是入口，真正干活的是 runtime。 |
| `evidence` | 证据记录。 | 完成不能只靠自然语言声明，要有输入、工具、diff、测试、审批、成本和结果记录。 |
| `completion rate` | 任务完成率。 | 这是产品目标和 gate 指标，不能在未验证前写成已经达成的事实。 |
| `cost per successful task` | 每成功任务成本。 | 省 token 只有在任务成功、可控、可恢复时才有意义。 |


## 5. Blueprint 文档怎么读

Blueprint 是证据库和阶段真相源，不是面向产品经理的轻量说明书。

阅读原则：

- 先看当前阶段和 gate 状态，再看具体实验或历史材料。
- 不改写历史证据正文，只在入口处补导航、术语和阅读路径。
- 对每个关键结论保留来源、置信度和验证方式。

如果只是判断当前产品下一步，看 PRD TechPlan 和 release gate；如果要追溯为什么这么判断，再回到 blueprint。
