# 11. 中文 B 端桌面 UI 重构计划

## 1. 问题定义

确定：当前桌面端 UI 仍然像工程诊断面板，而不是面向财务、行政、产品经理、律师的 B 端工作客户端。默认英文、常驻 `Task Center` / `Task Detail` / `Diagnostics`、route/cache/cost/JSON 暴露和 `Code` 优先入口，会迫使非技术用户理解 runtime、bridge、workspace、diff 等工程概念。

本计划的目标是把 DeepSeekAgent Desktop 从研究/开发者主导的工作台，重构为默认简体中文的本地 B 端任务客户端，同时保留审批、回滚、诊断和 code/diff 高级能力。

置信度：确定。依据：当前 `apps/desktop/src/App.tsx`、`apps/desktop/DESIGN.md`、`openspec/specs/consumer-desktop-ui/spec.md` 与用户给出的 Kun/Quinn 参考截图。

## 2. 产品设计部计划

### 2.1 用户与定位

目标用户：

- 财务人员：整理发票、报销、费用、付款节点和审计材料。
- 行政人员：整理会议纪要、制度文档、采购记录、资产台账和通知材料。
- 产品经理：整理需求、PRD、竞品材料、评审意见和行动项。
- 律师：整理合同摘要、条款风险清单、材料清单和事实时间线。

默认用户不是服务器运维人员，不要求理解 runtime、bridge、SSE、route、cache、ChangeSet 或 JSON evidence。高级诊断能力必须存在，但不能成为首屏主内容。

### 2.2 信息架构

桌面壳采用 Kun/Quinn 式结构：

```text
左侧侧栏
├─ 项目
├─ 任务列表
├─ 技能
├─ 自动化
└─ 设置

中间工作区
├─ 当前任务 / 空态
├─ 角色模板入口
├─ 任务过程与结果流
└─ 底部主输入框

右侧抽屉
├─ 任务详情
├─ 审批与回滚
├─ 错误与恢复路径
└─ 高级诊断
```

首屏只回答三个问题：

1. 我现在在哪个项目里？
2. 我可以让它做什么？
3. 点哪里开始？

### 2.3 页面方案

必须覆盖以下状态：

- 首次打开 / 未选择项目：显示“选择项目文件夹”和四类模板；运行按钮禁用。
- 已选择项目但无任务：显示项目名、任务模板、底部输入框。
- 新建任务：模板可填充草稿，用户确认后提交。
- 四类角色模板：财务、行政、产品经理、律师。
- 任务运行中：显示中文状态、可停止/查看详情，不显示 JSON。
- 任务完成：显示结果摘要、来源、下一步操作。
- 需要审批：右侧抽屉展示 diff、风险说明、`批准一次`、`拒绝`。
- 失败与恢复：展示失败原因和恢复建议，不展示 raw stack 作为首屏内容。
- 设置与诊断：版本、运行状态、数据位置、诊断生成和高级详情。

### 2.4 默认模板

财务：

- 发票与报销核对：整理本地材料，输出缺失项、金额异常和待确认清单。
- 费用摘要：按部门、项目或时间汇总材料中的费用信息。
- 付款节点整理：从合同、邮件或文档中提取付款时间、条件和负责人。

行政：

- 会议纪要整理：从会议记录或文档中提取结论、待办、负责人和截止时间。
- 制度文档整理：提炼制度要点、适用范围和需要更新的条款。
- 采购/资产台账摘要：整理采购记录、资产状态和待补材料。

产品经理：

- 需求拆解：把需求材料拆成背景、目标、范围、验收标准和风险。
- PRD 评审：检查 PRD 缺口、歧义、依赖和未定义边界。
- 竞品资料整理：提取竞品功能、差异点、证据来源和待验证问题。

律师：

- 合同摘要：提取主体、标的、金额、期限、义务和关键风险。
- 条款风险清单：列出高风险条款、待确认事实和需补充材料。
- 材料整理：按案件/项目阶段整理文件、事实时间线和证据清单。

法律模板只能声明“辅助整理材料”，不得显示为法律意见或替代律师判断。

### 2.5 禁用与延期

以下功能不得在本轮首屏展示为可用能力：

- Excel 深度分析、PDF OCR、合同深度解析、自动法律意见。
- 插件市场、手机连接、团队协作、云端自动化。
- 完整多会话系统、Kun runtime 迁移、React 19 / Tailwind / Zustand 迁移。

如需展示，只能标注为“后续支持”，并禁用主操作。

## 3. 产品反查

反查清单：

- 是否像 B 端工作客户端，而不是运维面板。
- 是否默认简体中文。
- 是否隐藏 runtime、bridge、route、cache、raw JSON 等工程噪音。
- 是否财务/行政用户能在 10 秒内知道如何开始。
- 是否保留审批、回滚、安全边界。
- 是否与 Kun/Quinn 的侧栏、主工作区、底部输入框和轻视觉语言一致。

结论：大概率(90%) 通过。需要在实现后用截图继续核对。

## 4. 技术研发部评审

### 4.1 可直接实现

- 中文桌面壳。
- 左侧项目/任务/技能/自动化/设置。
- 底部主输入框。
- 角色模板填充任务草稿。
- 任务详情抽屉。
- 审批/拒绝/回滚入口迁移。
- 默认隐藏 raw JSON 和诊断。

依据：现有 `api.ts` 已有 task、workspace task、approval、rollback、diagnostics 等函数。

### 4.2 只需前端重组

- `Task Center` 常驻面板改为任务列表和抽屉。
- `Code` 改为高级入口。
- 现有 Workspace / Automation 内容改为更业务化的中文呈现。
- 设置页保留版本和诊断，但不放在首屏主路径。

### 4.3 需要新增 bridge 能力

本轮不新增。若后续要做 Excel/PDF/OCR/合同结构化解析，应先新增 runtime/bridge 能力与测试，再开放 UI。

### 4.4 必须延期

- 插件市场。
- 手机连接。
- 云端团队协作。
- 云端常驻自动化。
- Kun runtime 或状态管理迁移。

### 4.5 组件拆分方案

- `App.tsx`：状态组合和 bridge 数据加载。
- `components/AppShell.tsx`：整体工作台壳。
- `components/Sidebar.tsx`：项目、任务、技能、自动化、设置。
- `components/Workbench.tsx`：中间任务工作区。
- `components/TaskComposer.tsx`：底部主输入框。
- `components/TaskTemplateGrid.tsx`：角色模板。
- `components/TaskDrawer.tsx`：任务详情抽屉。
- `components/ApprovalPanel.tsx`：diff、批准一次、拒绝、回滚。
- `components/SettingsPanel.tsx`：本地配置和诊断。

### 4.6 状态流

1. 启动加载 health 和任务列表。
2. 用户选择项目后刷新 workspace assets。
3. 模板点击填充 `TaskDraft`，不自动提交。
4. 用户提交后调用现有 task API。
5. 创建成功后选择任务并打开详情抽屉。
6. 审批、拒绝、回滚后刷新当前 snapshot 和任务列表。
7. 诊断只在用户主动展开高级详情或设置页生成。

### 4.7 风险与缓解

- 风险：前端单体文件拆分引发回归。缓解：先保持 API 函数不变，组件只接收现有状态和回调。
- 风险：业务模板暗示未实现的文件解析能力。缓解：模板只填充目标和约束，不承诺具体格式能力。
- 风险：隐藏诊断导致调试困难。缓解：保留高级详情和设置页诊断入口。
- 风险：视觉只像“换皮”。缓解：信息架构同步变更，删除常驻工程面板。
