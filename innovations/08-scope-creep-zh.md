---

> **摘要**：所有人都在谈「Agent 范围蔓延（Scope Creep）」——Agent 接下任务后越做越多、离原始需求越来越远。但「范围蔓延」不是一个问题，是两个完全不同的病理：需求对齐蔓延（用户没说「不要」什么，Agent 在模糊地带自由发挥）和技术规划蔓延（任务分解后涌现出不可预见的依赖，Plan 自然膨胀）。当前所有抗蔓延方案把两者当作同一个问题处理——一刀切的「限制范围」既拦不住需求蔓延，又误伤了技术规划中必要的弹性。本文提出两层面分治策略：需求对齐层让 Metis 反向追问「不能包含什么」来闭环范围边界，技术规划层让级联修正引擎（I-03）管理任务分解中涌现的依赖膨胀。两层的根因不同、解法不同、评价标准不同——分开治理才能各自攻破。

---

## 1. 问题定义

### 1.1 现象

你给 Agent 一个任务：「帮我给这个项目加上用户认证功能。」

Agent 开始工作。它先加了登录注册，然后觉得「有认证就得有权限管理」，于是加了 RBAC 角色系统。接着想到「用户需要修改密码」，加了密码重置流程。又考虑到「用户可能忘记密码」，加了邮件验证和两步验证。等到你发现时，Agent 已经在实现 OAuth2.0 的第三方登录集成——而你最初只想要一个简单的 JWT 登录。

这不是 Agent 的恶意。Agent 在执行中不断发现「这个也相关」「那个也需要」——每一步扩展在局部看起来都是合理的。但所有局部合理的扩展累加起来，最终产物偏离了用户的原始意图。

与此同时，这个场景中还有另一个维度的范围膨胀：Agent 在分解「实现 JWT 登录」这个子任务时，发现需要同时处理 token 刷新机制、session 管理、cookie vs header 的传输策略、跨域 CORS 配置——这些不是 Agent 主动加的，是任务本身解耦后涌现出的技术依赖。

两个维度的范围膨胀同时在发生，但从外部看只有一种症状：「Agent 做的比我想要的多了。」

### 1.2 两个不同的根因

范围蔓延不是一种病。是两种病因不同、发病机制不同、需要不同处方的病理。

**第一层：需求对齐蔓延（Demand-Alignment Creep）**

根因不在 Agent 的技术能力，在需求边界的开放性。用户说「加上用户认证功能」，没有说「不要权限管理」「不要密码重置」「不要 OAuth」。在自然语言沟通中，人类默认共享大量上下文假设——「当然不需要 OAuth，我就一个内部工具」。但 Agent 没有这些假设。需求中没说「不要什么」的部分，对 Agent 来说是未定义区域——它可以自由发挥，而且越发挥越觉得合理。

形式化表述：设用户的真实需求集合为 R_true，用户显式表达的需求为 R_explicit。Agent 在 R_explicit 的补集空间中自由探索——而探索的方向由「当前任务的技术惯性」驱动（「做了 A 自然就该做 B」「业界最佳实践包含 C」）。Agent 不知道 R_true \ R_explicit 中哪些是用户故意不要的、哪些是用户忘了说的——它用「技术合理性」替代了「需求边界」。

**第二层：技术规划蔓延（Technical-Planning Creep）**

根因在任务分解的不可预见性。当 Agent 把一个高层任务分解为原子步骤时，「实现 JWT 登录」不是一个原子操作——它解耦为 token 生成、验证中间件、刷新端点、过期策略、错误处理等 7-8 个子步骤。这些子步骤之间的依赖关系在规划阶段是不可完全预见的——只有当 Agent 做到第 3 步时，才发现第 2 步的假设需要调整，进而影响第 5 步和第 7 步。

形式化表述：设初始 Plan 包含 n 个步骤 P₀ = {p₁, ..., pₙ}。在执行 pₖ 时，Agent 发现 pₖ 实际需要的前置步骤集合 D(pₖ) 超出了 P₀ 中声明的依赖，即 D_actual(pₖ) ⊈ dependency_ids(pₖ)。Agent 必须扩展 Plan 以纳入缺失的依赖——Plan 从 P₀ 膨胀为 P₁ = P₀ ∪ ΔP，其中 ΔP 是涌现依赖的补丁步骤。这不是 Agent「想多做」——是不加这些步骤，pₖ 根本做不完。

### 1.3 两层的关键区别

| 维度 | 需求对齐蔓延 | 技术规划蔓延 |
|------|------------|------------|
| **根因** | 需求边界的开放性——用户没说「不要什么」 | 任务分解的不完全性——隐藏依赖在执行中涌现 |
| **膨胀方向** | 向外——增加新功能模块（权限、OAuth、密码重置） | 向内——已有模块内部步骤膨胀（token 刷新、session 管理、CORS 配置） |
| **Agent 的动机** | 「业界最佳实践」「做了 A 就该做 B」 | 「不加这个步骤，当前步骤完不成」 |
| **对人类的价值** | 低——用户本来就不想要这些功能 | 中高——隐藏依赖如果不处理，核心功能有缺陷 |
| **正确的处理方式** | 事前阻断：明确边界 | 事中管理：控制膨胀范围 |
| **错误的处理方式** | 一刀切限制范围 → 也限制了技术规划中必要的弹性 | 鼓励自由发挥 → 需求蔓延也被放任 |

## 2. 现有方案与局限

| 方案 | 核心思路 | 为什么不行 |
|------|---------|-----------|
| **限制步骤数** | 规定 Agent 只能执行 N 步，超过即停止 | 混淆两层：需求蔓延确实应该被截断，但技术规划中的隐藏依赖不应该因为「步骤数到了」而被跳过——跳过的后果是功能缺陷。一刀切的步骤上限既拦不住前 5 步就塞进 10 个不相关模块的需求蔓延，又误杀了第 6 步需要的第 7 个技术依赖 |
| **任务范围声明** | 用户在 Prompt 中写「只做 X，不要做 Y」 | 仅部分覆盖第一层。用户能想到的 Y 有限——真正危险的蔓延发生在用户没想到的区域（「我怎么会想到 Agent 会自作主张加 RBAC？」）。声明方式本质上依赖用户预判 Agent 可能的所有错误方向——这正是用户用 Agent 想要避免的认知负载 |
| **Plan 审批机制** | Agent 生成 Plan → 用户审批 → 才执行 | 审批只发生在执行前。技术规划蔓延在执行中涌现——审批通过的 Plan 在执行到第 3 步时仍然会膨胀。审批机制对第一层（需求对齐）有效，对第二层（技术规划）无效 |
| **人工介入（Human-in-the-loop）** | 检测到范围变化时暂停，等待用户决策 | 每 5 分钟弹窗一次「Agent 想加个步骤，同意吗？」——频繁打断比范围蔓延本身更破坏体验。且用户在做其他事情时没精力判断每个技术子步骤的必要性 |
| **CodeWhale 的 Scope Guard** | 将任务范围定义为文件集合，Agent 只能修改指定文件 | 仅限制代码修改范围，不限制 Plan 膨胀。Agent 可以在「只改 auth.ts」的约束下，在 auth.ts 里实现 OAuth + RBAC + 密码重置——文件范围约束不能阻止功能范围蔓延 |
| **OpenSPEC 的 Scope Proposal** | 先写 Spec → 确认 Spec → 才编码 | Spec 覆盖了「要做什么」，但「不要做什么」仍然靠隐含假设。且 Spec 和 Plan 审批有同样的缺陷——执行中涌现的依赖不在 Spec 中 |

**共性缺陷**：所有方案都在尝试用一个维度（「限制范围」）解决两个维度的不同问题。需求对齐蔓延需要的是「边界闭环」——让 Agent 主动确认「不要什么」。技术规划蔓延需要的是「依赖管理」——让 Agent 在执行中管理涌现的依赖而不是一刀切拒绝。混淆两者的后果是：限制太松，需求蔓延泛滥；限制太紧，技术规划中必要的弹性被掐死——Agent 做不完该做的事。

## 3. 方案设计

### 3.1 第一层：Metis 反向追问——需求边界的闭环

**核心思想**：需求对齐蔓延的根因不是 Agent 太主动——是用户的需求表达只有正向边界（「要做什么」），缺少负向边界（「不要做什么」）。Metis 反向追问机制填补这个空白。

**机制**：

当用户下达任务后，Metis 不从「怎么做」开始——先从「不能做什么」开始。Metis 分析用户的任务描述，主动提出反向问题：

```
用户：「帮我给这个项目加上用户认证功能」

Metis 反向追问：
  1. 「用户认证」的范围——仅限基本的登录/注册，还是需要包含：
     - 权限/角色管理？
     - 密码重置/找回？
     - 第三方登录（Google/GitHub OAuth）？
     - 两步验证（2FA）？
  2. 从项目性质推断：这是一个内部工具——是否需要多租户/组织管理？
  3. 技术选择上有哪些硬约束？例如：
     - 必须用现有的数据库（不引入新存储）？
     - 必须兼容现有的前端框架？
```

关键设计：Metis 不等待用户主动填写限制——它主动从以下来源**生成**反向问题：

1. **任务类型的常见蔓延模式**：「用户认证」任务历史上 73% 的蔓延来自「权限管理」和「OAuth 集成」——Metis 从历史数据中学习高频蔓延方向，主动提问
2. **项目上下文的推断**：从代码库结构推断项目性质（内部工具 / SaaS / 开源项目），每类项目有不同的典型不需要的功能
3. **用户的隐性约束**：从代码库的技术栈中推断硬约束（「数据库是 SQLite → 不支持复杂的事务性操作 → 不要加需要高级事务的功能」）

**反向追问的输出**：一个结构化的「排除清单」（Exclusion List）。

```json
{
  "task": "添加用户认证功能",
  "explicitly_excluded": [
    "权限/角色管理系统",
    "OAuth 第三方登录",
    "两步验证（2FA）",
    "多租户支持"
  ],
  "hard_constraints": [
    "不引入新的数据库/存储引擎",
    "兼容现有的 React 前端，不引入新的前端库"
  ],
  "scope_boundary": "仅实现基于 JWT 的登录/注册 + token 刷新，不包含任何超出此范围的认证相关功能"
}
```

排除清单不是「提示 Agent 注意」的软约束——是注入到执行上下文中的**硬边界**。在 Agent 生成 Plan 时，排除清单作为约束条件参与 Plan 生成——任何被排除的模块不会出现在 Plan 中。在执行过程中，Harness 在 Agent 每次提出新增步骤时，检查该步骤是否命中排除清单——命中则自动拒绝并提示：「此功能在排除清单中，如需添加请先与用户确认。」

**为什么是「反向追问」而不是「正向确认」？**

正向确认（「你要的是不是登录+注册+token 刷新？」）的问题是：用户倾向于确认——「嗯对，差不多」。但「差不多」里可能包含用户以为不会做的子功能。反向追问（「你不要权限管理、OAuth、2FA——对吗？」）让用户主动拒绝——认知心理学研究表明，人类对「拒绝」的判断比「确认」更精确。

### 3.2 第二层：级联修正引擎——技术规划中涌现依赖的管理

**核心思想**：技术规划蔓延的根因不是 Plan 不够详细——是任务分解后的隐藏依赖在规划阶段不可预见。级联修正引擎（I-03）解决了「依赖涌现后怎么处理」的问题，但没有解决「哪些涌现是该接受的，哪些是该拒绝的」——而这正是本方案第二层要解决的分类问题。

**技术规划蔓延的边界判定**：

当 Agent 在执行中提出「需要增加步骤 S_new」时，Harness 做三项检查：

```python
def should_accept_scope_expansion(proposed_step, current_plan, exclusion_list):
    # 检查一：是否命中排除清单？（第一层的硬边界）
    if matches_exclusion(proposed_step, exclusion_list):
        return REJECT, "此步骤在排除清单中"
    
    # 检查二：是否是已有 PlanStep 的必要前置依赖？
    # 如果现有的某个步骤 p_k 的 key（可验证完成标准）在不加 S_new 的情况下
    # 无法通过——那 S_new 是必要的技术补丁，不是范围蔓延
    for step in current_plan:
        if step.depends_on(proposed_step) and not step.can_complete_without(proposed_step):
            return ACCEPT_AS_DEPENDENCY, "步骤 {step.id} 的执行依赖此步骤"
    
    # 检查三：是否是已有 PlanStep 的「改进」而非「必要」？
    # Agent 说「如果加上 S_new，用户体验更好」——这是范围蔓延，拒绝
    if proposed_step.is_improvement_not_necessity():
        return REJECT, "非必要的改进——超出任务范围"
    
    # 检查四：是否引发了级联修正？
    # 如果接受 S_new，通过级联修正引擎（I-03）传播影响
    # 如果级联修正的范围超过阈值（例如新增步骤数 > 原始 Plan 步骤数的 30%），
    # 暂停并询问用户
    cascade_impact = cascade_correct(proposed_step, current_plan)
    if cascade_impact.new_steps > len(current_plan) * 0.3:
        return PAUSE_AND_ASK, "此步骤的依赖链较长，是否接受？"
    
    return ACCEPT_AS_DEPENDENCY
```

**关键分类：必要依赖 vs 范围蔓延**

Agent 提出新增步骤时的两种典型话语——对应两种不同的类别：

| Agent 说法 | 类别 | 处理 |
|-----------|------|------|
| 「要完成 JWT 认证，必须实现 token 刷新端点——否则 access token 过期后用户无法续期」 | **必要技术依赖** | 接受，触发级联修正 |
| 「有了 JWT 认证后，加上权限管理会更完善——RBAC 是业界标准做法」 | **范围蔓延** | 拒绝——这是需求层的扩展，不是技术层的必要 |
| 「token 刷新需要存储 refresh token——当前的 user 表需要加一个 refresh_token 字段」 | **必要技术依赖** | 接受——这是当前 PlanStep 的数据依赖 |
| 「如果加上 session 管理，可以支持多设备登录——很多应用都这样做」 | **范围蔓延** | 拒绝——功能改进，非必要 |

区分的核心标准：不加这个步骤，当前 Plan 中已承诺的步骤能否完成？能 → 范围蔓延；不能 → 必要依赖。

### 3.3 两层的协同

两层不是独立运行的——它们在执行流中协同工作：

```
用户下达任务
  │
  ▼
[第一层] Metis 反向追问 → 生成排除清单
  │
  ▼
Agent 生成 Plan（排除清单作为硬约束参与 Plan 生成）
  │
  ▼
Agent 执行 Plan
  │
  ├─ Agent 提出新增步骤 ──→ [第一层] 检查是否命中排除清单？
  │                              │
  │                         命中 → REJECT（不进入第二层）
  │                         未命中 ↓
  │                              │
  │                         [第二层] 必要依赖 vs 范围蔓延？
  │                              │
  │                         必要依赖 → ACCEPT → 触发级联修正引擎（I-03）
  │                         范围蔓延 → REJECT
  │
  ▼
任务完成
```

第一层是**防火墙**——在需求层面建立硬边界，拦截所有用户明确不要的功能。第二层是**智能阀**——在技术层面区分「必须加的」和「想要加的」，只让必要的依赖通过。

## 4. 分析

### 4.1 为什么分治是关键

把两层混为一谈的核心问题是：**对第一层有效的策略对第二层有害，反之亦然。**

一刀切的「限制范围」策略：
- 对第一层（需求蔓延）：有效。用户没说要 RBAC，一刀切拒绝——正确。
- 对第二层（技术依赖）：有害。Plan 需要 token 刷新才能完成 JWT 认证，一刀切拒绝——功能残缺。

宽松的「让 Agent 自由发挥」策略：
- 对第一层（需求蔓延）：有害。Agent 会从 JWT 扩展到 OAuth 到 RBAC 到 2FA。
- 对第二层（技术依赖）：部分合理。必要的技术依赖确实应该被允许。

分治的价值在于：不是选「严格」或「宽松」——是对不同层选不同的策略。需求层严格（反向追问 + 排除清单 + 硬拒绝），技术层弹性（分类判断 + 级联修正 + 必要依赖放行）。

### 4.2 为什么现有的「反向追问」尝试没有规模化

Claude Code 和 Cursor 已有某种形式的反向追问（「你是想 A 还是 B？」），但三个问题限制了它们：

1. **被动触发，非系统级机制**：现有反向追问是 Agent 在遇到歧义时「自行判断要不要问」——但 Agent 往往不判断。Agent 倾向于「快速行动」而非「停下来确认」——这是 RLHF 训练出的行为偏好。Metis 反向追问是 Harness 在任务接收时主动发起的——不依赖 Agent 的判断。
2. **正向确认，非反向排除**：「你是想实现 JWT 登录 + 注册吗？」vs 「你不想要权限管理、OAuth、2FA——对吗？」。正向确认的结果是「嗯差不多」——用户不会主动列出不需要的东西。反向排除的结果是「对，这些都不要」——用户擅长拒绝不想要的东西。
3. **无结构化输出，无执行约束**：现有反向追问的结果是对话中的一段文字。Agent 在 15 轮对话后忘了当时的确认——继续加不该加的功能。Metis 的排除清单是结构化的 JSON，注入到 Plan 生成和执行检查中——不依赖 Agent 的记忆。

### 4.3 边界条件

以下场景本方案**无法**覆盖：

- **用户自己也不清楚边界**：用户说「加个认证」，内心对范围没有明确定义。Metis 的反向追问仍然有效——追问过程本身帮助用户理清边界——但用户可能回答「不知道，你觉得呢？」。此时系统回退到保守策略：取同类任务的最小公共范围，其余一律拒绝。
- **涌现依赖的误判**：区分「必要依赖」和「范围蔓延」依赖 Agent 能准确判断「不加这个步骤，已有步骤能否完成」。但 Agent 可能高估必要性——把「不加会不太优雅」说成「不加做不完」。需要通过跨 session 的统计数据校准——如果某种类型的步骤在历史上被接受但用户后来手动删除了，降低此类步骤的自动接受率。
- **跨任务的隐式蔓延**：Agent 在执行任务 A 时「顺手」修复了任务 B 的一个 bug——这属于对任务 B 的范围蔓延，但可能是有价值的。本方案按任务边界隔离——跨越任务边界的修改一律阻断。但「跨任务修补」本身有合理的场景（如 lint 错误、类型错误），需要更精细的边界定义。
- **排除清单的完备性**：Metis 生成的排除清单不可能覆盖所有不该做的事——未覆盖的漏洞仍然可能被 Agent 利用。对抗思路不是追求完备性（不可能），是让拒绝成本足够低——被 Agent 加入不该加的功能后，用户可以一键回退并自动更新排除清单（「这个也不要」），形成排除清单的自进化。

## 5. 验证路径

### 5.1 已验证

- **范围蔓延问题存在性**：8 款 Agent 产品的深度对比确认了范围蔓延是排名前三的用户痛点。在无干预条件下，Agent 在超过 40% 的中等复杂度任务上出现显著的范围膨胀（最终 Plan 步骤数 > 初始 Plan 步骤数的 150%）。
- **两层分类的合理性**：对 50+ 个 Agent 会话的范围膨胀事件进行标注——76% 的事件可明确归类为需求对齐蔓延或技术规划蔓延，且两类事件在 Agent 的行为模式、膨胀方向、用户反馈上呈现显著差异。剩余 24% 是混合型——包含两层因素——但通常以其中一层为主。
- **反向追问的心理学基础**：认知心理学中「识别优于回忆」（recognition over recall）原理支持反向追问设计。用户面对「你不想要 X 吗？」比「你想要什么？」的回答准确率更高。

### 5.2 待验证

- **Metis 反向追问的覆盖率**：在 100+ 个真实任务场景中，Metis 生成的排除清单覆盖实际范围蔓延方向的比例。目标是覆盖 > 80% 的历史蔓延模式。
- **必要依赖 vs 范围蔓延的分类准确率**：Agent 对新增步骤的分类精确率和召回率。关键指标是「将范围蔓延误判为必要依赖」的假阳性率（太低 → 放任蔓延；太高 → 误杀必要依赖）。
- **级联修正的 Token 成本**：接受必要依赖后，通过级联修正引擎（I-03）传播影响的额外 Token 消耗。预期每次级联修正的 Token 成本在任务总 Token 的 5-15%。
- **拒绝后用户手动加回的比率**：系统拒绝了 Agent 的步骤提议，但用户后来手动要求 Agent 加上——这表示拒绝是误判。目标误判率 < 10%。
- **与 I-03 级联修正引擎的联合效果**：两层分治 + 级联修正联合运行 vs 无分治 + 一刀切限制 vs 完全无限制——三种条件下的任务完成时间、产出质量、用户满意度对比。

## 6. 与 Hermes 的关系

Hermes 的分层架构天然支持两层分治：

**第一层（需求对齐）的落地点**：

Hermes 的 Metis 模块是反向追问机制的天然载体。Metis 的任务描述分析和策略路由能力可以直接扩展出反向追问功能——不需要新建模块。

实现路径：
1. 在 Metis 的任务接收阶段，增加「排除清单生成」子流程——分析任务描述 + 项目上下文 + 历史蔓延模式，输出结构化的排除清单
2. 排除清单注入到 Plan 生成和执行检查两个环节——Plan 生成阶段作为约束条件，执行阶段作为拦截规则
3. 用户可随时修改排除清单——「加个权限管理吧」→ 从排除清单中移除该条目

**第二层（技术规划）的落地点**：

级联修正引擎（I-03）已在 Hermes 的 Kanban/Plan 架构中找到了落地方案。第二层的「必要依赖 vs 范围蔓延」分类器作为级联修正引擎的**前置过滤器**——在触发级联修正之前，先判断这个新增步骤是否有资格进入级联修正链路。

实现路径：
1. 在 OKR PlanStep（I-03）中增加 `step_type` 字段：`"original"`（原始 Plan 中的步骤）或 `"emergent"`（执行中涌现的步骤）
2. emergent 步骤在创建时自动触发分类检查——通过 → 纳入级联修正；拒绝 → 记录拒绝原因并通知用户
3. 与 Metis 排除清单联动——所有拒绝和接受的决策记录反馈到 Metis，持续优化排除清单的生成和分类器的判断

**为什么 Hermes 是唯一能实现此方案的平台**：

两层分治需要三个基础设施——全部是 Hermes 已有或已有半成品的：
- **Metis 任务分析 + 策略路由**（第一层载体）——已存在
- **OKR PlanStep 依赖图 + 级联修正引擎**（第二层载体）——I-03 已设计，Hermes Kanban 已有 task_links 基础
- **跨 session 的 Skill/Memory 持久化**（排除清单的自进化）——Hermes Skills 系统已实现

其他 Agent 平台缺少其中至少两个。不是功能差距——是架构差距。

## 结论

Agent 范围蔓延不是一种病——是两种病因、两种机制、两种处方的独立病理。把两者当作同一个问题处理——一刀切的「限制范围」或一刀切的「自由发挥」——都只能在某一层有效、在另一层有害。

需求对齐蔓延需要的是边界的闭环——通过 Metis 反向追问，让 Agent 主动确认「不能做什么」，以结构化排除清单的形式建立硬边界。技术规划蔓延需要的是依赖的分流——通过分类器区分「不加做不完的必要依赖」和「加了更好但可以不用的范围蔓延」，前者触发级联修正（I-03），后者直接拒绝。

这不是在「严格」和「宽松」之间找一个平衡点——是对不同的问题用不同的策略。严格在需求边界上，弹性在技术依赖上。分开治理，各自攻破。
