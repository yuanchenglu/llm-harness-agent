# DeepSeek-Coder：当大语言模型遇见编程——代码智能的崛起

**作者：** Daya Guo*, Qihao Zhu*, Dejian Yang, Zhenda Xie, Kai Dong, Wentao Zhang 等
**所属机构：** DeepSeek-AI，北京大学
**论文编号：** arXiv:2401.14196v2 [cs.SE] | 2024年1月26日

---

## 摘要

大语言模型的快速发展彻底改变了软件开发中的代码智能。然而，闭源模型的主导地位限制了广泛的研究和开发。为解决这一问题，我们推出了 **DeepSeek-Coder** 系列——一系列开源代码模型，规模从 1.3B 到 33B，从头在 2 万亿 token 上训练。这些模型在高质量的项目级代码语料上进行预训练，采用带有 16K 窗口的"填空"任务来增强代码生成和补全能力。我们的广泛评估表明，DeepSeek-Coder 不仅在多个基准上实现了开源代码模型中最先进的性能，而且超越了现有闭源模型如 Codex 和 GPT-3.5。此外，DeepSeek-Coder 模型采用宽松许可，允许研究和无限制的商业使用。

---

## 1. 引言

软件开发的领域已因大语言模型的快速发展而显著转型（OpenAI, 2023; Touvron et al., 2023），带来了代码智能的新时代。这些模型有潜力自动化和简化编码的许多方面，从错误检测到代码生成，从而提高生产力并减少人为错误的可能性。然而，该领域的一个主要挑战是开源模型与闭源模型之间的性能差距。

针对这一挑战，我们推出了 DeepSeek-Coder 系列。该系列包含一系列开源代码模型，规模从 1.3B 到 33B 不等，包括每个规模的基座版本和指令版本。系列中的每个模型都在来自 87 种编程语言的 2 万亿 token 上从头训练，确保了对编码语言和语法的全面理解。此外，我们尝试在仓库级别组织预训练数据，以增强预训练模型对仓库内跨文件上下文的理解能力。除了在预训练中使用下一个 token 预测损失外，我们还采用了 Fill-In-Middle（FIM）方法（Bavarian et al., 2022; Li et al., 2023），旨在进一步增强模型的代码补全能力。为满足处理更长代码输入的需求，我们将上下文长度扩展到 16K。

我们的主要贡献可总结如下：
- 推出了 DeepSeek-Coder-Base 和 DeepSeek-Coder-Instruct 系列代码模型，涵盖 87 种编程语言，提供多种规模。
- 首次在代码模型的预训练阶段尝试仓库级数据构建，发现其显著提升了跨文件代码生成能力。
- 严格分析了 FIM 训练策略对代码模型预训练阶段的影响，为 FIM 配置提供了有价值的见解。
- 在广泛的代码相关基准上进行了全面评估，DeepSeek-Coder-Base 超越所有现有开源代码模型，DeepSeek-Coder-Instruct 在代码任务上超越 GPT-3.5 Turbo。

## 2. 数据收集

DeepSeek-Coder 的训练数据集由 87% 源代码、10% 英文代码相关自然语言语料和 3% 代码无关中文自然语言语料组成。

### 2.1 GitHub 数据抓取与过滤

我们收集 GitHub 上 2023 年 2 月前创建的公开仓库，保留 87 种编程语言。应用类似 StarCoder 项目的过滤规则，将数据总量减少至原始大小的 32.8%。规则包括：过滤平均行超 100 字符或最大行超 1000 字符的文件；过滤字母字符少于 25% 的文件；对 HTML/JSON/YAML 等有特殊处理。

### 2.2 依赖解析

以往工作主要在文件级源代码上预训练代码模型，忽略了项目中不同文件间的依赖关系。因此，我们考虑利用同一仓库内文件间的依赖关系。具体来说，我们首先解析文件间的依赖关系，然后按拓扑排序排列这些文件，确保每个文件依赖的上下文在输入序列中排在该文件之前。算法采用修改的拓扑排序——选择入度最小的节点而非入度为零的节点——以处理图中的循环。文件路径信息也通过注释保存在训练数据中。

### 2.3 仓库级去重

我们在代码的仓库级别进行去重，而非文件级别（后者可能过滤掉仓库内的某些文件，破坏仓库结构）。具体来说，将仓库级拼接的代码作为单个样本，应用近去重算法以确保仓库结构的完整性。

### 2.4 质量筛选与去污染

除过滤规则外，我们还使用编译器和质量模型，结合启发式规则，进一步过滤低质量数据——包括语法错误、可读性差、模块化程度低的代码。总数据量 798 GB，6.03 亿文件，覆盖 87 种语言。为消除测试集污染，实现了 n-gram 过滤流程：若代码包含与测试数据相同的 10-gram 字符串，则被排除。

## 3. 训练策略

### 3.1 训练目标

**下一个 token 预测。** 将各种文件拼接成固定长度条目，训练模型根据已提供上下文预测后续 token。

**Fill-in-the-Middle (FIM)。** 代码预训练场景中，通常需要根据给定的上下文和后文生成相应的插入内容。由于编程语言中的特定依赖关系，仅依赖下一个 token 预测不足以学习这种能力。FIM 方法将文本随机分为三部分，打乱顺序后用特殊字符连接。我们采用 PSM（Prefix-Suffix-Middle）模式，引入三个哨兵 token。

在每个代码文件中，将内容分为三段 f_pre、f_middle、f_suf，PSM 模式构建训练样本为：

`<|fim_start|> f_pre <|fim_hole|> f_suf <|fim_end|> f_middle <|eos_token|>`

通过消融实验（图 3），我们发现 100% FIM 率在 HumanEval-FIM 上表现最佳但代码补全能力最弱，存在显著权衡。最终选择 **50% PSM 率**作为训练策略，在 FIM 效率与代码补全能力间取得平衡。

### 3.2 分词器

使用 HuggingFace Tokenizer 库训练 BPE 分词器，词汇表大小 32,000。

### 3.3 模型架构

开发 1.3B、6.7B 和 33B 三种规模的模型，均基于 DeepSeek LLM 框架。采用纯解码器 Transformer，集成旋转位置编码（RoPE）。DeepSeek 33B 采用分组查询注意力（GQA，组大小 8）以提升训练和推理效率。使用 FlashAttention v2 加速注意力计算。

| 超参数 | 1.3B | 6.7B | 33B |
|--------|------|------|-----|
| 激活函数 | SwiGLU | SwiGLU | SwiGLU |
| 隐藏维度 | 2048 | 4096 | 7168 |
| 中间维度 | 5504 | 11008 | 19200 |
| 层数 | 24 | 32 | 62 |
| 注意力头数 | 16 | 32 | 56 (GQA-8) |
| 批量大小 | 1024 | 2304 | 3840 |
| 最大学习率 | 5.3e-4 | 4.2e-4 | 3.5e-4 |

**表 2 | DeepSeek-Coder 超参数。**

### 3.4 优化

使用 AdamW 优化器（β₁=0.9，β₂=0.95）。根据 DeepSeek LLM 提出的缩放定律调整批量大小和学习率。采用三阶段学习率调度策略，包括 2000 预热步，最终学习率为初始率的 10%。

### 3.5 环境

使用 HAI-LLM 框架，集成张量并行、ZeRO 数据并行和 PipeDream 流水线并行。实验在配备 NVIDIA A100 和 H800 GPU 的集群上进行，节点间使用 InfiniBand 互连。

### 3.6 长上下文

重新配置 RoPE 参数以扩展默认上下文窗口——将缩放因子从 1 增加到 4，基础频率从 10000 改为 100000。进行了额外 1000 步训练（批量大小 512，序列长度 16K）。理论上模型可处理最多 64K token 上下文，经验上 16K token 范围内输出最可靠。

### 3.7 指令微调

通过基于 Alpaca 指令格式的高质量数据指令微调，开发 DeepSeek-Coder-Instruct。使用特殊分隔符 `<|EOT|>` 标记每个对话轮次结尾。训练使用余弦调度（100 预热步，初始学习率 1e-5），批量大小 4M token，总计 2B token。

## 4. 实验结果

### 4.1 代码生成

**HumanEval 和 MBPP 基准（多语言）。** DeepSeek-Coder-Base 33B 在 HumanEval 上平均准确率 56.1%，MBPP 上 66.0%——超越 CodeLlama-Base 34B 分别为 9% 和 11%。值得注意的是，DeepSeek-Coder-Base 6.7B 甚至超越了 CodeLlama-Base 34B。指令微调后，DeepSeek-Coder-Instruct 33B 在 HumanEval 上超越 GPT-3.5-Turbo（79.3% vs 76.2%），显著缩小了与 GPT-4 的差距。

| 模型 | Python | C++ | Java | PHP | TS | JS | Avg | MBPP |
|------|--------|-----|------|-----|-----|-----|-----|------|
| CodeLlama 34B | 48.2% | 44.7% | 44.9% | 41.0% | 42.1% | 42.2% | 41.0% | 55.2% |
| **DeepSeek-Coder-Base 33B** | **56.1%** | **58.4%** | **51.9%** | **44.1%** | **52.8%** | **55.3%** | **50.3%** | **66.0%** |
| GPT-3.5-Turbo | 76.2% | 63.4% | 69.2% | 60.9% | 69.1% | 67.1% | 64.9% | 70.8% |
| **DeepSeek-Coder-Instruct 33B** | **79.3%** | **68.9%** | **73.4%** | **72.7%** | **67.9%** | **73.9%** | **69.2%** | **70.0%** |

**表 3 | 多语言 HumanEval 和 MBPP 基准表现。**

**DS-1000 基准（数据科学工作流）。** DeepSeek-Coder-Base 33B 在 Matplotlib、NumPy、Pandas、PyTorch、SciPy、Scikit-Learn、TensorFlow 七个库上实现 40.2% 平均准确率，全面超越所有开源模型。

**LeetCode 竞赛基准。** 收集 2023.7-2024.1 期间 180 道最新 LeetCode 竞赛题。DeepSeek-Coder-Instruct 33B 以 27.8% Pass@1 超越 GPT-3.5-Turbo（23.3%）。CoT 提示策略显著增强性能，特别是对难题。

### 4.2 FIM 代码补全

使用单行填充基准（Python、Java、JavaScript）评估。DeepSeek-Coder-Base 1.3B 以 70.4% 平均准确率超越更大的 StarCoder 16B（69.7%）和 CodeLlama 7B（69.7%）。推荐 DeepSeek-Coder-Base 6.7B 用于实际代码补全工具——在效率和准确性间取得最佳平衡。

### 4.3 跨文件代码补全

使用 CrossCodeEval 基准（Python、Java、TypeScript、C#），要求模型理解和访问跨多个文件的仓库。DeepSeek-Coder-Base 6.7B 在所有语言上持续领先，配合 BM25 检索后优势更加明显。使用文件级语料（w/o Repo Pre-training）预训练时，跨文件补全性能下降，证实了仓库级预训练的有效性。

### 4.4 基于编程的数学推理

使用 PAL 方法在 7 个基准（GSM8K、MATH、GSM-Hard、SVAMP、TabMWP、ASDiv、MAWPS）上评估。DeepSeek-Coder-Base 33B 以平均 65.8% 领先所有开源模型，超过 CodeLlama 34B 的 62.0%。

## 5. 从通用 LLM 继续预训练

为进一步增强 DeepSeek-Coder 的自然语言理解和数学推理能力，我们从通用语言模型 DeepSeek-LLM-7B Base 出发，在 2 万亿 token 上继续预训练，得到 **DeepSeek-Coder-v1.5 7B**。

v1.5 相比 v1 在编程能力基本保持不变的同时，数学推理和自然语言能力显著提升：GSM8K 从 43.2%→62.4%，MMLU 从 36.6%→49.1%，HellaSwag 从 53.8%→69.9%。

| 模型 | HumanEval | GSM8K | MMLU | HellaSwag |
|------|----------|-------|------|-----------|
| DeepSeek-Coder-Base 6.7B | 44.7% | 43.2% | 36.6% | 53.8% |
| **DeepSeek-Coder-Base-v1.5 6.9B** | **43.2%** | **62.4%** | **49.1%** | **69.9%** |

**表 10 | v1.5 与 v1 性能对比。** 编程性能略有下降但数学和 NLP 显著提升。

## 6. 结论

我们推出了 DeepSeek-Coder 系列代码大语言模型，三种规模（1.3B、6.7B、33B）。模型在精心策划的项目级代码语料上使用"填空"预训练目标独特训练。33B 基座模型在多项标准测试中超越现有开源代码模型，6.7B 模型尽管规模更小，性能媲美 34B 的 CodeLlama。

DeepSeek-Coder-Instruct 33B 在多种代码相关任务上超越 OpenAI 的 GPT-3.5 Turbo。基于 DeepSeek-LLM 7B 继续预训练的 v1.5 版本在保持编码高性能的同时展现了增强的自然语言理解能力，证实了我们"最有效的代码 LLM 应建立在强大的通用 LLM 之上"的信念。
