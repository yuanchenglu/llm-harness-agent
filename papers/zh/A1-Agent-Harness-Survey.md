# 综合摘要：《面向大语言模型智能体的智能体框架：一项综述》

**论文信息：**
*   **标题：** Agent Harness for Large Language Model Agents: A Survey（面向大语言模型智能体的智能体框架：一项综述）
*   **作者：** 钱雨萌（Qianyu Meng）、王亚楠（Yanan Wang）、陈立一（Liyi Chen）等
*   **状态：** 预印本（v1，提交于2026年4月3日）

## 核心论点：框架而非模型才是制约因素

> "框架，而非模型，才是实际智能体系统性能的制约因素。"

核心主张是，智能体任务执行的可靠性更多地取决于**基础设施层（"智能体执行框架"）**，而非底层模型的能力。本文对该层给出了形式化定义、分类学以及系统性分析，论证它是一个独立的架构对象，需要专门的研究。

---

## 1. 关键定义：形式化框架 H=(E,T,C,S,L,V)

智能体框架被形式化定义为包含六个组件的运行时治理系统：

> **H = (E, T, C, S, L, V)**

- **E — 执行循环（Execution Loop）：** 管理"观察-思考-行动"循环、回合排序、终止条件以及错误恢复。形式化为一个**带标签的转换系统（Labeled Transition System, LTS）**，具有安全性（无失控运行）和活性（可达终止状态）属性。
- **T — 工具注册表（Tool Registry）：** 一个带类型、经过验证的可用工具目录。负责路由、监控和管理工具调用。
- **C — 上下文管理器（Context Manager）：** 管理哪些信息进入模型的上下文窗口。负责压缩、检索和优先级排序策略。
- **S — 状态存储（State Store）：** 跨回合和会话持久化任务相关状态；支持从部分失败中恢复。
- **L — 生命周期钩子（Lifecycle Hooks）：** 调用前和调用后的拦截点，用于认证、日志记录、策略执行和仪表化。
- **V — 评估接口（Evaluation Interface）：** 标准化的仪表化设施，用于捕获动作轨迹、中间状态和成功信号，支持离线分析和跨框架比较。**与运维日志（L）不同。**

**关键边界界定：**
- **最小框架：** 实现 E 和 T。
- **不属于框架：** *ReAct*（无错误恢复，无正式注册表）；*LangGraph*（属于框架，缺少 C/S/L/V）。
- **属于模块：** *MemGPT*（C 和 S 很强，缺少 E/T）。

---

## 2. 证据矩阵：框架变更 > 模型变更

汇聚的定量证据表明，仅靠框架层面的变更就能带来显著改进：

| 来源类型 | 干预措施 | 性能变化 | 模型变更？ |
|---|---|---|---|
| **实践者（Pi Research）** | 更改编辑工具格式 | 6.7% → **68.3%**（10倍提升） | 否 |
| **实践者（LangChain DeepAgents）** | 中间件、生命周期钩子、系统提示 | 52.8% → **66.5%**（+26%） | 否 |
| **预印本（Meta-Harness，斯坦福/MIT）** | 在框架空间上进行自动化智能体搜索 | **76.4%**（在 TerminalBench-2 上超越手工编码方案） | 否 |
| **预印本（AgencyBench）** | 模型在原生框架 vs. 外部框架中运行 | 原生生态系统得分显著更高（例如 48.4%） | 否 |
| **实践者（Vercel）** | 工具数量从 15 个缩减到 2 个 | 80% → **100%** 准确率 | 否 |

**关键实践者报告：**
- **OpenAI Codex：** 提出了"框架工程（Harness Engineering）"概念。一个团队用 0 行手写代码构建了约 100 万行的生产级代码。
- **Stripe Minions：** 每周约 1,300 个 PR。精心策划的工具注册表至关重要；暴露所有工具会降低性能。
- **OpenClaw PRISM：** 首个面向开源框架的系统性、已发表的运行时安全层。

---

## 3. 历史演进："框架转向"

框架概念源自三条脉络：
1.  **软件测试框架（JUnit）：** 治理包装器模式。
2.  **强化学习环境（OpenAI Gym）：** 接口标准化（`step()`/`observe()`）。
3.  **早期 LLM 智能体（ReAct、AutoGPT、MemGPT）：** 定义需要何种治理的失败模式目录。

**"框架转向"（2024–2026）** 标志着从以模型为中心的研究向以基础设施为中心的工程实践的转变，其催化剂包括协议标准化（MCP、A2A）、形式化（AIOS）以及大规模评估基础设施（HAL、AgencyBench）。

**三种工程范式：**
- 提示工程（2022-2024）："该给什么文本？"
- 上下文工程（2025）："该组织什么信息？"
- **框架工程（2026）：** "该设计什么样的治理机制？"

## 4. 系统分类学（分析了 22 个系统）

**框架完备性矩阵** 揭示了不同的类别：

| 类别 | 示例系统 | 组件覆盖范围 |
|---|---|---|
| **全栈框架** | DeepAgents、OpenClaw、OpenHands、AIOS、Claude Code | 通常覆盖完整的 E、T、C、S、L、V |
| **框架** | LangGraph、AutoGen、LlamaIndex | 部分覆盖 E 和 T；C、S、L、V 通常缺失（被委托出去） |
| **能力模块** | MemGPT（C/S）、MCP 服务器（T）、Voyager（S） | 在 1-2 个组件上表现优秀；无编排能力 |
| **评估基础设施** | HAL、AgencyBench、SWE-bench、OSWorld | 广泛的 E 和 V；对该领域的可复现性至关重要 |

**矩阵中的关键发现：** **V（评估接口）** 和 **L（生命周期钩子）** 是全栈框架与框架之间最具区分度的两个组件——它们是将编排系统从单纯执行提升为可测量、安全且高度可靠的智能体基础设施的关键要素。

## 5. 框架设计的五个维度

对于每个维度，本文描述了设计空间、已知的最佳实践以及开放性挑战。

### 5.1 维度一：执行策略

| 策略 | 描述 | 典型工具/模式 |
|---|---|---|
| 单步 | 每一步精确一次 LLM 调用 | ReAct、OpenAI Responses API |
| 循环 | 重复直到满足终止条件 | `while` 循环、Codex CLI |
| 有向无环图（DAG） | 预定义的工具管道 | DSPy、LangChain LCEL |
| 智能体图 | 自主规划、动态图遍历 | `planner.execute()`、MetaGPT |

**关键见解：** 单步和循环策略以其简单性主导实际应用。有向无环图和智能体图策略如果在框架层面缺乏正确的治理约束，容易产生级联故障。

### 5.2 维度二：工具治理

**关键设计问题：** 向模型暴露多少工具？以何种粒度暴露？

| 策略 | 风险/收益 |
|---|---|
| **精简注册表**（Vercel：15→2） | 准确率 ↑，但灵活性受限 |
| **分层工具**（AIOS、OpenClaw） | 通过权限级别平衡安全性与能力 |
| **动态注册**（MCP、A2A） | 可扩展性 ↑，但攻击面扩大（需要 L 进行治理） |

**安全关键发现：** Stripe 的 PRISM 报告显示，在 Curated 模式（严格过滤，仅允许经过审查的工具）下，恶意工具调用成功率为 0%；在 Open 模式（允许注册表外的动态工具）下，恶意调用成功率为 67%。

### 5.3 维度三：上下文管理

| 策略 | 描述 | 权衡 |
|---|---|---|
| **FIFO 窗口** | 保留最近的 N 条消息 | 简单，但会丢弃任务关键信息 |
| **Token 预算** | 根据 Token 计数进行压缩/丢弃 | 精确，但无法感知语义 |
| **语义检索**（MemGPT） | 基于嵌入的 RAG | 保留事实，但可能遗漏过程 |
| **分层/结构化** | 系统提示 + 工作记忆 + 长期记忆 | 组织良好，但系统设计复杂 |

**决定性发现：** Token 压缩几乎不影响性能——即使在高度压缩下，SWE-bench 上的正确率也保持稳定（压缩 4 倍时约 43%，未压缩时为 45%）。

### 5.4 维度四：生命周期与安全

**L 组件的功能：**
- **认证/授权：** 确认工具调用者（或代表其行事者）拥有权限
- **速率限制：** 执行频率和配额限制
- **输出验证：** 检查输出结果是否损坏或恶意
- **审计日志：** 不可变记录，用于事后分析和取证
- **断路器：** 在级联失败发生时终止执行

**关键安全发现：** 无框架的原始 LLM 调用注入攻击成功率为 **100%**。具有基本生命周期钩子的框架可将成功率降低至 **~12%**。

### 5.5 维度五：评估

**V 组件必须捕获的内容：**
- **任务成功率**（核心指标）
- **轨迹效率**（步骤数、token 消耗）
- **错误恢复率**（从故障中恢复而不失败的能力）
- **安全性/稳健性指标**（工具注入失败率、越狱鲁棒性）

**关键基础设施：** HAL（Harness Abstraction Layer）实现了跨框架的标准化评估，使得在首次将 DeepAgents 接入 HAL 时发现了 47 个先前未知的错误。

## 6. 对比与选定结果

| 特性 | 传统软件框架（JUnit） | LLM 智能体框架 |
|---|---|---|
| 执行主体 | 确定性代码 | 概率性 LLM |
| 失败模式 | 断言失败、异常 | 幻觉、注入、无限循环 |
| 治理粒度 | 测试套件级别 | 每次工具调用级别 |
| 可观测性 | 测试报告 | 轨迹 + 中间状态 + 安全性日志 |

**选定基准测试：**

| 基准测试 | 最佳框架实现 | 备注 |
|---|---|---|
| SWE-bench-Lite | OpenHands（CodeAct v2）— 43.0% | 仅需 1.6 秒/实例 |
| TerminalBench-2 | Meta-Harness（自动搜索框架）— 76.4% | 超越手工编码的 DeepAgents（62%） |
| GAIA（验证分割） | OpenClaw（Alpha）— 41.3% | 见下方关键方法论说明 |

**方法论警示：** GAIA 结果——随机种子产生高达 17% 的方差（OpenClaw 报告为 41.3%，但在不同的种子下同一系统运行结果为 24.2%–40.8%）。**任何少于 5 次运行的基准测试报告都不可靠。** 框架研究对评估标准化的需求甚至超过了对模型研究的需要。

## 7. 开放性挑战与未来方向

1.  **形式化验证的框架设计：** 将框架组件映射到可验证的正确性属性（预期行为、安全约束）。以 L 为例：你能否形式化地证明给定的一组生命周期钩子能防止所有工具注入攻击？
2.  **自动框架搜索（Meta-Harness）：** 自动化框架配置优化——超越了手动提示搜索。
3.  **多模型框架：** 异质模型池——并非所有模型都需要（或应当）获得相同的框架治理。
4.  **标准化协议（MCP、A2A）：** 朝着可互操作的框架组件发展，但扩大的攻击面需要创新的安全方案。
5.  **评估标准化：** 需要采用类似 HAL 的通用接口，以便跨框架进行可复现、可靠的基准测试。

---

## 8. 给从业者的经验教训

1.  **从精简开始：** 由 2-3 个工具组成的最小注册表往往优于过度暴露的注册表。
2.  **首先实施 E 和 T：** 只要有了可靠的执行循环和工具注册表，你就拥有了一个功能性的最小框架。
3.  **尽早添加生命周期钩子：** 框架越晚获得安全机制，攻击者就越容易利用其运行时中的漏洞。
4.  **在所有三种模式（无框架、原生框架、外部框架）下对模型进行基准测试：** 框架选择对排名的影响可能与模型选择一样大。
5.  **投资于 V（评估接口）：** 没有良好的评估接口，你就无法衡量改进或发现回归问题。
