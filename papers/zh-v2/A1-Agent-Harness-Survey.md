## 核心命题：装备层，而非模型，是约束条件

> "装备层，而非模型，是真实世界 Agent 系统性能的绑定约束。"

本文的核心主张是：Agent 任务执行的可靠性更多地依赖于**基础设施层（"Agent 执行装备层"）**而非底层模型的能力。论文对该层进行了形式化定义、分类法和系统分析，认为它是一个需要专门研究的独立架构对象。

---

## 一、核心定义：形式化框架 H=(E,T,C,S,L,V)

Agent 装备层被形式化定义为一个**六组件运行时治理系统**：

> **H = (E, T, C, S, L, V)**

| 组件 | 名称 | 功能描述 |
|------|------|---------|
| **E** | 执行循环（Execution Loop） | 管理观察→思考→行动的循环，处理轮次排序、终止条件和错误恢复。形式化为带有安全性（不失控）和活性（可达终止状态）属性的标记转移系统（LTS） |
| **T** | 工具注册表（Tool Registry） | 类型化、经过验证的可用工具目录。路由、监控和管理工具调用 |
| **C** | 上下文管理器（Context Manager） | 管理哪些信息进入模型的上下文窗口。管理压缩、检索和优先级策略 |
| **S** | 状态存储（State Store） | 跨轮次和会话持久化任务相关状态；支持从部分失败中恢复 |
| **L** | 生命周期钩子（Lifecycle Hooks） | 调用前后的拦截：认证、日志、策略执行和仪表化 |
| **V** | 评估接口（Evaluation Interface） | 标准化的工具化接口，用于捕获行动轨迹、中间状态和成功信号，支持离线分析和跨装备层对比。与操作日志（L）明确区分 |

**关键边界判断：**
- **最简装备层**：实现了 E 和 T
- **不算装备层**：ReAct（无错误恢复，无正式注册表）。LangGraph（框架，缺失 C/S/L/V）
- **能力模块**：MemGPT（强 C 和 S，缺失 E/T）

## 二、证据矩阵：装备层改变 > 模型改变

| 来源 | 干预措施 | 性能变化 | 模型是否改变？ |
|------|---------|---------|-------------|
| 从业者（Pi Research） | 改变编辑工具格式 | 6.7% → **68.3%**（10倍） | 否 |
| 从业者（LangChain DeepAgents） | 中间件、生命周期钩子、系统提示词 | 52.8% → **66.5%**（+26%） | 否 |
| 预印本（Meta-Harness, Stanford/MIT） | 对装备层空间的自动化 Agent 搜索 | **76.4%** on TerminalBench-2（超越手工设计） | 否 |
| 预印本（AgencyBench） | 模型在原生 vs. 外部装备层 | 原生生态得分显著更高（如 48.4%） | 否 |
| 从业者（Vercel） | 工具从 15 个减少到 2 个 | 80% → **100%** 准确率 | 否 |

**关键行业报告：**
- **OpenAI Codex**：创造了"装备层工程"一词。团队构建了约 100 万行生产代码，零手写代码
- **Stripe Minions**：~1300 个 PR/周。精心策划的工具注册表至关重要；暴露所有工具会降低性能
- **OpenClaw PRISM**：首次公开发布的开源装备层运行时安全层

## 三、历史演变："装备层转向"

装备层概念从三个谱系中结晶：

1. **软件测试装备层（JUnit）**：治理包装器模式
2. **RL 环境（OpenAI Gym）**：接口标准化（`step()`/`observe()`）
3. **早期 LLM 智能体（ReAct、AutoGPT、MemGPT）**：定义所需治理的失败模式目录

**"装备层转向"（2024–2026）**标志着从以模型为中心的研究到以基础设施为中心的工程的转变，催化剂包括：协议标准化（MCP、A2A）、形式化（AIOS）和大规模评估基础设施（HAL、AgencyBench）。

**三个工程范式：**
- **提示词工程（2022–2024）：** "给什么文本？"
- **上下文工程（2025）：** "结构什么信息？"
- **装备层工程（2026）：** "设计什么治理？"

## 四、系统分类法（22 个系统分析）

| 类别 | 示例系统 | 组件覆盖 |
|------|---------|---------|
| **全栈装备层** | DeepAgents、OpenClaw、OpenHands、AIOS、Claude Code | 完整 E、T、C、S、L、V |
| **框架** | LangGraph、AutoGen、LlamaIndex | 部分 E 和 T；C、S、L、V 通常缺失（委托） |
| **能力模块** | MemGPT（C/S）、MCP 服务器（T）、Voyager（S） | 擅长 1-2 个组件；缺乏编排 |
| **评估基础设施** | HAL、AgencyBench、SWE-bench、OSWorld | 广泛的 E、V；对领域可复现性至关重要 |

**关键发现：** **V（评估接口）**和 **L（生命周期钩子）**是目前最欠发达的组件。
