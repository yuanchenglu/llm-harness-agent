# Comprehensive Summary: "Agent Harness for Large Language Model Agents: A Survey"

**Paper:**
*   **Title:** Agent Harness for Large Language Model Agents: A Survey
*   **Authors:** Qianyu Meng, Yanan Wang, Liyi Chen, et al.
*   **Status:** Preprint (v1, submitted 03 April 2026)

## Core Thesis: The Harness, Not the Model, is the Binding Constraint

> "The harness, not the model, is the binding constraint for real-world agent system performance."

The central claim is that agent task execution reliability depends more on the **infrastructure layer (the "agent execution harness")** than on the underlying model's capabilities. The paper provides a formal definition, taxonomy, and systematic analysis of this layer, arguing it is a distinct architectural object requiring dedicated study.

---

## 1. Key Definition: The Formal Framework H=(E,T,C,S,L,V)

An agent harness is formally defined as a six-component runtime governance system:

> **H = (E, T, C, S, L, V)**

- **E — Execution Loop:** Manages the observe-think-act cycle, turn sequencing, termination conditions, and error recovery. Formalized as a **Labeled Transition System (LTS)** with safety (no runaway) and liveness (terminal state reachable) properties.
- **T — Tool Registry:** A typed, validated catalog of available tools. Routes, monitors, and governs tool invocations.
- **C — Context Manager:** Governs what information enters the model’s context window. Manages compaction, retrieval, and prioritization strategies.
- **S — State Store:** Persists task-relevant state across turns and sessions; enables recovery from partial failures.
- **L — Lifecycle Hooks:** Pre- and post-invocation interception for authentication, logging, policy enforcement, and instrumentation.
- **V — Evaluation Interface:** Standardized instrumentation for capturing action trajectories, intermediate states, and success signals for offline analysis and cross-harness comparison. **Distinct from operational logging (L).**

**Key Boundaries:**
- **Minimal Harness:** Implements E and T.
- **Not a Harness:** *ReAct* (no error recovery, no formal registry). *LangGraph* (framework, absent C/S/L/V).
- **Module:** *MemGPT* (strong C and S, absent E/T).

## 3. Historical Evolution: The "Harness Turn"

The harness concept crystallized from three lineages:
1.  **Software Test Harness (JUnit):** The governance wrapper pattern.
2.  **RL Environments (OpenAI Gym):** Interface standardization (`step()`/`observe()`).
3.  **Early LLM Agents (ReAct, AutoGPT, MemGPT):** The failure mode catalog defining what governance is needed.

The **"Harness Turn" (2024–2026)** marks the shift from model-centric research to infrastructure-focused engineering, catalyzed by protocol standardization (MCP, A2A), formalization (AIOS), and large-scale evaluation infra (HAL, AgencyBench).

**Three Engineering Paradigms:**
- Prompt Engineering (2022-2024): "What text to give?"
- Context Engineering (2025): "What information to structure?"
- **Harness Engineering (2026):** "What governance to design?"
