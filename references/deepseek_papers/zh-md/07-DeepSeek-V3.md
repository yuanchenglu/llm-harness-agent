# DeepSeek-V3 技术报告

**作者：** DeepSeek-AI
**论文编号：** arXiv:2412.19437v2 [cs.CL] | 2025年2月18日

---

## 摘要

我们推出 **DeepSeek-V3**，一个强大的混合专家（MoE）语言模型，拥有 671B 总参数（每 token 激活 37B）。DeepSeek-V3 采用经过 DeepSeek-V2 充分验证的多头潜在注意力（MLA）和 DeepSeekMoE 架构。在此基础上，DeepSeek-V3 开创了**无辅助损失（auxiliary-loss-free）负载均衡策略**，并设定了**多 token 预测（MTP）训练目标**以获得更强性能。我们在 14.8 万亿多样化高质量 token 上预训练 DeepSeek-V3，然后进行监督微调和强化学习阶段以充分发挥其能力。全面评估显示，DeepSeek-V3 超越其他开源模型，性能与领先闭源模型相当。尽管性能卓越，DeepSeek-V3 的全训练仅需 **278.8 万 H800 GPU 小时**，训练过程极其稳定，全程未出现不可恢复的损失尖峰或需要回滚。

---

## 1. 引言

近年来，大语言模型快速迭代演化，逐步缩小着与 AGI 的差距。为进一步推动开源模型能力边界，我们扩展了模型规模并推出 DeepSeek-V3。

架构方面，DeepSeek-V3 采用 MLA（高效推理）和 DeepSeekMoE（经济训练）。在这之上实施了两项额外策略：一是**无辅助损失负载均衡**——最小化鼓励负载均衡对模型性能的不利影响；二是**多 token 预测训练目标**——提升评估基准上的整体性能。

训练效率方面，我们支持 FP8 混合精度训练，首次在极大规模模型上验证其可行性。设计了 **DualPipe** 算法实现高效流水线并行，通过计算-通信重叠隐藏了大部分跨节点通信开销。

预训练在 14.8T token 上进行，过程极其稳定。随后进行两阶段上下文扩展（32K→128K）。后训练阶段包括 SFT 和 RL，并从 DeepSeek-R1 系列模型蒸馏推理能力。

训练总成本：仅 **278.8 万 H800 GPU 小时（约 557.6 万美元）**，其中预训练 266.4 万小时，上下文扩展 11.9 万小时，后训练 0.5 万小时。

## 2. 架构

### 2.1 基础架构

**MLA（多头潜在注意力）。** 与 DeepSeek-V2 相同的低秩键值联合压缩机制，显著减少推理时的 KV 缓存。

**DeepSeekMoE。** 采用细粒度专家分割和共享专家隔离。不同之处：DeepSeek-V3 使用 Sigmoid 函数计算亲和度分数，并对所有选中的亲和度分数进行归一化来产生门控值。

**无辅助损失负载均衡。** 创新性地引入偏置项 b_i 为每个专家，将其加到对应亲和度分数 s_i,t 上用于 Top-K 路由选择。通过动态调整偏置项来平衡负载，无需使用辅助损失。指标：在训练过程中监控整个 batch 上每个专家负载的溢出率和欠载率。

**序列级辅助损失。** 在单个序列级别补充使用细粒度辅助损失，确保每个序列内的负载也是均衡的。

### 2.2 多 token 预测（MTP）

扩展训练目标：让模型在每个位置同时预测多个未来 token。使用 D 个顺序的 MTP 模块，每个模块包含自注意力 + 交叉注意力，按深度方向预测。MTP 在推理时可删除，也可用于推测解码加速。

## 3. 基础设施

### 3.1 计算集群

配备 2048 块 H800 GPU 的集群，每节点 8 块 GPU，节点间通过 InfiniBand 互连，NVLink + NVSwitch 节点内互连。

### 3.2 训练框架

**DualPipe 算法。** 创新的双向流水线并行方案，通过将前向和反向计算块拆分为更细粒度的步骤，重叠计算和通信。相比传统 1F1B 方案，显著减少流水线气泡（仅 1.2% 的理论气泡占比）。

**高效跨节点 All-to-All 通信。** 开发了针对 IB 和 NVLink 带宽的优化通信 kernel，通过计算与通信的重叠实现接近零的跨节点 all-to-all 通信开销。

**极致内存节省。** 通过细粒度"专家卸载"和 Re-Computation（重计算）策略，在不使用张量并行的情况下训练整个模型。

### 3.3 FP8 训练

首次在极大规模模型上验证 FP8 训练的可行性：

- **混合精度框架**：对核心计算（矩阵乘法）使用 FP8，关键部分保持 BF16 精度
- **量化改进**：分块量化策略，对激活值使用 E4M3 格式，对权重使用 E5M2 格式
- **低精度存储和通信**：在反向传播期间将激活值压缩为 FP8，存储和通信开销减半

相比原生 BF16 训练，FP8 训练仅引入不到 0.25% 的相对损失偏差。

### 3.4 推理与部署

**Prefilling** 阶段使用更小的微批量 + 更长序列长度，最大化计算利用率。**Decoding** 阶段使用更大的批量 + 更短序列，采用多 GPU 冗余部署分摊 KV 缓存。

### 3.5 硬件设计建议

建议通信硬件在 IB 链路上整合更高带宽，计算硬件引入稀疏计算单元和低位宽数据路径。

## 4. 预训练

### 4.1 数据构建

14.8T 高质量多样化 token，相比 DeepSeek-V2 的 8.1T 大幅扩展。通过优化过滤和去重流程，显著提升数据质量。

### 4.2 超参数

| 参数 | DeepSeek-V3 |
|------|------------|
| 总参数 | 671B |
| 激活参数 | 37B/token |
| 隐藏维度 | 7168 |
| 注意力头数 | 128 |
| KV 压缩维度 | 512 |
| 层数 | 74 |
| 共享专家 | 1 |
| 路由专家 | 256 |
| 激活路由专家 | 8 |
| 训练 token 数 | 14.8T |
| MTP 深度 | 1 |

### 4.3 长上下文扩展

两阶段：先扩展至 32K，再至 128K。使用 YaRN 方法。

### 4.4 评估结果

| 基准 | DeepSeek-V3 | GPT-4o | Claude-3.5-Sonnet | LLaMA-3.1-405B |
|------|------------|--------|------------------|----------------|
| MMLU | **88.5** | 87.2 | 88.3 | 88.6 |
| MMLU-Pro | **75.9** | 72.6 | 73.3 | 73.3 |
| GPQA-Diamond | **59.1** | 53.6 | 65.0 | 49.0 |
| MATH-500 | **90.2** | 76.6 | 78.3 | 73.8 |
| AIME 2024 | **39.2** | — | 16.0 | 23.3 |
| HumanEval | **92.0** | 90.2 | 92.0 | 89.0 |
| SWE-bench Verified | **42.0** | 38.8 | 50.8 | 24.5 |

DeepSeek-V3 在知识、代码、数学推理方面均达到开源模型 SOTA，性能与 GPT-4o 和 Claude-3.5-Sonnet 相当。

### 4.5 讨论

**MTP 消融：** MTP 在标准评估（如 HumanEval、MBPP）和推理基准上均带来稳定提升。

**无辅助损失策略消融：** 在不使用辅助损失的情况下达到良好负载均衡，负载均衡效果与使用辅助损失相当。

**序列级 vs Batch 级均衡：** 序列级负载均衡在推理任务上表现略优。

## 5. 后训练

### 5.1 监督微调

收集 SFT 数据（包含推理、代码、数学、写作、角色扮演等），结合 R1 蒸馏数据，提升推理能力同时保持输出风格可控。

### 5.2 强化学习

使用 GRPO 算法。奖励模型涵盖正确性奖励（如编译器反馈、测试用例）和格式奖励（如思维链格式、语言一致性）。

### 5.3 评估结果

| 基准 | DeepSeek-V3 | GPT-4o | Claude-3.5-Sonnet |
|------|------------|--------|------------------|
| AlpacaEval 2.0 (LC) | **49.6** | 46.4 | 52.4 |
| MT-Bench | **9.00** | 8.56 | 8.81 |
| Arena-Hard | **85.5** | 79.8 | 84.5 |
| LiveCodeBench | **39.2** | 38.8 | — |

### 5.4 讨论

**从 DeepSeek-R1 蒸馏。** 通过将 R1 的长思维链蒸馏到 V3，显著提升了推理性能，特别是在数学和代码基准上。

**自奖励。** 探索了使用模型自身作为奖励模型的可能性。

**MTP 评估。** 在推理过程中移除 MTP 头仅使用主模型评估时，MTP 预训练带来的收益仍然保持。

---

## 6. 结论、局限与未来方向

**训练成本总结：** 278.8 万 H800 GPU 小时（约 557.6 万美元），是同等规模模型中极低的训练成本。

**局限：**
- 推理时的部署成本仍然较高（671B 总参数）
- 推理时采用冗余部署来分摊 KV 缓存，效率有待提升
- 尚未将 MoE 架构与长思维链模型（R1）更深度整合

**未来方向：**
- 在专家并行中优化不均衡负载
- 进一步推进 FP8+ 低精度训练
- 探索更高效的推理方案
