1 核心挑战
挑战 1

并行草稿质量衰减

并行草稿器独立预测每个 token，缺乏块内依赖建模。当上下文有多种合理延续时（如"of course" / "no problem"），可能产生不连贯组合（如"of problem"），接受率沿块快速衰减。

DSpark 方案

半自回归架构

保持并行主干（DFlash）的高吞吐，附加轻量级顺序头（马尔可夫头 / RNN 头）注入局部转移信息，以极小延迟代价大幅提升接受率。

挑战 2

验证计算浪费

不加区分地验证所有草稿 token 会将批次容量浪费在高拒绝风险的 token 上。高并发场景下，不必要的验证严重降低系统吞吐量。

DSpark 方案

置信度调度验证

置信度头预测每位置接受概率 + 硬件感知前缀调度器动态裁剪低置信度后缀，将验证预算仅分配给预期回报最高的 token。

2 一个完整的 DSpark 解码周期
P

并行主干生成

以锚点 token D 为输入，DFlash 并行主干一次前向传播产生所有位置的隐藏状态和基础 logits。草稿延迟与块大小 γ 几乎无关。

S

顺序头注入依赖

轻量级顺序头（马尔可夫 / RNN）从左到右采样，以之前已采样的 token 为条件注入转移偏差。缓解跨模式冲突，生成连贯的草稿序列 EFGH。

C

置信度评估 + 调度

置信度头输出每位置接受概率 c₁–c₄。硬件感知调度器结合实时引擎吞吐量特征，保留预期回报最高的前缀（EFG），丢弃高拒绝风险的 token（H）。

V

目标模型验证

目标模型并行验证调度后的前缀。E、F 被接受，G 被拒绝并生成修正 token G*。周期结束，G* 成为下一轮锚点。

并行主干（DFlash）一次前向生成所有草稿 → 轻量级顺序头（马尔可夫 / RNN）从左到右注入 token 间依赖。Tdraft 几乎不变，但 τ 大幅提升。

苏格拉底问答

从浅入深，逐步理解 DSpark 的核心思想与技术细节

1 LLM 推理为什么慢？推测解码怎么解决？ ▾

问题根源：LLM 是自回归的——每生成一个 token 都需要一次完整的前向传播。这意味着生成 1000 个 token 就需要 1000 次前向传播，延迟与输出长度成正比。

推测解码的思路：用一个小而快的"草稿模型"先快速生成一段候选 token，然后大模型（目标模型）一次性验证这整段。验证是并行的——目标模型一次前向传播就能检查所有候选 token。如果草稿质量好，一个周期就能前进多个 token，大幅减少目标模型的调用次数。

核心公式：每 token 延迟 L = (T_draft + T_verify) / τ，其中 τ 是每周期接受的 token 数。

基础 加速的关键就是：让 τ 尽可能大，同时让 T_draft 和 T_verify 尽可能小。

2 现有草稿器有什么问题？为什么 DSpark 要"半自回归"？ ▾

现有方案有两种，各有各的毛病：

自回归草稿器（如 Eagle3）： 每个 token 依赖前面的 token，质量高，但是顺序生成——草稿延迟随块大小线性增长（Tdraft ∝ γ）。只能用短块，限制了加速潜力。

并行草稿器（如 DFlash）： 一次性生成所有草稿 token，速度极快（Tdraft 几乎与 γ 无关），但每个位置独立预测，缺乏 token 间依赖。假设上下文说"好的，____"，合理的回答可能是"当然"或"没问题"。并行草稿器可能预测出"当然问题"这种不连贯的组合，因为位置 1 选了"当然"，位置 2 却选了"问题"。

DSpark 方案 半自回归架构：既要并行速度，又要自回归质量。并行主干处理主要计算，一个轻量级顺序头再从左到右注入 token 间依赖，修正不连贯的组合。主干带来速度，顺序头带来质量，两者兼得。

3 顺序头具体怎么工作？马尔可夫头和 RNN 头有什么区别？ ▾

顺序头的核心公式：p_k(v) ∝ exp(U_k(v) + B_k(...))

其中 Uk 是并行主干给出的基础分数，Bk 是顺序头注入的"转移偏差"——它根据前面已经采样的 token 来调整当前 token 的分数。

马尔可夫头：只依赖前一个 token。一阶转移 B(x_{k-1}, x_k)，用低秩分解 W₁ × W₂ 实现。上面例子中，一旦位置 1 采了"当然"，马尔可夫头就会提升"没问题"、抑制"问题"。计算量极小。

RNN 头：维护一个循环状态 sk，累积整个前缀的历史。可以依赖前面所有的 token，而不只是上一个。建模能力更强，但计算量相应增加。

关键 顺序头必须足够轻量（Tsequential ≪ Tparallel），这样总草稿延迟仍然由并行阶段主导。

4 为什么不能永远验证全部草稿 token？"置信度调度"解决什么问题？ ▾

直觉上，草稿块越长越好——生成更多 token，总有机会多接受几个。但问题在于：验证 token 不是免费的。

在高并发系统中，目标模型需要同时服务多个请求。每次验证都在批次容量里占一个位置。验证一个几乎肯定被拒绝的 token，等于占用了一个本可以服务其他活跃请求的位置。这会导致整体吞吐量下降。

而且接受率不是均匀的——代码问题接受率高，开放对话接受率低。固定长度的验证策略无法适应这种变化。

DSpark 方案 置信度调度验证：不固定验证长度，而是为每个位置预测接受概率，然后动态决定保留哪些 token。只验证"值得"的 token，扔掉几乎肯定被拒绝的尾巴。

5 置信度头和硬件感知调度器具体是怎么配合的？ ▾

这是两个组件的前后配合：

1️⃣ 置信度头：一个轻量级线性投影 + sigmoid，以主干隐藏状态和前一个 token 的嵌入为输入，输出 c_k ∈ (0,1)——位置 k 的草稿 token 被条件接受的估计概率（假设前面所有 token 已经通过验证）。训练时用确切的总变差距离来监督。

但神经网络经常过于自信。因此用 顺序温度缩放（STS） 校准：从左到右逐个位置做网格搜索，找到使累积乘积的期望校准误差最小的温度系数。校准后，累积概率 ∏cᵢ 能可靠地估计完整前缀的存活概率。

2️⃣ 硬件感知调度器：形式化为一个全局吞吐量最大化问题。输入是一批请求的置信度序列 + 预分析的引擎吞吐曲线 SPS(B)。贪心算法逐个添加"存活概率最高"的 token，计算期望吞吐量 Θ = τ · SPS(B)，一旦吞吐量开始下降就停止。

效果 轻负载时验证所有 token（几乎免费），重负载时智能裁剪——把宝贵的批次容量留给最有希望的 token。

6 DSpark 在真实部署中表现如何？比之前的方案好在哪？ ▾

DSpark 部署在 DeepSeek-V4 服务系统中，经受真实用户流量考验：

离线基准（Qwen3 系列）：

  • 比自回归 Eagle3：接受长度提升 26.7%–30.9%
  • 比并行 DFlash：接受长度提升 16.3%–18.4%

在线部署（vs MTP-1 生产基线）：

  • V4-Flash：单用户生成速度提升 60%–85%
  • V4-Pro：单用户生成速度提升 57%–78%

关键价值 更关键的是，在严格的 SLA 约束下，基线容量严重恶化，而 DSpark 通过避免不必要的验证，保持了稳健的吞吐量。这推动了 LLM 服务的帕累托前沿——解锁了之前不可能达到的交互层级。

7 DSpark 开源了什么？别人怎么用？ ▾

DSpark 团队开源了两样东西：

1. DSpark 检查点：针对 DeepSeek-V4-Flash（预览版）和 DeepSeek-V4-Pro（预览版）训练好的模型权重。可以直接用于推理加速。

2. DeepSpec 训练仓库：一个算法驱动的推测解码训练仓库，包含 Eagle3、DFlash 和 DSpark 的实现。研究人员可以基于此进行二次开发。

社区 这使得整个推测解码领域有了一个统一的开源基准平台。

8 一句话总结：DSpark 到底做了什么？ ▾

DSpark 解决了并行推测解码中的两个核心矛盾：

更好地草稿 + 更聪明地验证

更好地草稿：用半自回归架构（并行主干 + 轻量级顺序头）同时获得并行速度和自回归质量。

更聪明地验证：用置信度头 + 硬件感知调度器动态裁剪低质量后缀，把 GPU 算力花在最值得验证的 token 上。

结果 在 DeepSeek-V4 生产系统中，将单用户生成速度提升了 60%–85%，并推动了 LLM 服务的性能极限。
