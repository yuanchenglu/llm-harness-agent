**摘要：** 近年来，大型语言模型（LLM）在理解和生成代码方面展现出强大能力，涵盖从竞赛编程到仓库级软件工程等多个领域。在新兴的智能体系统中，代码不再仅仅是目标输出，它日益成为智能体推理、行动、环境建模以及基于执行的验证的操作性基础。我们将这一转变置于"智能体支架"（agent harness）的视角下，并引入**代码即智能体支架**（code as agent harness）这一统一观点，将代码视为智能体基础设施的核心基础。为系统性地研究这一视角，我们围绕三个相互关联的层次组织本综述。首先，研究**支架接口**（harness interface），即代码如何连接智能体的推理、行动和环境建模。其次，考察**支架机制**（harness mechanisms）：用于长周期执行的规划、记忆和工具使用，以及使支架可靠且自适应的反馈驱动控制与优化。第三，讨论**支架的规模化扩展**（scaling the harness），从单智能体系统走向多智能体设置，其中共享的代码制品支持多智能体协调、审查和验证。在这些层次中，我们总结了"代码即智能体支架"的代表性方法和实际应用，涵盖编码助手、GUI/OS自动化、具身智能体、科学发现、个性化和推荐、DevOps和企业工作流。我们进一步概述了支架工程面临的开放性挑战，包括超越最终任务成功的评估、不完整反馈下的验证、无回归的支架改进、跨多智能体的一致共享状态、面向安全关键操作的人工监督，以及向多模态环境的扩展。通过将代码置于智能体AI支架的中心位置，本综述为构建可执行、可验证、有状态的AI智能体系统提供了统一的路线图。

**关键词：** 智能体支架，编码智能体，支架工程，智能体AI

# 1. 引言

近年来，大型语言模型（LLM）在理解和生成代码方面展现出强大能力[1,2,3]，在从竞赛编程[4]到仓库级软件工程[5]等任务中均取得了优异表现。基于这些能力，代码在智能体系统中的作用正在扩展，不再仅仅是待生成的目标制品。程序越来越多地被用作LLM智能体推理、行动和建模其环境的媒介。程序辅助推理方法将中间计算外部化为可执行代码[6,7,8]；机器人和具身智能体使用生成的程序作为与物理或模拟世界交互的可执行策略[9,10]；软件工程或交互环境使用代码库、执行轨迹、测试和运行时反馈作为环境状态和动态的结构化表示，智能体在其中规划、行动和修正其行为[11,5,12]。综合来看，这些发展暗示了一个更广阔的观点：代码不仅是LLM生成的制品，还是一个可执行、可检查、有状态的媒介，智能体通过它进行推理、行动、观察反馈和验证进展。我们将这一观点称为**代码即智能体支架**。

近期关于**智能体支架**（agent harnesses）的讨论[13,14,15,16]为理解这一转变提供了有用的系统级视角。智能体支架指的是围绕LLM的软件层，包括工具、API、沙箱、内存、验证器、权限边界、执行循环和反馈通道，从而将无状态模型转变为能够执行长期任务的功能性智能体[17,18,19,20,21,22,23]。在这一视角下，自主性的瓶颈不仅在于基础模型的推理能力，还在于连接模型输出与长周期行动和持久状态的系统的可靠性。

为澄清代码在这一更广泛的支架视角中的角色，我们区分了长周期智能体系统的三个耦合要素：**模型内部能力**（model-internal capabilities）、**系统提供的支架基础设施**（system-provided harness infrastructure）和**智能体发起的代码制品**（agent-initiated code artifacts）。模型内部能力指模型的推理、感知、规划、模拟和评估能力。系统提供的支架基础设施指预定义的工具、API、沙箱、内存系统、验证器、权限边界、遥测和工作流，它们将模型输出连接到外部行动和反馈，构成支架工程的主要关注点[24,25]。相比之下，智能体发起的代码制品（相对尚未充分探索）是智能体在任务执行循环中创建、执行、观察、修改、持久化和共享的交互式代码对象。通过执行反馈，这些制品帮助智能体推理、行动、验证进展、存储状态和与其他智能体协调。示例包括回归测试、临时工具、DSL程序、可执行工作流、可复用技能和中间程序状态。代表性系统如Claude Code[26]、Codex[27]、LangChain[28]和企业智能体平台展示了这些要素如何共同实现在长周期智能体系统中的适应。

基于这一区分，我们重新审视代码在智能体系统中的作用。现有综述通常将代码视为LLM的最终产品。相比之下，我们聚焦于**智能体发起的代码制品**，以及模型能力如何通过与支架基础设施的交互来构建和演化这些制品，代码在其中充当接口、智能体能力和多智能体协调的组织中心。在多样化的智能体系统中，代码不仅用于产生解决方案，还用于执行推理、落地行动、维护状态和暴露反馈。我们将这一观点称为**代码即智能体支架**：代码作为可执行和可检查的媒介，智能体通过它推理、行动和适应。这使关注点从生成正确程序转向理解代码如何支持可靠的闭环智能体行为。

为系统性地刻画**代码即智能体支架**，我们按三个相互关联的层次组织本综述，如图1所示。这种组织方式遵循代码如何成为智能体循环中的操作性媒介：它首先作为推理、行动和环境表示（§2）的支架接口进入；然后支持管理规划、内存、工具使用、执行和随时间修复的支架机制（§3）；最后成为多个智能体通过仓库、测试、轨迹、工作流和执行状态进行协调的共享制品（§4）。

**简要预览各节：**

**首先，支架接口：用于推理、行动和环境建模的代码（§2）** 研究代码如何构成模型与其任务环境之间的基本接口。在这一层，代码是将模型输出转换为可执行和可检查结构的媒介。我们回顾**用于推理的代码**，其中程序将中间计算外部化，允许解释器、符号求解器、执行轨迹或过程奖励来检查和优化推理[7,6,8,29,30,31]。然后回顾**用于行动的代码**，其中生成的程序作为策略、工具调用、行为树或可复用技能用于具身、GUI和软件环境[9,10,32,33,34,35]。最后考察**用于环境建模的代码**，其中程序状态、仓库、轨迹、模拟器和测试表示智能体交互的状态、动态和反馈信号[36,37,38,5,12,39]。这一层建立了核心支架接口：代码使推理可执行、行动可编程、环境状态可检查。

**其次，支架机制：规划、内存、工具使用、控制和优化（§3）** 研究代码支架化的智能体如何在单次生成步骤之外保持可靠。一旦代码被放入智能体循环，支架必须决定接下来执行什么、保留有用的状态、暴露正确的工具，并将失败转化为纠正行动。因此，我们回顾通过分解、结构接地、轨迹搜索或工作流编排来组织长周期软件任务的规划方法[40,41,42,43,44]；维护工作状态、检索仓库证据、存储可复用体验和支持共享交互历史的内存方法[45,46,47,48]；将智能体连接到API、仓库、执行环境和验证工具的工具使用方法[19,49]；以及使用静态分析、运行时错误、测试和人类反馈通过重复执行来修改代码的反馈驱动控制和支架优化方法[50,51,52,53]。这一层将§2中的接口转化为操作性支架：规划控制执行轨迹，内存保持状态，工具扩展行动空间，反馈驱动适应在失败和修改之间形成闭环。

**最后，规模化支架：多智能体代码编排（§4）** 将支架从单个智能体扩展到协作生态系统。当多个智能体在代码上运行时，支架不仅需要支持个体推理和执行，还需要协调角色、共享中间制品、维护公共状态和验证集体进展。我们通过智能体角色（如管理者、规划者、编码者、审查者和测试者）、协作模式（如编程、修复、辩论、红队测试和对抗交互）以及工作流拓扑（从集中式协调到分布式或流式协作）来回顾以代码为中心的多智能体系统[54,55,56]。这一层展示了代码如何成为编排式自主性的共享支架：仓库、测试、轨迹和结构化制品提供了智能体协调、检查和改进彼此行为的公共工作空间。

在分类之外，我们考察智能体发起的代码交互在五个应用领域中的表现。在编码辅助中，智能体在实时仓库上编写补丁、测试和问题解决工作流[5,57,58]。在GUI和OS自动化中，智能体基于DOM树、可访问性API和可执行评估器合成和执行界面命令[59,60]。在科学发现中，智能体动态组合和执行涵盖模拟、实验室协议和数据分析的假设测试管道[61,62,63,64]。在个性化和具身控制中，智能体根据环境反馈编写和修改可执行策略、模拟器和技能库[9,10,32]。我们进一步概述支架工程的开放性挑战，包括超越最终任务成功的评估、不完整反馈下的验证、无回归的支架改进、跨多智能体的一致共享状态、人工监督和向多模态环境的扩展。本综述提供了研究代码的路线图——不仅作为智能体生成的东西，还作为它们执行、适应和协调可靠行为的运行时媒介。

# 2. 支架接口：用于推理、行动和环境建模的代码

支架通过将无状态语言模型的输出建立在外部执行、持久状态和可验证反馈的基础上，将其转变为功能性智能体。因此，任何支架最根本的设计问题是：**什么媒介将模型连接到其任务环境？**

我们认为答案是代码。与自然语言不同，代码是**可执行的**，意味着模型输出变为具有形式上可验证结果的操作；**可检查的**，意味着中间计算作为结构化轨迹暴露出来，支架可以读取、存储和对其采取行动；**有状态的**，意味着演化中的程序以持久、可修改的形式跨步骤表示任务进展。关键的是，这些不仅仅是代码作为符号表示的特性；它们是使代码能够作为支架接口发挥功能的特性。可执行性意味着支架可以验证模型的意图。可检查性意味着可以诊断失败并反馈。有状态性意味着智能体的交互历史不会在步骤之间丢失。

**范围界定。** 我们广泛使用"代码"一词，但不是隐喻性的。在本综述中，代码指的是可执行或可机器检查的制品，包括程序、脚本、形式规范、证明脚本、API模式、工具定义、测试、仓库、模拟器、配置文件以及与代码相邻的执行制品（如轨迹和日志），只要它们是由可执行系统产生或被其消费。相比之下，原始感知、物理状态、人类意图和模型内部的潜在推理本身不是代码。它们可能通过代码被感知、估计、序列化、验证或采取行动，但不应与代码接口相混淆。这一边界很重要，因为代码作为支架接口并不取代感知、具身化、人类目标或模型推理；相反，它使这些方面的选定部分在智能体循环中变得可执行、可检查和有状态。

我们围绕代码在智能体系统中扮演的三个角色来组织这个接口。**用于推理的代码**将内部逻辑外部化为可验证的计算，允许外部解释器、符号求解器、执行轨迹或过程奖励来检查和优化推理（§2.1）。**用于行动的代码**将高层次意图转化为在具身、GUI、软件或工具使用环境中可执行的操作（§2.2）。**用于环境建模的代码**通过程序状态、仓库、模拟器、测试和日志来表示世界状态、转换动态和反馈信号，智能体可以执行、编辑和查询这些内容（§2.3）。总体而言，这些角色定义了支架接口：代码使推理可执行、行动可编程、环境状态可检查。

## 2.1 用于推理的代码

智能体支架的一个核心作用是将模型推理从瞬态的文本生成转变为可执行和可验证的计算。早期的提示技术如纯思维链（CoT）[65]完全在**自然语言**中进行推理和计算，迫使模型在单个潜在的文本过程中既分解问题又执行中间操作。虽然语言模型通常能有效地提出推理步骤，但它们在忠实执行符号、逻辑或算术计算方面仍然不可靠[7]。更重要的是，纯文本推理使智能体支架几乎无法验证中间状态、检查执行行为或跨步骤保持计算进展。

**用于推理的代码**因此引入代码作为模型和支架之间的执行接口，超越纯文本推理。模型生成可执行程序，外部运行时、解释器、符号求解器或验证模块可以执行和评估这些程序。这分离了高层推理和低层计算：模型提出过程，而支架执行它们、观察运行时行为、存储中间状态，并将执行结果反馈到未来的推理中。

近期工作进一步将这一接口从作为外部计算器的程序执行扩展到作为可复用推理信号的执行制品。输入和输出、执行轨迹、变量状态、控制流结构和函数级测试都可以作为中间状态，支架可以验证、评分并反馈到后续推理中。现有工作因此可以组织为三种范式：程序委托推理、形式验证与符号推理接口，以及迭代代码接地推理。

### 2.1.1 程序委托推理

程序委托推理使用可执行程序作为问题分解和计算之间的主要接口。模型不是仅仅依赖自然语言推理，而是生成代码，由外部解释器执行以产生形式上可靠的输出。早期工作[66,7]证明，将计算委托给程序通过将中间推理移入结构化、可验证的执行轨迹，显著提高了可靠性。思维程序（PoT）提示[6]通过明确将推理分解为可执行程序进一步系统化了这一范式，随后有POET[67]和MathCoder[68]等扩展，提高了执行保真度和领域专门化。后续研究考察了程序委托有效的条件，包括执行正确性、任务结构和运行时交互的作用。例如，Chain of Code（CoC）[8]和CIRS[69]分析了可执行推理如何相对于纯语言推理改变失败模式。后续方向将这一接口扩展到孤立的任务执行之外。跨语言推理框架[70]证明，基于程序的推理可以通过共享的可执行结构在不同语言环境中泛化，而基于方法的推理[71]引入了跨任务持久化的可重用程序化过程。更近期的系统如CodeAdapt[72]进一步表明，将语言模型与可执行推理接口紧密耦合可以超越专门的推理导向模型。此外，CodeI/O[73]将上下文相关的程序转换为代码输入输出预测任务，暴露出诸如逻辑流规划、状态空间搜索、决策树遍历和模块化分解等推理原语，同时通过可执行验证保持过程的严谨性。

### 2.1.2 形式验证与符号推理接口

混合神经符号方法将灵活基于语言的推理与结构化符号计算相结合，使用代码和符号制品作为持久的中间表示，而非将程序仅视为生成的文本。早期的图思维（Graph-of-Thoughts）[74]将思维链推理推广为图结构轨迹，使中间状态可以分支、合并和重用。在此基础上，自验证反思[75]、MA-LoT[76]和苏格拉底式自优化（Socratic self-refine）[77]引入了迭代验证循环，其中符号一致性检查指导生成解路径的优化。

近期工作通过基于代码的接口进一步收紧了神经生成和符号执行之间的耦合。CodeSteer[78]和Code-as-Symbolic-Planner[79]明确协调自由形式的语言推理与可执行的符号操作，将程序视为支架可以跨多个阶段检查、转换和执行的结构化基板。VisualCoder[80]通过控制流表示使程序行为可视化，将这一思想扩展了一步。通过将生成的推理与可视化的控制流图和执行路径对齐，它将动态程序行为转变为可检查的制品，用于程序行为预测。这些方法共同将神经符号接口从文本代码扩展到多模态执行制品，支架可以引用、验证和重用这些制品。

另一个相关工作线使用**机器可验证的形式语言**作为推理接口本身。证明助手如Lean[81]、Isabelle[82]和Coq[83]提供了基于严谨逻辑基础的形式证明语言，使每个推导步骤都可以由验证器检查。早期的基于LLM的定理证明系统，包括ReProver[84]、DeepSeek-Prover[85]和TheoremLlama[86]，建立了将语言模型与证明助手反馈相结合的实用方法。更近期的系统，如DeepSeek-Prover-V2[87]、Kimina-Prover[88]、MA-LoT[76]和Goedel-Prover-V2[89]，通过深思熟虑的证明搜索、自纠正和重复证明生成与验证改进了这一过程。形式验证接口也在超越数学定理证明进行扩展。HybridReasoning[90]应用形式证明器支持自然语言推理；Lean4Physics[91]和PhysLib[92]将基于Lean的验证扩展到物理学；VERINA[93]和Goedel-Code-Prover[94]将形式方法应用于代码验证。Lean4Agent[95]进一步将这一轨迹扩展到智能体系统，使用Lean4来建模和验证智能体工作流和轨迹。从支架的角度看，这些系统展示了形式语言如何不仅作为推理工具，还作为约束、认证和审计智能体行为的可执行契约。

### 2.1.3 迭代代码接地推理

迭代代码接地推理专注于生成、执行和反馈之间的闭环交互。在这些系统中，推理不是单次通过的过程，而是基于可执行状态转换的迭代计算轨迹。早期工作如NExT[30]训练模型通过推理程序轨迹来预测执行行为，从而将中间推理建立在运行时语义之上。相关工作[96]同样强调，可执行轨迹提供了比最终文本输出更丰富的监督信号。在此基础上，后续方法引入了显式的生成-执行-验证-优化循环。CodePRM[31]和ORPS[97]等方法使用执行结果来评估和优化中间推理轨迹，使支架能够通过运行时反馈而不是纯下一个词预测来引导推理。在同一方向上，CYCLE[98]和Self-Edit[99]等系统使用执行感知的修正信号迭代地修改生成的解决方案。

强化学习进一步加强了这一范式，将执行反馈视为推理轨迹上的优化信号。CodeRL[100]、CodeRL+[101]和RLTF[102]等方法通过基于单元测试的奖励来优化功能正确性，而StepCoder[103]等方法在优化过程中纳入了细粒度的编译器和运行时反馈。RLEF[104]将这种交互形式化为基于多步执行反馈的策略优化，使推理策略能够通过迭代的运行时交互来适应。更近期的方法走向完全交互式的推理环境。例如，EG-CFG[21]在生成过程中直接注入执行信号以支持逐步修正，而R1-Code-Interpreter[105]等系统在持久的交互会话中交错推理和多轮代码执行。

## 2.2 用于行动的代码

超越推理，智能体还必须将模型连接到外部环境，在其中决策产生真实的可执行效果。在这一阶段，代码主要不再作为计算的媒介，而是作为行动接口，将模型输出转换为落地的操作，如工具调用、机器人控制策略、GUI操作或软件命令。通过这个接口，支架将高层意图转化为能够与具身、数字和交互环境交互的可执行行为。

核心挑战因此是**接地**（grounding）：支架必须将抽象的语言输出映射到尊重目标环境约束的可执行行为上，包括具身限制、接口API、环境动态和安全要求。与用于推理的代码不同（解释器通常可以直接验证正确性），行动执行发生在部分观察和动态演化的环境中，失败可能通过无效状态转换、延迟反馈或静默执行错误而出现。例如，一个机器人可能试图抓取其工作空间之外的对象，而不会产生显式的运行时异常。

重要的是，可执行的行动代码是这些组件的接口，而不是它们的替代品。在具身环境中，感知模块提供观察，可负担性或可行性模型估计哪些动作是可能的，运动规划器和控制器将符号命令连接到传感器和执行器，安全层约束危险或无效的行为。在GUI和软件环境中，类似的组件包括屏幕解析器、DOM或可访问性树、后端API、用户意图模型、权限系统和程序化验证器。代码位于模型和这些组件之间：它序列化观察、调用接地和规划模块、调用可执行动作，并将验证结果暴露回支架。

**用于行动的代码**因此引入结构化可执行程序作为模型和环境之间的控制接口，使支架能够通过交互反馈来执行、监控、验证、重用和优化行动。这个接口可以通过不同的形式实现：预定义的技能库、生成的控制策略、持久的技能记忆、GUI/API工具协议，或显式的行动验证支架。AutoHarness[109]通过自动合成一个在LLM和环境之间进行中介的代码支架，在执行前过滤无效动作，使后一种形式变得明确。这突出了用于行动的代码的核心支架观点：代码不仅是要执行的动作，还是连接模型意图与感知、接地、可负担性估计、控制器、API、执行器和安全约束的可执行边界。

### 2.2.1 接地技能选择

接地技能选择研究智能体如何通过可重用的技能接口将高层语言意图映射为可执行行为。这些系统不是直接生成低层动作，而是将环境视为一组可执行能力的集合，智能体支架可以在环境约束下调用、组合和优化这些能力。SayCan[9]通过将语言规划与接地技能执行相结合确立了核心范式，使智能体不仅基于语义相关性还基于具身可行性来选择动作。后续工作从几个方向扩展了这一执行接口。KnowNo[110]通过一致预测引入了不确定性感知控制，使支架能够检测模糊状态并在不安全执行前触发澄清。BOSS[112]通过使用语言引导的实践来合成新的可执行技能链，解决了固定技能库的僵化问题，允许支架随时间扩展其行动空间。类似地，[113]通过使用LLM引导的生成来构建多样化的操作轨迹和可执行成功条件，用于自动重试和重新标记，解决了接地交互的数据瓶颈。超越静态执行，LRLL[114]引入了记忆和自我引导探索，以维护跨任务的持久和演化的技能接口。最后，SkillVLA[111]将这一范式扩展到组合性双臂交互，强调接地行动接口必须支持在日益复杂的具身设置中的结构化技能重用和重组。

### 2.2.2 程序化策略生成

程序化策略生成将代码本身视为模型和环境之间的控制接口。支架不是从预定义的技能中选择，而是直接将可执行策略具体化为指定控制逻辑、感知条件分支、反馈循环和API交互的程序。CaP[10]通过将LLM生成的Python程序框定为可执行的机器人策略，确立了这一范式。在此基础上，RoboCodeX[33]引入了多模态和树结构代码生成，以支持更复杂的操作和导航行为。后续工作专注于扩展交互基板。RoboPro[124]从大规模野外视频中合成可执行的策略代码，而Code-BT[34]将生成的程序编译为行为树控制器，支持约束执行和迭代运行时反馈。超越机器人学，CP-Agent[116]证明了持久的执行循环可以通过迭代执行和修复来支持形式约束求解智能体。为减少对昂贵物理环境的依赖，[117]将语言模型配置为机器人代码评估的静态执行模拟器。GenSwarm[118]进一步将程序化控制扩展到多智能体机器人系统，支架必须协调跨多个具身智能体的策略生成、约束分析和部署。在系统层面，NormCode[119]通过引入带有强制数据隔离的半形式化编程接口，强调治理和可审计性，使执行轨迹和控制逻辑保持可检查和受约束。最后，ALRM[115]和RACAS[120]将这些思想整合到持久的闭环控制架构中，在统一的智能体支架内集成代码生成、执行、监控和迭代交互。

### 2.2.3 终身代码型智能体

终身代码型智能体研究可执行交互接口如何在长周期交互中持续存在、演化和积累能力。在这些系统中，代码不仅是执行机制，还是持久的记忆基板，支架通过它存储可重用的行为、交互轨迹和环境知识。Voyager[32]通过自动课程和持续扩展的可执行技能库为Minecraft中的开放式交互确立了这一范式。将此思想扩展到具身环境，LRLL[114]引入了持久记忆、自我引导任务探索和技能抽象，以克服固定策略库的限制而不需要梯度更新。终身支架的一个核心挑战是交互反馈和修正通常是短暂的且难以重用。LYRA[121]通过将人工修正转换为可重用的可执行技能和检索增强的记忆结构来解决这个问题。类似地，ViReSkill[122]结合了视觉引导的重新规划和技能记忆缓存，以在环境失败和输出变异性下保持稳定的交互。近期工作进一步关注在持久部署下的持续适应和自演化。SkillsCrafter[123]引入持续语言条件操作结构，以减轻可执行能力积累时的灾难性遗忘，而UI-Voyager[35]通过失败驱动适应和自蒸馏将自演化交互范式推广到GUI智能体。这些系统共同超越了单次执行，走向持续扩展、优化和重用可执行交互接口的持久智能体支架。

## 2.3 用于环境的代码

智能体还必须维护其交互环境的显式表示。没有这样的表示，环境只能间接地通过文本观察、API返回或稀疏反馈信号暴露给智能体。结果，环境状态通常保持隐式、短暂且难以验证，使得跟踪状态转换、评估交互结果或在长周期任务中重用过去的交互历史变得困难。这一限制在复杂的软件、机器人和多步交互环境中尤为严重，成功的交互依赖于随时间保持一致的世界状态和接地反馈。

**用于环境的代码**通过引入可执行程序作为环境接口本身来解决这一限制。这些系统不将环境视为不透明的外部过程，而是通过计算制品如模拟器、仓库、测试、执行轨迹、日志和状态转换程序来具体化环境结构和动态。这使得智能体能够在整个交互过程中显式地存储、检查、执行和修改环境状态。通过可执行代码表示环境有两个主要优势。首先，可执行环境暴露可验证的状态转换，使智能体能够通过执行而非模糊的自然语言判断来评估交互结果。其次，基于代码的环境是持久和可修改的，智能体可以在交互过程中查询、模拟、编辑和优化它们。智能体支架不是通过语言与不透明的世界交互，而是可以将推理和行动建立在显式计算状态和运行时动态之上。现有工作在这一方向上可以组织为四个范式：结构化世界表示、执行轨迹世界建模、代码接地评估环境和可验证环境构建。

### 2.3.1 结构化世界表示

结构化世界表示通过显式的程序化结构来建模环境，智能体可以执行、检查和操作这些结构。这些方法不是仅通过潜在嵌入或文本描述来表示环境，而是将世界状态、对象关系、空间布局和交互动态编码为结构化的计算制品。例如，ViStruct[125]使用编程语言结构作为视觉结构知识提取的显式接口，使多粒度视觉事件能够通过一致的可执行结构来表示。FactoredScenes[126]类似地将室内环境建模为组合的"房间程序"，其中可重用对象和关系函数定义了物理上一致的场景布局。将这一思想扩展到可扩展的符号世界建模，PoE-World[127]引入了一个组合框架，结合许多小型程序化专家来表示日益复杂的环境动态。更近期的系统将结构化环境接口扩展到高保真交互世界。Code2World[38]将GUI状态预测重新框定为可渲染的HTML生成，使环境转换可以通过可执行的渲染代码来表示和评估。Code2Worlds[139]进一步将这一范式扩展到4D模拟环境，通过语言到模拟的程序生成，其中物理感知的执行循环在环境构建和交互过程中减少了语义-物理不一致性。

### 2.3.2 执行轨迹世界建模

执行轨迹世界建模研究智能体如何直接从可执行交互轨迹中学习环境动态。这些方法不将执行仅视为最终评估步骤，而是将运行时转换本身建模为环境行为的主要表示。SemCoder[128]通过训练语言模型推理功能行为、语句级执行效果和输入输出转换，弥合了静态程序和运行时语义之间的鸿沟。在此基础上，代码世界模型（CWM）[37]直接从程序轨迹中学习预测性世界模型，使智能体能够通过可执行动态来预测未来环境状态。WorldCoder[36]进一步引入了一个基于模型的交互框架，智能体在其中显式编写和更新表示为Python程序的可执行世界模型。智能体不是仅将环境知识隐式存储在模型参数中，而是维护可编辑的计算表示，可以在规划和交互过程中执行、修改和重用。后续工作将这一范式扩展到持续和交互式的世界模型适应。RWML[129]结合执行轨迹与强化学习，通过运行时交互来优化环境动态，而AWM[130]和WorldMind[131]研究如何跨任务和知识源对齐、融合和协调多个可执行世界模型。

### 2.3.3 代码接地评估环境

代码接地评估环境使用可执行系统作为衡量智能体行为和交互质量的接口。与仅基于文本输出的静态基准不同，这些环境暴露显式的运行时状态转换、执行反馈和可验证的交互结果，智能体可以直接观察和评估。InterCode[11]通过将编程任务重新框定为交互式执行环境确立了这一范式，其中代码作为行动，执行反馈作为观察，沙箱运行时提供接地交互。CRUXEval[132]进一步通过可执行的输入输出预测任务评估程序理解，而LiveCodeBench[133]引入持续更新的评估管道，评估在不断演化的问题分布下的执行、自修复和运行时推理能力。SWE-bench[5]将可执行评估扩展到现实世界的软件仓库，智能体必须修改大规模代码库，并通过仓库级单元测试执行而非仅文本正确性来评估。更广泛地说，AgentBench[12]证明了可执行交互环境可以评估跨多种具身和数字任务的推理和决策能力。后续的基准如CRUXEval-X[134]、CoRe[135]、GeoGramBench[140]、CodeGlance[136]和Endless Terminals[39]进一步将这一范式扩展到多语言、多模态和持续交互的评估设置，其中运行时交互而非静态答案匹配成为主要的评估接口。

### 2.3.4 可验证环境构建

一个较新的方向将可执行环境不仅视为评估智能体的基准，还视为可以通过编程方式合成、扩展和验证的支架制品。这对于长周期智能体尤为重要，支架不仅需要提供任务提示，还需要提供可运行的状态、转换动态、反馈通道和验证预言机。SWE-smith[137]通过从现有代码库构建仓库级任务和执行环境来扩展软件工程智能体数据，将软件仓库转化为用于智能体训练和评估的可复现程序世界。EnvScaler[138]将这一思想扩展到软件工程之外，通过编程方式合成工具交互环境以及场景和基于规则的轨迹验证器。从支架的角度看，这些方法使环境接口本身成为构建的对象：代码不仅指定智能体编辑或执行的内容，还指定决定交互是否成功的状态转换、工具可供性和验证器。

# 3. 支架机制：规划、内存、工具使用、控制和优化

支架机制构成中心系统层，使代码支架化的智能体在单次生成步骤之外保持可靠。一旦代码进入智能体循环，软件生成不再仅仅是根据提示产生正确程序的问题。它变成模型、可变任务状态和人类设计的支架基础设施之间的交互。模型提供判断：它分解目标、选择行动、解释反馈并决定何时修改。可变状态记录仓库证据、工作上下文、执行轨迹、验证结果、记忆和关于任务的中介信念。支架基础设施暴露工具和执行基板、持久化和压缩状态、通过策略和权限层级约束行动、路由反馈，并验证每次状态转换是否可接受。从这个角度看，支架机制不是孤立的附加模块，而是协调的控制面板，将模型决策转化为可执行环境中受约束、可观察和可修改的变化。在其基本形式中，代码允许智能体调用现有可执行接口。进一步地，智能体可以动态编写任务特定的可执行接口。这些智能体创作的制品使支架更具适应性，因为它们允许围绕当前任务重塑执行环境。然而，动态创作的代码并不取代更广泛的人类设计支架基础设施。可靠性仍然依赖于模型侧的判断以及人类设计的策略、沙箱边界、权限层级、验证预言机、审计日志和人工审查关卡。因此，代码作为支架内的可执行媒介，而支架仍然是更大的策略治理系统，决定哪些代码可以被执行、信任、持久化、重用或提升到未来的工作流中。

在本节中，我们回顾代码智能体的五个交互类别的支架机制。**规划（§3.1）**通过将目标外部化为分解、结构约束、搜索轨迹或工作流级编排来组织长周期任务执行。**内存与上下文工程（§3.2）**通过在长交互中保留工作上下文、检索仓库证据、存储可复用体验、支持共享历史和将状态卸载到活动上下文窗口之外来管理可变状态。**工具使用（§3.3）**将智能体连接到受治理的可执行接口，包括API、仓库、终端、沙箱、验证工具和工作流编排器。**通过规划-执行-验证循环的支架控制（§3.4）**将反馈引导的调试重新框定为更广泛的控制过程：计划形成关于意图变更的契约，执行在沙箱化和权限化的环境中应用变更，验证使用确定性传感器和人工审查关卡来决定状态是否应被接受、修改、升级或回滚。最后，**智能体支架工程（§3.5）**研究支架本身如何通过深度遥测、演化智能体、基于回放的评估和受治理的支架变异来衡量和改进。

## 3.1 智能体支架的规划

规划在智能体支架中扮演核心角色，因为现实世界的软件工程任务很少允许从自然语言意图到正确实现的直接一次性映射。从支架的角度看，规划不仅仅是LLM的内部推理能力，而是一种**支架控制**形式：它结构化了智能体如何将意图外部化到可执行步骤中、调度与代码制品和工具的交互，并规范推理、执行和修改随时间的轨迹。超越生成代码标记，有效的智能体支架必须将长周期问题解决组织为连贯的行动路线，决定追求哪些中间目标、按什么顺序执行、检查或修改哪些制品，以及当执行反馈揭示错误、缺失依赖或违反约束时如何修改轨迹。这种需求在仓库级编辑、Web交互、竞赛编程和硬件设计中尤为突出，智能体必须在大的行动空间、稀疏反馈和深度互依的子问题上操作。在这样的设置中，目标任务的复杂性和不受约束的智能体执行的有限可靠性之间出现了一个基本挑战：没有显式的规划机制作为支架控制，智能体可能过早地投入到脆弱的解决方案路径中，忽视潜在的依赖关系，或无法将推理、检索、执行和修改协调为稳定的工作流。

早期面向规划的系统主要将规划视为线性分解步骤，模型首先产生自然语言的解决方案大纲，然后将其翻译为代码。然而，随着代码智能体应用于更复杂的环境，规划逐渐从简单的预生成支架演变为更丰富的支架级控制机制。它可以基于仓库结构或外部知识来约束智能体的行动空间，通过跨多个候选轨迹的显式搜索来扩展以提升鲁棒性，或分布在专门的智能体角色和反馈循环之间以在系统层面协调执行。基于支架控制实现的主要场所，我们将代码智能体中的现有规划方法分为四类：**线性分解规划**、**结构接地规划**、**基于搜索的规划**和**基于编排的规划**。

### 3.1.1 线性分解规划

在这种规划范式中，智能体首先产生一个单一的显式、可执行的步骤序列，然后按照这个分解来执行生成[141,40,41,142,143]。此模式的一个轻量级前身是ReAct[144]，智能体在其中交错思维、行动和观察，形成串行轨迹。在这个框架中，每个推理步骤外部化当前的子目标并约束下一个行动，将轨迹本身转变为逐步的支架控制。这一模式在Self-Planning[40]中得到了最直接的实例化：模型首先将意图分解为简洁、高层次的编号步骤，然后在此计划的指导下逐步生成代码。Plan-And-Act[145]通过分离一个产生结构化高层计划的规划器，进一步使这一支架显式化：规划器在新观察到达时反复刷新线性支架，使规划策略能够在适应环境反馈的同时保持任务级控制。WebAgent[41]将这一思想扩展到Web自动化：它将用户指令分解为连续的子指令，根据当前子目标总结任务相关的HTML，然后从该线性子指令序列中合成可执行的Python动作。KareCoder[141]在知识增强的设置中遵循类似的模板，模型首先从外部知识库构建知识感知的逐步提示，然后使用此提示生成代码，使规划成为问题理解和实现之间的结构化中间层。近期的工业实践表明，这种线性支架可以从短暂的提示制品提升为持久的支架对象。在长周期编码工作流中，诸如PLAN.md、Implement.md和状态日志等文件记录里程碑、接受标准、验证命令和恢复规则，使智能体能够在上下文重置或多会话执行中重新加载、更新、验证和记录进展[146,147]。在这个视角下，规划不再仅仅是内部的推理轨迹，而是文件系统支持的控制对象：它可以被人类审查、用Git进行版本控制、被子智能体消费，并用作用于实现的真相源。主要的局限性在于这些方法通常承诺单一的分解轨迹：当初始计划不完整或不对齐时，支架可以提高持久性和可审计性，但仍然提供超出所选路径的有限探索。

### 3.1.2 结构接地规划

在这一工作线中，智能体不单从自由形式的自然语言提示中推导出其行动序列，而是将规划建立在任务环境的显式结构化表示上，如依赖图、仓库图、电路图或知识图。这些结构充当天然的支架脚手架：它们暴露相关实体、编码依赖关系，并指导子任务应被生成、修改或验证的顺序。例如，CodePlan[42]在编辑义务上构建计划图，并通过依赖分析和变更影响传播推导出新步骤。同时，仓库理解方法[148,149,150,148]将代码库转换为异构图或文本丰富的代码图，然后使用图集成推理来定位相关实体，并基于结构依赖而非平面文本上下文来约束下游生成。GraphCodeAgent[151]通过双图支架扩展了这一思想，其中需求图捕获自然语言需求之间的关系，结构语义代码图捕获仓库依赖关系。同样的原则也出现在近期智能体本地的仓库实践中。诸如架构说明、API规范、测试指南等文件将项目知识转化为持久的、可检查和版本控制的制品，智能体可以在行动前查阅[152,153,154]。这拓宽了结构接地规划超越图构建：相关结构决定显式规则、构建命令、目录边界、编码约定和设计约束，从而促进对程序的连贯和稳定的支架控制。专门领域遵循相同的模式[155,156]。VerilogCoder[156]将子任务规划建立在任务和电路关系图上，使每个子任务都附有信号、转换和示例，而DomAgent[155]使用知识图谱结合自上而下的结构化知识与自下而上的示例，用于领域特定的代码生成。总体而言，这些工作表明结构接地规划通过将项目或领域知识转变为显式和可检查的支架对象来指导智能体随时间的行动，从而改善连贯性、依赖感知和长周期一致性。

### 3.1.3 基于搜索的规划

基于搜索的规划分配推理时间计算来系统地探索、评估和选择多个候选解决方案路径。关键思想不是将智能体承诺到单一计划，而是扩展决策空间并使用反馈来控制哪些替代方案应被追求、修改或丢弃。第一组方法[157,158]在思维空间中实例化这一支架。它们不是直接编写代码，而是首先在高层观察、策略或推理轨迹上进行分支，目的是在实现之前增加概念多样性。在这一视角下，更好的规划来自于覆盖更广泛的想法空间并使用反馈来优化推理本身，而不仅仅是修复最终代码。第二组方法[43,159,160,161]在编码动作的**轨迹空间**中进行搜索：这些方法将编码建模为在策略选择、实现、调试和修改上的分支过程，并依赖执行信号或学习的评判者来决定扩展哪些节点。因此，当智能体可以从次优决策回溯并比较部分轨迹时，长周期编码质量会提高。另一条工作线，如ReLoc[162]和SFS[163]，将规划视为**代码空间**中的搜索。这些方法通过变异、修改或局部优化，在验证反馈或细粒度评分信号的引导下迭代地探索相邻程序。超越上述方法，近期系统日益将候选计划、补丁、日志、测试和执行轨迹视为持久的制品而非短暂的生成。SWE-Search[164]将蒙特卡洛树搜索与软件工程智能体相结合，探索替代修复轨迹，而CodeTree[43]在统一的树内组织策略探索、解决方案生成和优化。更广泛地说，Meta-Harness[13]将这一思想推到了支架级别本身：它通过给智能体访问先前的源代码、分数和执行轨迹（通过文件系统），在支架代码上进行搜索。这些发展表明基于搜索的规划不仅是模型侧的采样策略，也是支架级状态管理问题：运行时必须保留候选方案、暴露证据、运行验证器，并决定哪个分支值得进一步计算。

### 3.1.4 基于编排的规划

基于编排的规划指的是一种规划范式，其中核心规划功能通过系统级协调的支架设计来实现。在这种范式中，支架管理智能体或模块如何专门化角色、执行阶段、路由反馈和触发验证循环，从而决定在长周期代码生成工作流中接下来应采取什么行动。第一种常见模式[50,51,52]是以反馈为中心的编排，系统将编码、测试、分析和修复分布在不同的模块上，使得进展由重复的基于执行的反馈和自适应升级驱动。在这一组中，规划不是预先的制品，而是失败如何被检测、解释和路由回后续行动的一个新兴属性。第二种模式[44,166,165]是分阶段工作流编排，它将代码生成视为结构化的软件过程流水线，如理解、检索或预览、规划或蓝图设计、编码、调试和修复。这一组的主要优势在于将复杂生成分解为具有显式交接规则的可解释阶段，而实际的规划能力来自于跨阶段控制、候选剪枝和迭代优化。第三种模式[167,168,169,170]是以控制器为中心的编排，其中规划嵌入在中间制品的转换和路由基板本身中。在这里，系统通过形式规范流水线、定位和修复之间的建议阶段、类型化中间表示、共享黑板或专门的规划器-编码器协调等机制来组织决策制定，使得下一个计划由脚手架的控制器逻辑而非单个文本提示决定。

近期的支架系统使这一编排视角尤为显式。Anthropic的长周期支架将规划、生成和评估分离为不同的角色，使用结构化制品和独立评估来在长时间会话中保持进展[15,171]。Cursor的大规模自主编码实验类似地强调了规划器-工作者协调，作为从专注的单智能体任务扩展到共享项目上的许多并行智能体的一种方式[172]。最一般的表述出现在自然语言智能体支架中，其中高层支架逻辑（如角色、阶段、契约、适配器、状态约定和失败分类）被编写为可编辑的自然语言，并由智能支架运行时（IHR）执行[173]。IHR在运行时解释这些高层自然语言指令，并将其转换为在显式契约、预算、工具接口和环境状态下的受约束执行步骤。这将基于编排的规划重新框定为运行时解释问题：计划不仅是文档，而是可执行的支架规范，在模型输出、文件系统状态、工具、验证器和多智能体委派之间进行中介。

**讨论：** 用于代码生成的规划可以理解为智能体支架的核心形式：一个控制层，组织LLM智能体如何分解任务、将决策建立在程序结构上、在推理时探索替代方案以及协调多阶段软件工程工作流。从这个角度看，规划是一组支架机制，围绕一个基本问题：如何决定智能体下一步应做什么，以及如何在长周期编码任务中保持该决策过程受约束、可检查和连贯。值得注意的是，代码生成中的规划不能与评估问题干净地分离。目前许多关于规划优势的结论严重依赖于周围的执行条件，包括执行环境、反馈质量、工具访问、轨迹预算以及基准是否真正考验长范围依赖管理而非局部补丁生成。如果执行信号弱、修改预算不现实或基准未能暴露多步协调错误，那么报告的规划增益可能不反映智能级问题解决的真正改进。因此，规划不仅是方法设计问题，也是智能体和环境之间的支架问题。展望未来，核心挑战不仅仅是构建更大的规划器或更长的推理轨迹，而是为规划设计更可靠的智能体支架：决定何时遵循、修改或放弃计划的自适应承诺机制；暴露依赖关系和进展的结构上有意义的规划状态；使用反馈而不过度计算的高效探索和修正策略；以及能够可靠衡量规划质量超越最终通过准确性的严谨长周期评估范式。

## 3.2 智能体支架的内存与上下文工程

内存已成为代码智能体的核心基础设施，很大程度上是因为现实世界的软件工程任务本质上是长周期和状态密集型的[174,175]。与单轮代码补全不同，实际编码场景要求智能体跨多轮交互维持一系列相互依赖的步骤，如需求理解、代码定位、证据检索、多文件编辑、测试执行、错误修复和回归验证[176,177]。这引入了模型有限上下文窗口和任务持续扩展的中间状态之间的基本矛盾。从支架的角度看，内存不仅仅是更大的上下文窗口或向量数据库。它是一个状态管理层，决定哪些信息应保留在活动模型上下文中，哪些信息应被压缩为摘要，以及哪些信息应被卸载到持久的外部存储[178]。没有有效的内存机制和上下文管理，智能体在长距离推理中容易丢失关键线索，重复已经完成的搜索和分析，或在后期修改中破坏早期建立的一致性[179,175]。

早期系统主要依赖提示来保留历史信息，将内存视为无非是对话历史或无结构的草稿本。然而，随着仓库级修复和其他长周期编码任务的出现，简单地积累自然语言历史已无法可靠地支持复杂的软件工程循环[180]。因此，内存现在越来越多地外部化为可检索、可治理和可追溯的系统组件。在本小节中，我们根据内存**在软件工程循环中的主要功能角色**对代码智能体中的内存进行分类。在这一视角下，现有方法大致可以分为五种类型：**工作内存**、**语义内存**、**经验内存**、**长期内存**和**多智能体内存**。此外，我们讨论**上下文压缩和状态卸载**作为跨领域的上下文工程机制，决定大型执行制品如何在活动模型上下文和持久任务状态之间移动。

### 3.2.1 工作内存

工作内存支持沿当前编码任务轨迹的状态维护[181]。其核心关注点不是保留多少历史，而是在有限的上下文预算下哪些信息对下一个行动最为有用。在代码智能体中，工作内存通常表现为结构化提示区域、状态摘要、失败测试记录、文件列表或关键栈信息。其目的是缓解上下文爆炸、减少重复定位，并保持正在进行的修复或编辑轨迹的局部一致性[57,182,183,45]。从支架的角度看，工作内存是模型和代码环境之间的活动控制面板：它决定智能体在选择下一个工具调用、编辑或验证步骤之前观察什么。代表性系统如SWE-agent[57]和RepairAgent[183]表明，即使使用相同的基础模型，仓库级修复性能也可能因交互状态和执行反馈的组织方式而有显著差异。CodeMem[45]类似地将上下文视为受管理的资源，使用有预算的内存槽来稳定多步编辑。

### 3.2.2 语义内存

语义内存为当前的编码过程提供任务相关的外部证据[184,175]。在代码智能体设置中，这种证据通常是仓库特定的和程序结构化的，包括类定义、函数实现、调用关系、配置文件、文档、问题描述、依赖元数据和历史实现模式。语义内存因此将外部代码库转换为可查询的证据空间，支架可以从中检索并注入活动上下文中[46,185,186,187,188]。AutoCodeRover[46]和RepoCoder[47]等代表性工作表明，仓库级编码任务不仅受益于检索更多内容，更受益于检索与程序结构对齐的证据。基于AST的结构化分块、迭代查询重写和以当前定位线索为条件的检索策略等机制可以显著提高检索上下文对下游生成的实用性。在这个意义上，语义内存将代码库转化为当前决策过程的结构化证据层。

### 3.2.3 经验内存

随着代码智能体从单任务完成走向持续修复和跨项目泛化，越来越多的关注投向经验或情景记忆[197,198]。与维护当前轨迹的工作内存或检索仓库证据的语义内存不同，经验内存捕获跨任务积累的可重用体验，如修复轨迹、失败案例、调试记录和更高层次的策略模式[189,199,200]。其主要价值在于实现跨任务迁移。通过经验卡、反思缓冲区和记录-重放流水线等机制，系统可以将过去成功或失败的调试过程转化为未来问题解决的可重用单元[199,48,201]。MemGovern[48]等工作进一步表明，存储的经验质量比规模更重要。无治理的历史记录可能引入语义噪声、错误传播和错误检索，而经过策划和质量控制的经验内存更可能成为仓库级修复的有用资产。

### 3.2.4 长期内存

当编码轨迹变长时，仅靠工作内存和语义内存是不够的，因为系统还必须应对内存增长、压缩引起的证据失真和长期漂移。这使得长期检索规划和内存控制成为一个日益重要的研究方向[202,203,204,205,206]。关注点因此从内存容量转向内存治理。代表性系统如MemGPT[207]和MemoryOS[208]将讨论从存储什么转向何时写入、何时压缩、何时检索以及如何避免污染。近期的以代码为中心的研究进一步将这一工作线建立在软件工程工作流中。MemCoder[190]利用结构化的历史提交和人类验证的解决方案作为持久内存，实现了随时间推移的仓库特定经验积累。TALM[191]将长期内存纳入多智能体代码生成，检索先前的问题-解决方案轨迹并合并重叠的记忆以控制冗余。这些工作表明，对于代码智能体，长期内存不应简单积累更多历史，而应以紧凑可控的形式保留经过验证和可重用的经验。否则，内存可能从长周期软件工程的资源转变为放大噪声、陈旧和错误的负担。

### 3.2.5 多智能体内存

多智能体内存将状态管理从单个智能体扩展到共享支架。从系统角度看，代码生成中的内存具有很强的协作维度[209,210]。在多智能体框架中，内存不仅是个体状态的容器，还是跨专门化角色的信息共享、意图传递和一致性维护的媒介[211]。AgentCoder[50]、MapCoder[44]、MIRIX[192]、ChatDev[193]和G-Memory[211]等代表性工作说明了内存如何支持多智能体规划、测试、审查和轨迹协调。在这种设置中，核心挑战不再是检索相关内容，而是控制共享的粒度、防止信息泛滥，以及支持高层决策和细粒度执行轨迹之间的双向访问[210]。因此，多智能体代码生成中的内存日益类似于共享黑板或协作状态图，而非纯粹的个人存储单元[212,213]。

### 3.2.6 上下文压缩与状态卸载

上下文压缩和状态卸载是代码智能体支架中内存的跨领域上下文工程机制[214]。其目标不是定义另一种内存类别，而是控制活动模型上下文和持久任务状态之间的边界。长周期软件工程工作流持续产生高容量的制品，如构建日志、执行轨迹、仓库差异、测试输出和中间计划。直接将这些制品放入提示中会迅速使上下文窗口过载、放大噪声并模糊决策相关的证据。支架因此必须决定哪些观察应保留在活动上下文中，哪些应压缩为简洁摘要，哪些应卸载到带有可检索句柄的外部存储中[178]。上下文压缩将长的交互历史和庞大的工具输出压缩为结构化的、保留来源的摘要。例如，一个失败的测试报告可以被简化为失败的测试名称、关键栈帧、怀疑的文件和完整日志的链接[196,215,194,195]。状态卸载通过将全保真制品保留在活动窗口之外来补充这一过程，如在文件、数据库、轨迹存储或协议式资源接口（如MCP风格的服务器）中。智能体随后收到紧凑摘要和资源标识符，而不是原始日志或轨迹。通过将决策相关上下文与持久证据分离，上下文压缩和状态卸载使内存更具可扩展性、可审计性，并与执行时验证兼容。

**讨论：** 代码即智能体支架系统中的内存可以理解为统一的状态管理层，连接上下文管理、仓库证据检索、经验迁移、长期控制和多智能体同步。它既不是单一的数据结构、放大的上下文窗口，也不是简单的向量数据库，而是协调任务相关状态应驻留何处以及如何在长周期软件工程工作流中重用。工作内存保持下一个行动的接地；语义内存暴露仓库证据；经验内存支持跨任务迁移；长期内存保留经过验证的知识；多智能体内存在角色间同步共享状态。上下文压缩和状态卸载通过将决策相关的活动上下文与持久的全保真制品分离，进一步扩展了这一层，使内存更具可扩展性、可审计性，并与执行时验证兼容。重要的是，代码智能体中的内存研究不能脱离**评估可靠性**。许多关于内存优势的结论依赖于评估管道的质量[216,217]：如果测试不足、日志解析有缺陷或基准存在记忆化和污染问题，那么报告的改进可能不反映鲁棒的长期行为。展望未来，关键挑战不仅仅是扩大内存容量，而是构建更高质量的写入关卡、结构对齐的检索键、保留来源的压缩机制、可靠的状态卸载协议，以及能够衡量内存是否真正帮助智能体在扩展轨迹中保持接地、一致和可验证的严谨评估范式。

## 3.3 智能体支架的工具使用

工具使用是代码智能体支架的行动和观察层。一旦代码被放入智能体循环，模型不仅要生成文本，还要搜索仓库、编辑代码、执行测试、调用API、查询文档和验证中间结果[218,219]。工具因此扩展了智能体的行动空间，同时也暴露了使支架可执行和可检查的外部反馈信号。从代码即智能体支架的角度看，工具使用不仅仅是代码生成的辅助能力。它是模型意图和外部系统之间的受治理接口。一个可靠的支架必须决定哪些工具可用、它们的模式如何暴露、每个工具获得什么权限、执行在哪里发生、结果如何被净化或压缩，以及何时危险行动需要人类批准。近期的智能体SDK和软件智能体平台通过将工具、会话、护栏、交接、工作区和执行环境打包为可重用的支架组件，使这一转变显式化[58,220,221]。同时，沙箱化执行环境（包括容器化或微VM化工作区）将智能体行动与主机系统隔离，使代码执行更具可复现性和可审计性[22,222,223]。

这种支架级视角也突出了**工具生命周期控制**的重要性。工具执行前，支架可能应用权限检查、策略规则、参数验证或人在环关卡。执行后，支架可能净化输出、总结大型日志、将轨迹卸载到持久存储、更新内存或触发验证工具。生命周期钩子使这些控制点显式化。它们将工具使用从模型选择的原始行动转变为智能体执行循环中受监控的转换。

现有关于代码智能体工具使用的工作因此可以根据工具服务的主要支架功能来组织：（1）**面向功能的工具使用**，（2）**环境交互工具使用**，（3）**验证驱动的工具使用**，和（4）**工作流编排工具使用**。面向功能的工具将智能体接地到API、库和外部文档。环境交互工具允许智能体在仓库、终端、IDE、浏览器和沙箱中行动。验证驱动的工具通过测试、linter、类型检查器、静态分析器和运行时错误提供确定性反馈。工作流编排工具将多个工具、角色、内存更新和生命周期策略协调为可靠的长周期执行过程。

### 3.3.1 面向功能的工具使用

这一工作线主要使用工具来填补模型编程知识中的空白，特别是API、库、文档和外部编码工具[19,224,225,227,228,229]。例如，ToolCoder[19]从一个明确的瓶颈出发：代码模型经常幻觉API、选择不适当的函数，或在训练覆盖稀疏的公共和私有库上失败。为解决这个问题，它将API搜索工具集成到代码生成过程中，并训练模型决定何时查询工具以及如何从检索结果中选择API。关键贡献因此不仅仅是更好的语法生成，而是更好的知识获取和API接地。更广泛地说，检索导向的方法减少了对参数化内存的依赖，使代码生成更能适应长尾API、私有库和不断演化的软件生态系统[225,230]。当主要瓶颈是模型缺乏关于应使用哪个函数、API或库构造的可靠知识时，这些方法最为有效。相应地，核心设计挑战在于查询表述、结果选择、证据压缩以及将检索知识稳健地注入下游生成中。这些智能体方法特别适用于面向API的生成、库迁移和私有SDK使用，但仅靠检索在需要跨文件理解和推理、运行时调试或仓库级依赖分析的任务中往往不足。

### 3.3.2 环境交互工具使用

与面向功能的工具不同，环境交互方法将工具视为智能体在软件工程环境中行动的接口[231,232,233,234]。其核心问题不再是获取缺失的函数，而是在仓库、开发制品和执行环境中有效操作。CodeAgent[185]表明，现实世界仓库级代码生成不仅仅是根据提示完成单个函数。模型必须定位相关文件、理解依赖关系、检查文档、实现修改并通过测试验证结果。为支持这一过程，CodeAgent集成了编程工具和智能体策略，用于在真实仓库中进行信息检索、代码符号导航、代码实现和测试交互。SWE-agent[57]通过形式化智能体-计算机接口进一步推动了这一思想，其中shell命令、文件编辑和测试执行成为主要的交互通道。RepairAgent[183]类似地为智能体配备了专门用于读取代码、搜索修复材料、应用补丁和运行测试的修复工具。这些方法共同定义了环境交互工具使用的核心轨迹，这对于仓库级生成、问题解决和开放式软件工程任务尤为重要。

### 3.3.3 验证驱动的工具使用

第三条工作线主要使用工具进行后生成验证和迭代改进。验证驱动的工具使用将外部工具视为支架的确定性传感器。与面向功能和环境交互的工具相比，这些方法不一定强调外部检索或广泛的仓库导航。相反，它们使用测试、执行结果、编译器错误、运行时轨迹、类型检查器、静态分析器和验证器反馈作为提高代码质量的主要信号[226,235,236,237]。例如，AgentCoder[50]使用程序员智能体、测试设计智能体和测试执行智能体形成代码生成、测试构建、执行和优化的闭环。在这个范式中，工具的核心作用是验证而非检索。从代码即智能体支架的角度看，验证工具使智能体的进展变得可检查：测试失败、栈轨迹、覆盖缺口、类型错误和静态分析警告成为更新工作内存和指导下一个行动的结构化观察。关键设计问题是如何将这些观察路由回循环中[226]。由于原始日志对于活动上下文可能过长或过于噪声，支架应解析、总结和卸载验证轨迹，同时保留全保真制品用于审计和回放。

### 3.3.4 工作流编排工具使用

工作流编排工具使用关注多个工具、角色和控制策略如何被组织成一个连贯的智能体工作流[238,239,240,241]。在长周期软件任务中，智能体可能需要检索证据、定位错误、修改文件、运行测试、检查失败、更新内存、请求批准并重复此循环多次。挑战不仅仅是添加更多工具，而是决定何时每个工具应被调用、以什么权限、在什么上下文中、以及其结果应如何更新支架状态[49]。近期的智能体SDK和软件智能体平台通过将类型化工具模式、会话状态、工作区、护栏、交接、追踪和人工审查机制打包为可重用的支架组件，使这一编排层显式化。生命周期钩子进一步细化这一边界：使用前钩子可以验证参数、强制权限策略或阻止危险命令，而使用后钩子可以净化输出、压缩日志、更新内存或触发后续验证。MapCoder[44]等代表性系统通过将智能体分配给示例回忆、规划、代码生成和调试来说明工作流编排，从而将困难的编码问题分解为协调的子问题。CodeAgent[185]也研究了在仓库级工作流中工具调用应如何调度和结构化。这一类对于长周期代码智能体尤为重要，因为实际软件任务需要在显式控制策略下进行需求分解、上下文选择、候选探索、基于执行的验证和最终修复[49,242]。

**讨论：** 代码智能体中的工具使用已从孤立的API检索演变为用于行动、观察、验证和治理的完整支架机制。面向功能的工具将实现选择建立在外部知识之上；环境交互工具允许智能体在仓库和执行环境中行动；验证驱动的工具提供确定性反馈；工作流编排工具通过SDK、沙箱、护栏和生命周期钩子协调这些能力。核心挑战不再是模型能否调用工具，而是支架能否使工具使用安全、可审计且对长周期执行有用。未来的代码智能体支架应支持类型化工具模式、权限感知调用、沙箱化执行、生命周期钩子、结果净化、上下文压缩、状态卸载和可复现轨迹。这些机制确保工具在扩展智能体行动空间的同时不牺牲可靠性、安全性或可验证性。

## 3.4 通过规划-执行-验证循环的支架控制

代码即支架系统需要一个控制循环，将模型意图转变为受约束、可观察和可修改的状态转换。本小节将该循环框定为**规划-执行-验证（PEV）**：支架首先将意图变更及其验证标准外部化，然后在沙箱化和权限化的环境中执行变更，最后通过确定性传感器和人工审查关卡验证结果状态。这一框架将规划、执行、调试、验证和升级统一为单一支架级控制过程的各个部分。

### 3.4.1 从调试到支架级控制

前面的小节将规划描述为轨迹控制、内存为状态管理、工具使用为受治理的行动接口。反馈引导的调试将这些机制连接为闭环：计划指定意图变更，内存保留相关证据，工具执行和检查行动，验证信号决定智能体是否应继续、修改或停止。随着以代码为中心的智能体从单次生成转向仓库级软件工作，调试因此更应理解为对可执行程序状态的控制，而非事后修正阶段。生成的程序可能因语法错误、运行时异常、错误输出、不完整的边缘情况处理、不安全操作或违反项目特定约定而失败，使得单次生成不足[243]。近期系统通过编译器、运行时、测试、静态分析器、人类和辅助智能体的反馈来修改代码[244,245,246,23]。从支架的角度看，这个过程可以被重新框定为规划-执行-验证（PEV）循环：智能体外化意图轨迹，在受控环境中执行受约束的行动，并在下次转换前验证结果状态。围绕智能体支架不断增长的工程生态系统强化了这一观点：近期策划的资源将编排、工作状态、执行基板、评估支架、可观察性和治理区分为可分离的支架层，而非偶然的实现细节[247,248,249,25]。

在这一视角下，支架充当**控制论调节器**：一个控制层，观察智能体行动的效果并调节后续状态转换。它不仅仅将错误消息转发给模型，而是通过确定性传感器如linter、解析器、编译器、类型检查器、单元测试、集成测试、静态分析器、模糊测试器、运行时监控器和CI流水线来观察仓库和执行环境。这些传感器将编码轨迹转变为可检查的信号，包括通过/失败结果、诊断信息、失败轨迹、覆盖缺口、安全警告、资源限制和策略违规。支架随后可以决定是否继续执行、修改补丁、请求更多上下文、将任务路由到另一个模块、减少权限或升级到人工审查员。表7总结了这一控制面；本小节的其余部分遵循从契约形成，经过沙箱化状态转换，到确定性验证和基于证据的修复的循环。

### 3.4.2 规划作为契约形成

规划阶段将用户请求转化为关于下一个状态转换的显式契约。一个稳健的计划不仅仅是分解请求为实现步骤；它还识别相关文件、预期不变量、验证命令、回滚点和风险操作。这使得规划成为支架制品而非不被观察的推理轨迹。在仓库级任务中，这样的制品通过指定哪些组件可以被读取、哪些文件可以编辑以及完成前必须满足哪些验证标准来约束后续行动空间[40,42,44]。仓库本地指令和工具协议加强了这一契约层：AGENTS.md风格的指导、MCP服务器注册、类型化工具模式、适配器和协议网关使可用行动在执行前可检查，而非在执行过程中偶然发现[255,256,257,258,259,260,261,262]。PEV框架也澄清了为什么规划和调试不应分离：失败验证更新计划，而计划决定哪些验证证据有意义。

### 3.4.3 沙箱化执行与权限化状态转换

执行阶段将计划实现为受约束和可观察的状态转换。沙箱环境是循环的操作基板：它提供隔离的文件系统、依赖状态、shell、语言运行时、浏览器或IDE接口以及资源边界，智能体生成的操作可以在其中运行而不会直接危害主机系统[263,22]。当代执行基板工作最好被理解为功能集群而非无差别的目录。编码沙箱暴露文件系统、Git操作、shell、包管理器和代码执行后端[251,252,264,265,266,250]；计算机使用基板添加浏览器、桌面、LSP或IDE状态[267,268,269,270,271]；持久运行时强调微VM或WASM隔离、快照、温池、可恢复会话、基准环境和始终在线的操作上下文[272,273,274,275,276,277,278]。沙箱也提高了可复现性，因为支架可以在可比较条件下回放相同的补丁、命令、种子、依赖锁文件或测试配置。没有这种稳定的基板，验证信号难以解释，失败可能反映环境漂移而非程序缺陷[250,217,279]。

执行也必须是权限化的。一个多层模型将低风险观察与高风险行动分开：只读层支持仓库浏览、检索、静态检查和日志分析；沙箱编辑层支持在隔离工作区内进行本地修补、测试执行和临时依赖安装；完全访问层涵盖网络访问、凭据、部署命令、包发布、破坏性文件系统操作或Git历史变异。最后一层的行动应由强制性的**人在环（HITL）**关卡保护，因为其后果可能超出沙箱。近期的软件智能体系统和支架工程工作日益通过显式的工具、会话、策略、批准提示和审计日志来暴露这些控制点[280,250,281,178,282,283]。网关和策略层随后提供生产对应物：用于模型路由、工具注册、代理级日志记录、集中护栏、安全自动化和可证伪的批准证据的系统，使治理超出提示本身[254,284,285,262,286,287,288,289,290,291]。

### 3.4.4 通过确定性传感器的验证

验证阶段通过将新状态与显式约束进行比较来闭环，并在必要时重新打开循环。编译和静态分析反馈在全执行之前提供低成本传感器，包括解析器诊断、类型错误、lint警告和安全警报[246,292,293]。运行时信号暴露仅沿具体执行路径出现的失败，如异常、断言失败、无效API使用、资源耗尽和超时[294,295,245]。基于测试的反馈随后评估观察到的行为是否满足预期规范，使用单元测试、集成测试、回归测试、模糊测试或基准特定评估器[243,296,297,298]。评估支架将这一思想从单个测试命令扩展到可重复的任务分布：它们编码评估器逻辑、模拟钩子、红队案例或RL风格环境，可以在受控条件下比较支架变体[299,300,301,302,303,304,305,306,307,308,309]。与自然语言批判相比，这些传感器是确定性的或至少足够可复现以作为控制信号。当失败证据稀疏时，人类或智能体批判仍然有用，但在受治理的PEV循环中，它们应解释传感器输出而非取代它们[244,310,54]。

验证也为修复、反思和终止提供证据，因此这些活动被视为验证阶段的后果而非独立阶段。当检查失败时，相同的传感器证据可以确定支架是否应要求模型诊断失败、检索缺失上下文、重新生成局部补丁、将任务路由到测试或安全智能体、或放弃当前分支。自反思机制帮助将原始诊断转化为可操作的假设，如失败是否来自不正确的控制流、缺失的边缘情况、误解的API或不充分的测试[311,166]。然而，反思仅在基于可执行证据时才是可靠的。AgentCoder、AutoSafeCoder和QualityFlow等系统通过将智能体批判与独立执行、静态分析、模糊测试或测试质量关卡相结合来说明这一原则[50,52,253]。终止同样应由验证而非模型置信度来治理：循环可以在所需检查通过时、当额外尝试不再改善状态时、当风险层级变化时或当需要人工审查时停止。

**讨论：** 将迭代调试重新框定为PEV循环强调了可靠性来自受治理的状态转换，而非仅仅更好的修复提示。规划外部化意图变更和风险假设；执行在沙箱化和权限化环境中应用它们；验证使用确定性传感器来决定状态是否可接受；当行动空间跨越安全边界时，HITL关卡保持问责。这一框架将静态分析、运行时错误、测试、批判、自反思和人工审查统一为控制论支架的组成部分，该支架在可执行程序状态上调节智能体的轨迹。

## 3.5 用于自适应支架优化的智能体支架工程

智能体支架工程（AHE）命名了一个支架级设计问题：如何衡量和修改将语言模型转变为编码智能体的软件基板。提示工程改变指令，上下文工程改变呈现给模型的证据，而AHE将操作环境本身视为分析的对象，包括工具模式、规划制品、内存策略、检索策略、沙箱配置、验证传感器、权限层级、路由规则、多智能体工作流和人工审查关卡[281,178]。这一视角很有用，因为在代码智能体中观察到的许多失败源于缺失的仓库上下文、脆弱的工具接口、薄弱的验证器、过多的令牌成本、糟糕的重试策略或错配的权限边界，而非模型生成。

现有工作可以理解为三个互补的线索。AutoHarness研究代码支架的自动合成[14]；Meta-Harness将支架设计形式化为模型面向基础设施的优化问题[13]；可观察性驱动的AHE强调对智能体循环失败位置以及应更改哪个支架组件的遥测丰富的诊断[281]。关于反思性提示演化、自演化工作流和实时软件工程智能体的相关工作支持相同的系统观点：改变模型周围的脚手架可以在不重新训练基础模型的情况下改变智能体行为[18,312,182]。来自OpenAI、Anthropic和LangChain的工程指南汇聚在相同的实践教训上：可靠的智能体需要显式的支架循环、工具契约、轨迹回放、评估套件、上下文预算和受控的执行边界[248,249,313,314,28]。

### 3.5.1 深度遥测作为优化基板

AHE的核心基板是**深度遥测**：连接模型决策、支架行动、环境状态和结果的结构化轨迹。浅日志可能只记录最终答案或通过/失败结果。深度遥测更详细地记录决策过程：提示和检索上下文、令牌使用和成本、模型/工具延迟、工具参数、权限请求、编辑的文件、沙箱快照、命令输出、测试结果、栈轨迹、lint警告、分支决策、被拒绝的替代方案、人工干预和最终任务结果。在以代码为中心的环境中，这些轨迹特别有价值，因为程序执行已经通过日志、测试、差异和运行时行为暴露了状态转换[128,96,37]。在生产系统中，这一角色日益由记录轨迹、指标、提示、模型流量、评估结果和成本信号的可观察性和可靠性栈来服务[315,316,317,318,319,320,321,322,323,324,325,326,327]。评估、可观察性和治理系统因此提供互补的遥测通道：评估器暴露任务级回归，追踪栈暴露轨迹级原因，策略网关暴露边界违规，演化智能体可以将其转化为支架修改。

遥测将支架修改从轶事式调试转变为比较诊断。令牌成本轨迹揭示检索或反思阶段何时消耗预算而不改善验证结果。决策树轨迹显示智能体何时重复选择非生产性工具、编辑无关文件或在失败策略之间循环。失败轨迹聚类重复模式，如缺失依赖、薄弱测试、幻觉API、不稳定沙箱、过度许可的工具调用或过早终止。由于这些信号与具体制品相关联，它们可以在不同支架版本之间回放和比较，使得可以评估变更是否提高了可靠性而非仅仅改变了表面行为[216,217]。

### 3.5.2 演化智能体

**演化智能体**是一个元级智能体，使用深度遥测来提出、评估和推广对支架组件的修改。与编辑目标仓库的任务智能体不同，演化智能体编辑后续任务智能体工作的操作条件。其输入是轨迹语料库；其输出可能是修改的提示模板、检索策略、更精确的工具模式、添加的验证器、更改的权限规则、工作流拓扑调整或新的回归测试。这一角色与自演化多智能体系统密切相关，其中专门智能体检查执行日志、将失败归因于工作流组件并更新协作结构[328,329]。在支架设置中，同一思想从多智能体拓扑泛化到智能体运行时的控制面。

一个典型的演化智能体循环包含五个阶段。首先，它通过收集来自PEV执行的遥测来**观察**轨迹。其次，它通过将成本、延迟、无效行动、测试失败或权限拒绝归因于特定支架组件来**诊断**失败模式。第三，它**提出**候选修改，如重写工具描述、更改上下文打包规则、添加linter、修改重试限制或在危险命令前插入HITL关卡。第四，它在保留任务或回放轨迹上使用确定性传感器和回归测试**评估**修改后的支架。最后，它仅**推广**那些提高可靠性、成本或安全性而不在先前解决案例中引入回归的变更。这使AHE保持在与PEV循环相同的工程纪律内：提议的变更必须在采纳前被执行、验证并成为可审计的。

### 3.5.3 受治理的支架变异

AHE不应与无约束的自修改混淆。因为演化智能体改变控制后续任务智能体的支架，其行动需要比普通代码修复更强的治理。候选支架变更应在沙箱内评估、针对固定回归套件进行比较并以可审计的理由记录。改变权限边界、网络访问、凭据处理、部署行为或人工审查要求的变更应在激活前需要HITL批准。在这个意义上，演化智能体本身受PEV循环约束：它规划支架变异，在隔离评估环境中执行，通过遥测和回归测试验证结果，并将风险变更升级给人类。

**讨论：** 智能体支架工程将代码即支架的视图从操作智能体扩展到分析操作它们的基础设施。深度遥测提供证据，用于定位提示、工具、内存、沙箱、验证器、权限和工作流中的失败。演化智能体使用这些证据来提出和评估支架变异，将支架设计转变为由验证和人工批准治理的迭代和可测量的工程过程。

# 4. 规模化支架：多智能体代码编排

随着AI系统解决日益复杂的问题（从函数级代码合成到仓库级系统工程），单智能体的根本局限性显现出来：（1）上下文窗口约束阻止单个智能体在内存中容纳整个代码库、长交互历史和执行轨迹；（2）专门化需求使得使用一个通用智能体同时进行规划、合成、测试、审查和调试效率低下；（3）缺乏独立的协调和验证通道阻止智能体在长周期执行中可靠地检测和纠正自身错误。多智能体系统引入了一个强大原则：一旦这些责任被分配到专门的角色上，智能体支架本身就变得更具模块化、可检查和可适应。ChatDev[330]、MetaGPT[55]和AgentCoder[50]等早期系统通过在架构师、程序员、测试员、审查员和执行员等不同智能体之间划分软件开发职责，展示了这一转变。通过结构化通信协议和共享代码制品进行协调，这些角色专门化的智能体将代码从单纯的输出目标转变为整体支架规划、行动、验证和自我改进的共享基板。

在本节中，我们系统调查利用MAS扩展编码支架这一快速发展的方向，并提出关于为AI智能体构建共享以代码为中心的支架基板的新立场。

## 4.1 通过多智能体协作改进编码支持

多智能体系统最直接的贡献是通过将支架分解为专门但协调的组件来改进编码支持。这些系统不将规划、合成、执行和验证集成到单个智能体循环中，而是将责任分布在通过共享代码制品和反馈信号交互的角色之间。这种劳动分工使整体支架更能处理复杂的软件任务，同时也使其内部工作流更具可检查和可控性。在实践中，这种改进通过三个紧密相关的设计维度实现：角色如何专门化、智能体如何在共享程序制品上交互以及工作流拓扑如何组织它们的协作。

**图11：多智能体编码支架路线图。**

### 4.1.1 功能角色专门化与人工引导规划

在人类软件开发中，不同角色在开发过程的不同方面有所专长。许多MAS自然通过为不同智能体分配不同的功能角色来镜像这种劳动分工。这种专门化允许每个智能体专注于共享代码支架的特定切片，利用其独特能力和视角为整体任务做出贡献。

**程序合成智能体**负责生成或转换代码。它们消耗规范、计划或反馈信号，并产生或修改代码制品。这是调查系统中最常见的角色。代表性实例包括Self-Collaboration[56]中的Coder、AgentCoder[50]中的Programmer、MetaGPT[55]中的Engineer、ChatDev[330]中的Developer以及MAGE[339]中的RTL Generation Agent。

**程序理解智能体**分析现有代码或规范以产生更高层次的表示。它们负责代码含义的解释而非代码的功能。这一类别包括MAGIS[332]中的Repository Custodian、HyperAgent[333]中的Navigator、Lingma SWE-GPT[340]中的RepoUer以及CleanAgent[341]中的Column-type Annotator。

**验证智能体**评估代码质量，通常通过生成测试用例、运行静态分析或模拟执行。AgentCoder[50]中的Test Designer独立于代码生成测试用例以避免循环推理——这是针对模式崩溃问题的设计原则，即智能体的有偏测试会通过其自身有缺陷的代码。QualityFlow[253]中的Test Quality Checker在元层面解决此问题，在合成的测试被用作反馈前进行过滤。AutoSafeCoder[52]中的Static Analyzer和Fuzzing Agent分别通过静态CWE分析和动态崩溃检测提供安全导向的验证。CANDOR[342]中的Panelists独立审计预言机正确性，针对自然语言规范而非代码本身，从而有意识地避免受缺陷实现的污染。

**执行智能体**直接与程序运行时交互。关键的是，AgentCoder[50]中的Test Executor是一个确定性的Python脚本（而非LLM），干净地将推理与执行分离，并将反馈信号建立在客观程序行为之上。HyperAgent[333]中的Executor通过交互式bash shell运行单元和集成测试。MAGE[339]中的Judge Agent与RTL模拟工具接口，产生每时钟边沿波形快照。

**规划智能体**将整体软件开发任务分解为子任务并将其分配给合成智能体。MetaGPT[55]中的Architect和Project Manager、MAGIS[332]中的Manager、FlowGen[335]中的Scrum Master以及SoA[212]中的Mother智能体都执行任务分解。SoA[212]中的Mother智能体特别值得注意：它们根据推断的每个子功能复杂性在运行时动态地生成Child智能体，使规划和智能体初始化相互依赖。

EvoMAC[328]的一个独特特征是引入了两个在其他调查系统中不存在的元角色：Gradient Agent，它读取执行日志以识别哪些智能体导致了失败；以及Updating Agent，它相应地修改智能体提示和重构工作流DAG。这些角色在MAS本身而非程序的层面上操作，使系统能够根据执行反馈适应其自身结构。

### 4.1.2 基于共享程序状态的多样化交互模式

与通用MAS（智能体交互主要是消息传递）不同，以代码为中心的交互的特点是**制品介导的通信**：智能体观察和修改共享代码制品，它们的交互基于这些制品及其执行结果所暴露的客观状态。这些协调通道比源代码更广泛：智能体通过API、文件、差异、测试、日志、模式、黑板和显式工作流状态进行通信。在大多数系统中，这些通道是人工设计的支架的一部分，而智能体动态写入或修改在其中流通的制品。我们识别四种交互模式。

**协作合成**发生在两个智能体共同构建程序组件时，类似于结对编程[343]。PairCoder[334]中的Navigator-Driver配对是最直接的实例化：Navigator生成和选择解决方案计划，而Driver实现它们，具有双向信息流。CodePori[331]在Dev_01和Dev_02之间实现协作合成，他们在三轮中交换代码草稿。这种模式在调查系统中相对罕见，因为大多数系统更喜欢顺序交接而非真正的共建。

**批判与修复**是调查文献中的主导交互模式。验证或评估智能体检查代码制品并产生结构化反馈；合成智能体随后根据反馈修改制品。这种模式几乎出现在每一个调查系统中。其关键设计决策是：（a）批判是LLM模拟的还是有执行基础的（Self-Collaboration[56]使用模拟的LLM测试员，而AgentCoder[50]使用真实的Python执行器）；（b）反馈信号的丰富程度（从SEW[312]的二元通过/失败到EvoMAC[328]中列举满足需求、函数错误和未满足需求的结构化执行日志）；（c）在回退前允许的修复迭代次数。

**对抗性验证**是一种更主动的验证形式，其中一个智能体试图通过对抗性输入破坏代码，而非被动审查。AutoSafeCoder[52]通过其Fuzzing Agent实现此功能，该智能体使用类型感知变异生成崩溃诱导的输入种子并执行代码以产生崩溃轨迹。这种模式与批判-修复有根本不同的特征：模糊器不解释哪里出了问题，而是展示一个具体的执行失败——编码智能体必须解决的反例。MAGE[339]类似地将模拟不匹配用作对抗性信号：Debug Agent接收第一个时钟边沿失败周围的精确波形窗口，支持有针对性的修复。

**推理辩论**涉及智能体就决策的正确性或规范的解释进行辩论，然后达成共识。ChatDev[330]引入了沟通去幻觉机制，其中助手智能体在承诺响应前反转角色以提出澄清问题。FlowGen[335]中的Scrum冲刺会议使多智能体能够在共享上下文缓冲区周围进行无序讨论，然后由Scrum Master综合决策。CANDOR[342]实现了最结构化的辩论机制：三个独立的Panelists评估预言机正确性，Curator通过多数投票汇总其裁决。MAGIS[332]中的启动会议涉及Manager和所有Developer智能体之间的循环发言，以协商任务依赖关系并防止冲突。

### 4.1.3 用于智能体协调的优化工作流拓扑

智能体交互的拓扑——谁与谁通信、按什么顺序、多少次——是代码生成MAS中最具后果的设计决策之一。我们沿两个主要轴组织拓扑。

**预定义启发式拓扑：** 大多数调查系统使用镜像已建立的软件开发生命周期（SDLC）模型的拓扑。这些拓扑在设计时固定，不会响应任务复杂性或系统性能而改变。

**链式（瀑布）拓扑**以严格线性顺序排列智能体，制品单向从规划流经合成到验证。ChatDev[330]和MetaGPT[55]是典型示例，显式建模瀑布SDLC：设计→编码→测试。FlowGen[335]将三种SDLC模型作为不同拓扑操作化：FlowWater（严格瀑布链）、FlowTDD（需求→设计→测试→实现→修复，测试驱动重排序）和FlowScrum（循环迭代冲刺）。该论文独特地直接比较了不同SDLC镜像拓扑对代码质量的影响。L2MAC[344]也遵循链式拓扑，但有一个新颖之处：指令计划中的每一步都由一个全新上下文的智能体执行，使链成为共享仅外部文件存储的独立LLM调用序列。

**循环（敏捷/迭代）拓扑**引入反向边，允许代码响应验证反馈而被修改。AgentCoder[50]实现了一个程序员→测试执行器→（如果失败）→程序员循环，限制为5次迭代。Self-Collaboration[56]在其瀑布链中嵌入了一个编码者↔测试者反向边，最多4次迭代。PairCoder[334]通过多计划探索增强了循环模式：通过k-means++聚类预生成n个解决方案计划的池以确保多样性，当通过基于历史的循环分析检测到死胡同时，循环可以切换到下一个候选计划。MAGE[339]结合了线性初始化链和循环调试-判断循环，并引入高温候选采样以同时探索多个程序变体。

**层次拓扑**将一个或多个管理者智能体置于一组工作者智能体之上，实现分解-委派模式。MAGIS[332]有一个Manager，它在运行时为每个候选文件动态实例化一个Developer智能体；每个Developer编辑其分配的文件并向管理者-审查层报告。HyperAgent[333]在多个仓库导航和编辑工作者之上使用规划器，结合自上而下的分解与自下而上的仓库证据。SoA[212]通过允许Mother智能体根据推断的子任务复杂性递归生成Child智能体，进一步推进了这一层次。这些系统将支架编排本身视为资源分配问题。

**星形拓扑**围绕一个中枢智能体集中，该中枢协调多个并行工作者智能体。CANDOR[342]的Stage 3面板是一个示例：Requirement Engineer扇出到三个独立的Panelist+Interpreter流水线，Curator聚合其输出。MetaGPT[55]的发布-订阅消息池创建了一个事实上的星形拓扑，共享池作为中枢。

**目标驱动和自适应拓扑：** 一个较小但快速增长的类别的系统将拓扑本身视为设计变量，以针对代码质量信号进行优化。FlowReasoner[338]和BOAD[337]等近期系统通过将多智能体组织本身视为按任务生成、搜索或优化的自适应对象，进一步强化了这一趋势。

**动态智能体池扩展**是最简单的自适应形式：智能体数量随任务复杂性扩展，但拓扑类型固定。SoA[212]通过Mother和Child智能体的层次树实现此功能，其中Mother智能体在运行时决定分解为多少个子功能，生成相应的Child智能体。关键洞察是每个智能体的上下文窗口保持有界，因为复杂性通过增长智能体池而非增长个体上下文窗口来处理。MAGIS[332]类似地根据仓库分析期间识别的候选文件数量动态实例化Developer智能体。BOAD[337]将这一思路从动态扩展扩展到层次发现：它不手动固定专门的子智能体结构，而是将有用定位、编辑和验证子智能体的选择形式化为赌博机优化问题，表明自动发现的层次团队可以胜过手动设计的团队。

**反馈驱动的DAG重构**以EvoMAC[328]为代表。其工作流是一个DAG，节点对应智能体，边定义信息流。每次迭代后，Gradient Agent读取执行日志以将失败归因于智能体，Updating Agent修改提示和图结构。这是调查中唯一支架拓扑根据执行反馈进行结构性修改的系统。

**运行时自组织**是SEW[312]的方法：系统使用应用于LLM生成的工作流描述（以BPMN、CoRE、Python、YAML等结构化格式）的直接演化（DE）和超演化（HE）算子来生成和变异整个工作流规范。SEW[312]不优化智能体参数，而是优化工作流结构，包括智能体调用序列、路由逻辑和反馈路径。它发现的两种规范性拓扑（线性链和反馈循环）来自优化而非手工设计。FlowReasoner[338]通过训练一个查询级元智能体，在外部执行反馈下为每个输入问题生成定制的多智能体系统，使拓扑选择本身成为深思熟虑推理过程的一部分而非固定系统设计，进一步推动了这一自适应视角。

## 4.2 执行反馈与共享支架同步

我们讨论一组智能体如何利用代码的可执行性，以及它们如何维护程序状态的一致共享视图。这一维度对于以代码为中心的MAS具有定义性意义：共享支架是唯一可执行的，并产生客观的预言机信号。我们处理两个子问题：使用哪些类型的执行反馈，以及共享状态如何在智能体间同步。

### 4.2.1 执行反馈集成

**编译器和语法反馈**在运行时前捕获结构错误，被许多系统使用。ChatDev[330]将来自测试阶段的编译器错误反馈给程序员，尽管仅作为单阶段内的一次性修正。L2MAC[344]在其评估器模块E(D)每次文件写入后运行语法检查，将其视为阻止指令流水线推进的阻塞条件。

**测试通过/失败信号**是最常用的执行反馈类型。AgentCoder[50]将其整个循环围绕独立生成的测试用例是否通过；迭代在全部通过或达到5次迭代预算时终止。QualityFlow[253]引入了一个值得注意的变体：想象执行，其中LLM模拟Python解释器逐步执行并预测测试结果而不实际运行代码，在MBPP上达到98%以上的精确率和召回率，同时避免了可见测试用例的标签泄露。Self-Collaboration[56]的模拟LLM测试员与其真实编译器消融的几乎相同性能提出了一个经验性问题：何时实际执行是必要的，何时语言对执行的模拟就足够了？

**模糊器崩溃轨迹**代表了一种质上不同的反馈类型：不是通过/失败结果，而是提供具体的失败输入。AutoSafeCoder[52]使用类型感知变异生成崩溃诱导的输入种子，并将崩溃输入加退出码传递给Coding Agent。这种对抗性反馈比通用失败信号更具信息性，因为它将错误定位到特定的输入类别。

**静态分析警告**提供关于代码结构和安全属性的反馈，无需执行。AutoSafeCoder[52]使用针对MITRE漏洞数据库的CWE映射静态分析，使Static Analyzer Agent能够建议针对特定漏洞类别的修复策略。

**性能分析结果**被MACRO[345]独特地利用，它将代码优化而非正确性作为主要任务。Performance Evaluator Agent测量执行时间、内存使用和FLOPS，MACRO[345]独特地通过实时网页搜索从研究文献中检索相关优化技术来增强此功能。

**细粒度模拟反馈**MAGE[339]的独特贡献是调查文献中最细粒度的执行反馈。它不是仅报告测试台通过或失败，而是通过State Checkpoint机制记录每个时钟边沿的信号值，并向Debug Agent传递第一个失败时钟周期周围的波形窗口。这实现了子测试粒度的针对性修复。

### 4.2.2 共享支架同步

**顺序交接**是最常见的同步机制：每个智能体接收前驱的输出并将其自身输出传递给后继。程序状态仅以流水线中最新的制品形式存在。这对于简单的线性流水线足够，但在多个智能体并行或迭代修改代码库的多智能体设置中创造不可见的状态分歧。这也是代码介导的协调限制变得清晰的地方。即使智能体共享可执行制品，支架仍然施加信息论约束：通道有有限带宽，摘要引入压缩损失，日志变得嘈杂，缓存视图变得过时，并行分支引发关于权威和一致性的未解决问题。代码为协调提供了更丰富的基板，但它不消除这些分布式系统约束。

**共享黑板**提供所有智能体可以读取和更新的全局可访问程序状态。L2MAC[344]最干净地实现了这一点：文件存储D是一个外部持久结构，从不被覆写而是被扩展和修改。Control Unit管理所有读写操作，确保每次智能体调用接收到精确控制的上下文窗口。MAGIS[332]的仓库演化记忆M是一个持久键值存储，将文件版本映射到LLM生成的摘要，通过专门的仓库级推理黑板递增更新。Self-Collaboration[56]是首批显式命名和调用黑板架构的系统之一，建立了所有三个角色读写共享内存。

**带合并的并行分支**在多个智能体同时修改独立组件时出现，其变更在后期阶段集成。MAGIS[332]为每个候选文件实例化一个Developer；每个独立修改其分配的文件，所有变更合并到最终仓库差异中。HyperAgent[333]通过Redis队列并行运行多个Navigator和Editor实例，结果在Planner级别合并。

**结构化上下文调度**是对每个智能体看到什么以及何时看到的显式管理。它是L2MAC[344]的主要创新。Control Unit在指令步骤之间重置上下文窗口，为每次新调用提供先前进展的针对性摘要（Mrs），而非完整的对话历史。当上下文窗口接近容量时，Control Unit将部分结果存储到D并使用压缩视图重新初始化，明确指示LLM在剩余上下文裕量下哪些文件应读取或跳过。这一机制不是通过扩展窗口而是通过仔细控制其内容来解决上下文窗口问题。MetaGPT[55]通过发布-订阅消息池实现了较轻形式的上下文调度：每个智能体仅订阅与其角色相关的文档类型，接收共享状态的过滤视图。

**层次内存**结合短期工作上下文与长期积累的知识。ChatDev[330]明确将短期内存（阶段内的完整对话）与长期内存（跨阶段携带的提取解决方案）分离。Cogito[346]实现层次内存，借鉴神经生物学架构：短期内存用于即时任务状态，长期知识库用于积累的专业知识，以及成长单元用于随时间改进的演化抽象。HyperAgent[333]使用轻量级LLaMA-3.1-8B摘要器在执行日志存储到层次内存前进行压缩，防止上下文膨胀。

**智能体池扩展**从正交角度解决上下文管理问题：它不是管理单个智能体看到的内容，而是将上下文负载分布到更多智能体上。SoA[212]是典型示例：通过随任务复杂性增长而生成更多智能体，每个智能体的上下文保持有界。这是对支架状态问题的结构性解决方案：SoA[212]不构建所有智能体可查询的共享表示，而是将任务状态分区到智能体之间，每个持有一个有界切片。其局限性在于全局一致性被牺牲：智能体无法推理完整程序，只能推理其分配的子功能。

**其他**QualityFlow[253]的回滚机制代表了一种同步模式：初始代码制品从不被覆写，使系统能够在调试轨迹降低质量时回滚到先前的共享支架状态。这是调查系统中唯一显式管理状态历史而非总是向前推进的工作。

## 4.3 立场：共享以代码为中心的支架基板

我们为下一代多智能体智能体提出一个新立场：**共享以代码为中心的支架基板**（shared code-centric harness substrate）。这一立场由文献中识别的核心差距驱动：缺乏智能体可以跨迭代查询和更新的共享代码状态的正式、持久表示。我们认为构建这样的支架基板对于实现稳健、可扩展的多智能体智能体既是可行的也是必要的。

### 4.3.1 共享支架表示

任何MAS的基础问题是：**这些智能体栖居的基板是什么？** 在代码即智能体支架中，自然答案是共享程序环境，即智能体共同作用并随着智能体产生、修改和评估代码而演化的制品、执行上下文和质量信号的集合。我们称之为**共享支架基板**，并区分现有系统表示它的四个形式化级别。

**隐式/仅文件表示：** 最常见且最不形式化的类别将共享支架简单地视为当前代码文件或代码文件集。智能体在输入上下文中接收最新代码制品，并产生修改或评估版本。没有持久的可查询表示：共享状态通过对话历史在每次智能体调用时隐式重建。这一类别包括许多基础系统：ChatDev[330]、MetaGPT[55]、FlowGen[335]、MapCoder[44]、CodeCoR[166]、SEW[312]和CodePori[331]。虽然这种表示简单实现，但它蕴含一个根本限制：智能体只能通过其最近上下文窗口的狭窄视角来推理共享基板。状态分歧[347]——智能体关于代码状态的内部信念偏离真实状态——对系统不可见且无法被检测或纠正。

**基于仓库的表示：** 一个更丰富的系统类别将共享支架表示为可导航的仓库：具有目录结构、文件间依赖图、调用层次和版本历史的文件系统。这种表示支持智能体推理需要在代码库中何处进行更改、哪些其他组件依赖于更改的函数以及代码库如何随时间演化。MAGIS[332]引入了仓库演化记忆，缓存文件级摘要并在文件跨问题解决情节通过git diff变化时递增更新。HyperAgent[333]为智能体提供仓库导航工具（get_tree_structure、go_to_definition、code_search、get_all_references），将仓库视为结构化知识库。Lingma SWE-GPT[340]通过抽象语法树（AST）骨架压缩仓库视图，保留函数签名和类定义以实现高效导航。SyncMind[347]是唯一正式将仓库基板定义为真值状态S_k并测量S_k与智能体信念状态B_k之间分歧的工作。

**基于执行的表示：** 基于执行的表示是代码生成最独特的类别。它在通用MAS中没有直接对应物，通过执行行为表示共享基板。状态不是代码看起来如何，而是代码做什么：是否编译、通过哪些测试、模糊器发现什么漏洞、运行多快、其运行时行为是否匹配规范。这种基于执行的表示提供客观的预言机信号，不受纯粹语言智能体评估中存在的幻觉或偏差影响。利用这种表示的系统包括AgentCoder[50]、AutoSafeCoder[52]、QualityFlow[253]、MACRO[345]、EvoMAC[328]、CANDOR[342]和MAGE[339]。值得注意的是，MAGE[339]实现了文献中最细粒度的执行反馈，通过State Checkpoint波形快照在时钟边沿粒度操作。

**黑板/共享状态表示：** 第四类别引入了所有智能体可以读取和写入的显式、全局可访问的数据结构（类似于AI中的经典黑板架构[348]）。这种共享状态是文献中最接近正式支架基板的近似：它跨智能体调用持久存在、可被查询和更新，并为所有智能体提供程序状态的一致视图。Self-Collaboration[56]是首批显式调用黑板隐喻的系统之一，建立了所有三个角色（Analyst、Coder、Tester）读写共享内存。L2MAC[344]实现了文献中最原则化的黑板：具有语义有意义路径的持久文件存储D，通过显式管理每次智能体调用看到的状态切片的Control Unit访问。GameGPT[210]使用共享上下文缓冲区减少多轮游戏开发中的冗余信息重传。Cogito[346]借鉴神经生物学架构实现三层内存：短期工作状态、长期知识库和用于演化抽象的成长单元，作为结构化的支架表示。

**核心差距：** 系统在这四个类别上的分布揭示了一个显著模式：大部分文献停留在隐式/仅文件类别中，缺乏共享支架基板的任何形式模型。这是推动代码即智能体支架框架的核心差距。程序在多智能体领域中唯一地是一个**执行**的制品。它产生客观的、非语言的信号，原则上可以锚定正式的共享基板。然而，大多数系统未能在架构层面利用这一属性，而是依赖智能体通过自然语言单独推理代码质量。

### 4.3.2 支架状态收敛

**收敛**决定多智能体编码支架何时应停止迭代并将当前程序状态接受为满意结果。在许多现有MAS中，收敛仍然是隐式定义的，要么通过智能体间的共识，要么通过外部迭代预算。然而，代码即智能体支架有一个独特的优势：因为共享基板是可执行的，收敛可以建立在客观行为信号之上，而非仅依赖于会话协议。我们识别六种收敛模式，范围从广泛使用的测试门控和隐式收敛到较少常见的基于安全、性能和共识的标准。

**正确性收敛（测试门控）**是最原则化和广泛使用的客观标准：系统在所有测试用例通过时成功终止。AgentCoder[50]、L2MAC[344]、SyncMind[347]和CANDOR[342]实现测试门控收敛。PairCoder[334]通过死胡同检测增强此功能：如果相同的错误代码或反馈出现在迭代历史中，系统切换到下一个候选计划而非循环。FlowGen[335]使用测试门控收敛，但针对的是LLM生成的测试而非真值测试，引入了潜在的质量问题：系统可能收敛到通过自身有偏测试但在外部评估中失败的代码。

**安全收敛**由AutoSafeCoder[52]独特实现：当静态分析未标记CWE漏洞且模糊器未诱导崩溃时，系统成功终止。这种多标准收敛是基于执行的支架框架的有力论据。两个收敛标准都建立在客观程序行为之上，而非智能体意见。

**性能收敛**是MACRO[345]的重点：优化循环在用户定义的运行时和内存阈值满足时终止，由Performance Evaluator根据实际执行基准测量。这是唯一将性能而非正确性作为主要收敛标准的系统。

**基于分数的收敛**使用智能体评估中间输出的定量质量分数来决定何时停止。MAGE[339]按其模拟不匹配分数s(r)=1-m(r)/tc(r)对候选程序排序，并继续迭代直到最大分数达到1.0。CodeCoR[166]使用四标准二进制分数（清晰度、相关性、简洁性、上下文）在每个智能体阶段剪枝中间输出，并选择其Ranked Code Set中排名最高的代码作为最终输出。它设置了软正确性收敛，提交最佳可用结果而非等待完美解决方案。Trae Agent[336]在仓库规模引入了一个密切相关的搜索和选择视角：它将问题解决形式化为最优解决方案搜索问题，并使用模块化生成、剪枝和选择智能体来导航候选补丁的大型集合空间。在这种设置中，收敛不仅是重复修复的问题，也是在仓库感知证据下对竞争解决方案进行排名、过滤和选择的问题。

**共识收敛**聚合来自多个审查智能体的判断。CANDOR[342]在三个Panelists之间就预言机正确性实施多数投票。MAGIS[332]使用QA Engineer的LLM判断作为接受信号，尽管这是单智能体共识而非多智能体投票。QualityFlow[253]使用其Code Quality Checker作为单一门控信号。这是一种高效设计，质量检查器既作为收敛预言机又作为系统控制器，支持提前退出（75-84%的问题在首次生成器调用后收敛）。

**隐式收敛：** 在固定阶段或迭代次数后终止而没有客观质量标准，是文献中最普遍的收敛模式，代表了该领域最重要的差距。ChatDev[330]在固定阶段数后终止，或当两轮产生相同代码时，或10轮后，这些都不是客观的质量信号。MetaGPT[55]在完成固定SOP阶段后终止。Self-Collaboration[56]在n=4次迭代后如果测试员从未批准则回退到隐式收敛。EvoMAC[328]运行固定K次文本反向传播循环。隐式收敛的普遍性是缺乏正式共享基板的直接后果：没有程序状态的客观表示，系统就没有原则性的收敛标准。

## 4.4 模式与趋势

跨系统，角色专门化、共享状态表示、执行接地和工作流拓扑的差异不是独立的工程选择；它们交互决定一组智能体能在多大程度上在长周期编码任务中保持连贯性。在本小节中，我们提炼从调查系统中出现的主要趋势，突出当前系统的共同结构性瓶颈和指向更稳健共享支架的设计原则。

**隐式支架状态约束：** 大多数调查系统（ChatDev[330]、MetaGPT[55]、FlowGen[335]、CodePori[331]、SEW[312]、MapCoder[44]、CodeCoR[166]）在没有共享代码支架显式表示的情况下运行。这些系统依赖智能体在每次调用时从对话历史隐式重建状态。这种设计选择适用于程序状态简单且不会在智能体间分片的函数级任务。然而，这种隐式方法创造了一个根本脆弱性：没有正式的共享基板，智能体无法可靠地检测其内部理解何时偏离真实程序状态[347]。从代码即智能体支架的角度看，对隐式状态表示的依赖是系统脆弱性的技术根源，而非可扩展性的便利。

**代码介导的通道不消除协调瓶颈：** 从自由形式对话到代码介导协调的转变是一个真正的架构进步，但不应被夸大。文件、API、差异、测试、日志、模式、黑板和工作流状态都是任务状态被编码、传输和重建的部分通道。每个通道在保真度、延迟和范围之间权衡：测试将语义压缩为通过/失败，摘要以细节为代价节省上下文，日志接地但嘈杂，共享黑板提高持久性同时创造权威和一致性问题。核心设计问题因此不仅是代码是否存在，而是哪些制品是权威的、它们如何被压缩以及跨通道冲突如何解决。

**执行反馈作为语言和形式推理之间的桥梁：** 文献中最深的分歧在于使用执行作为真值的系统和依赖语言模型判断的系统。将共享状态建立在执行上的系统（AgentCoder[50]、AutoSafeCoder[52]、QualityFlow[253]、EvoMAC[328]、MAGE[339]）可以访问客观的预言机信号——不能产生幻觉的信号。然而，一个令人惊讶的发现使这一图景复杂化：Self-Collaboration[56]和QualityFlow[253]证明LLM模拟的执行可以在不运行代码的情况下达到98%以上的精确率和召回率，预测实际结果。这表明执行反馈的价值并非在所有失败模式上一致。它在检测语言模拟结构上无法想象的边缘情况（运行时崩溃、资源耗尽、边界条件错误、性能回归）方面表现出色，但对于许多可纠正的错误，模拟推理可能就足够了。一个成熟的支架将集成两者：使用语言推理作为快速路径，仅对需要它的失败模式委派执行作为验证预言机。

**共享支架的两种互补表示：** 调查系统揭示了两个概念上正交的视图：基于仓库的表示（结构：什么函数调用什么、数据流往何处、依赖关系是什么）和基于执行的表示（行为：代码运行时做什么、状态如何在运行时演化、不同输入下出现什么紧急失败）。MAGIS[332]和HyperAgent[333]主要在仓库视图中操作，使智能体能够推理代码库架构。AgentCoder[50]和MAGE[339]主要在执行视图中操作，将共享状态建立在运行时信号之上。然而，没有一个调查系统完全将两种视图统一到单个支架基板中，使智能体能够跨代码的静态结构和动态行为进行推理。最深入的支架将集成这两种视角，回答诸如"哪些组件慢"（需要调用图和分析数据）或"这个重构是否破坏外部代码依赖的API"（需要静态分析和动态测试）等问题。

**拓扑复杂性与支架状态形式化负相关：** 具有显式、形式化共享基板的系统使用更简单的拓扑，而缺乏形式化共享状态的系统使用日益复杂的拓扑模式作为结构性变通。L2MAC[344]具有最清晰的正式支架基板（具有显式上下文调度的持久文件存储），使用带有复杂状态管理的简单顺序链。相比之下，EvoMAC[328]和SEW[312]等隐式状态系统开发了精心设计的自适应拓扑（动态DAG、工作流变异、智能体池扩展），试图在缺乏原则性共享表示的情况下优化协作结构。这表明拓扑复杂性部分是症状：当基板被形式化表示且可查询时，智能体可以通过简单、透明的协议进行协调。当基板隐式时，智能体需要更丰富的交互模式来补偿缺失的状态信息。

**上下文管理是隐式共享状态的代价：** 一个显著模式是许多系统开发了复杂的上下文管理机制，正是因为他们缺乏正式的共享基板。L2MAC[344]的Control Unit、MetaGPT[55]的发布-订阅池、SoA[212]的智能体池扩展和Cogito[346]的三层内存都是对同一根本问题的响应：如何给智能体提供对太大以至于无法容纳在任何单个上下文窗口中的代码支架的连贯视图。一个成熟的支架基板可以通过提供任务状态的原则性、可查询表示来统一这些不同的解决方案，智能体按需访问，而非强迫系统在每一步仔细管理每个智能体看到什么。

**智能体专门化增加了共享状态度量的关键性：** 随着智能体角色多样性的增加——从基本的编码者-测试者对到具有Architect、Manager、Navigator、Executor和Verifier角色的系统——对统一共享基板的需求变得迫切。没有对代码状态的共享理解，Planning Agent可能基于过时的代码库快照分解任务，Execution Agent可能针对与Synthesis Agent意图不同的版本运行测试，Verification Agent的反馈可能误射。EvoMAC[328]通过其Gradient和Updating智能体在MAS层面显式监控失败归因来解决这个问题。SyncMind[347]将问题形式化为智能体信念分歧|B_k - S_k|，提出显式的同步协议。智能体角色的 proliferation 因此不仅仅是工程选择。它是开发更成熟共享支架的强制函数。具有丰富角色库的多智能体系统没有它们就无法稳健运行。

# 5. 新兴领域与开放问题

在通过其接口、机制和编排模式刻画了代码即智能体支架之后，我们现在考察这一范式如何在具体应用领域中具体化，以及它暴露了哪些开放问题。在编码助手、GUI/OS智能体、科学发现、个性化和具身智能体中，代码不仅作为模型输出，还作为状态表示、行动执行、内存、反馈和治理的操作基板。这些领域使以代码为中心的智能体系统的承诺变得具体，同时揭示了一组关于评估、验证、安全、协调、多模态接地和支架演化的共同未解决挑战。

## 5.1 新兴领域与具体应用

本小节调查五个应用领域，其中代码即支架系统变得尤为可见。编码助手在仓库、测试、开发工具和协作工作流上操作；GUI和OS智能体通过可执行动作和程序化检查器操作渲染界面；科学智能体将假设、实验、分析和实验室协议组织为可执行流水线；个性化智能体通过结构化用户反馈和可编辑偏好状态自适应推荐策略；具身智能体将高层意图基于可执行技能，受物理约束。这些领域共同展示了代码如何将模型输出连接到现实世界系统，以及周围支架的设计如何塑造可靠性、可控性和长周期自主性。

**图12：代码作为智能体支架在五个新兴领域的概览。**

### 5.1.1 编码助手

编码助手提供了最清晰的应用领域之一，以代码为中心的智能体系统在其中变得可操作。早期系统主要支持局部补全或单轮代码生成。近期的助手转而跨仓库级工作流操作，其中编辑、工具使用、验证和拉取请求交互形成闭环智能体过程。这一转变体现在SWE-agent[57]和OpenHands[58]等研究系统，以及Claude Code[26]、Codex[27]、GitHub Copilot编码智能体[349]和DeepAgents[350]等面向生产的平台中。在这些系统中，助手不再是独立的代码生成器。它嵌入在开发环境中，仓库状态、工具、验证例程和协作工作流提供行动和反馈的操作上下文。

**以仓库为中心的工作区：** 现代编码助手在仓库而非孤立代码片段上操作。源文件、测试、构建脚本、依赖元数据、问题、分支和拉取请求构成一个持久的工作区，智能体可以在多个步骤中检查、修改和验证。这使得仓库级助手更少关注将相关文件放入提示中，而更多关注在大型演化代码库上构建任务特定的工作视图。RepoCoder[47]、CodexGraph[351]和AutoCodeRover[46]等系统通过仓库索引、依赖感知检索、基于图的代码表示和编辑前的智能体定位来解决这个问题。在这个意义上，仓库成为编码助手规划、行动和接收反馈的操作基板。

**可执行开发支架：** 可执行开发支架正在成为编码助手的运行和控制平面。近期系统不将模型暴露给扁平的工具有列表，而是将其包裹在管理开发循环中，控制仓库访问、文件编辑、命令执行、批准边界、上下文隔离、日志记录和验证。这一趋势在生产系统中可见：Claude Code将本地终端/IDE/浏览器编码打包到工具介导的循环中，包括编辑、命令执行、权限、钩子、内存和子智能体；Codex和GitHub Copilot编码智能体将类似循环移动到托管云或GitHub本地工作区，具有沙箱、分支、批准和可审计的拉取请求输出；DeepAgents将规划、文件系统支持的状态、上下文管理、代码执行和子智能体委派暴露为可重用的支架组件[26,27,350,349]。此类循环日益由开放协议如模型上下文协议（MCP）[352,260]中介，该协议标准化了支架如何向模型暴露工具、上下文和资源，并实现了跨系统工具重用。与此同时，近期研究将支架本身视为优化对象而非固定包装器：AutoHarness[14]从环境反馈合成代码支架，Meta-Harness[13]使用先前候选和执行轨迹搜索支架代码，Agentic Harness Engineering[281]通过可观察性演化编码智能体支架，Natural-Language Agent Harnesses[353]将角色、契约、适配器和状态约定外部化为可编辑的支架规范。这些发展共同表明，编码助手的实际进步日益不仅受基础模型改进的影响，也受周围执行运行时的影响，包括其沙箱、权限、上下文管道、遥测和验证钩子。

**执行反馈作为接地验证：** 编码助手的一个显著特性是可机器检查反馈的可用性：编译器诊断、测试结果、linter警告和运行时轨迹。Agentless[354]表明，一个由测试执行引导的故障定位和补丁生成流水线在SWE-bench[216]上取得了竞争性结果，无需精细的智能体控制。RepairAgent[183]和Live-SWE-agent[182]将这一循环扩展到由测试结果驱动的自主程序修复，而AlphaCodium[355]表明测试驱动的流工程在竞赛编程性能上显著优于单次提示。执行因此将每个候选编辑从文本假设转变为程序世界的可验证转换。

**仓库规模的内存和上下文管理：** 仓库经常超过任何合理的上下文窗口，迫使编码助手维护显式的结构化内存。检索增强的补全[47]、基于图的代码索引[351]、面向文档的智能体如RepoAgent[356]以及近期的上下文检索基准如ContextBench[357]实例化了§3.2中的内存抽象，并带有代码特定的转折：存储的项目如函数、测试、轨迹和检索到的问题上下文本身是可执行的或直接与可执行状态相关，可以重新运行、检查或定位，而非仅重新读取。近期的内存系统通过将可重用的智能体过程或仓库经验存储为程序和体验内存进一步扩展了这一观点[45,48]。这缩小了传统智能体架构中内存与环境之间的差距，并使抽象管理特别严峻，因为助手必须为给定的子任务选择合适的代码和经验尺度。

**开发者意图和项目约定作为潜在状态：** 超越显式的仓库状态，实际编码助手必须推理潜在的开发者意图和项目约定。一个有用的补丁不仅应通过可见测试，还应与仓库的架构、编码风格和内部API重用对齐——近期工作描述为生成代码的**有机性**（organicity）的属性[358]。忽略这些约束的智能体可以产生技术上正确的补丁，但维护者仍然拒绝[358,359]，而基准分析显示一些看似解决的SWE-bench问题依赖于问题文本中的解决方案泄漏而非真正的意图推断[360]。编码辅助因此是一个部分可观察的程序世界问题：文件、测试和工具输出提供可观察状态，而设计原理、隐含约束和团队约定必须从问题线程、先前提交、代码审查和积累的交互历史中推断。这将SyncMind中研究的信念状态分歧从共享多智能体状态扩展到个体智能体和用户对齐[347]。建模这种潜在状态对于从功能性代码生成向可信的开发者协作迈进至关重要。

**从内联补全到自主SWE智能体：** 编码助手的演化可以被视为模型周围开发支架的扩展。早期系统如基于Codex的补全[1]和商业助手如Copilot[361]依赖轻量级IDE支架，其中局部上下文被浮出，内联建议被生成，开发者仍是主要的执行者、验证者和状态管理者。生产力[361]和可用性[362,363]研究表明即使是这种轻量级支架也很重要，因为建议的价值取决于其与开发者演化中的程序状态和意图的对齐。在自主端，SWE-agent、OpenHands、AutoCodeRover和Agentless等系统在仓库级支架内操作，从孤立的代码生成转向有状态的检查、编辑、执行和修改。

**从补丁生成到软件生命周期参与：** 编码助手也在从孤立的补丁生成走向更广泛的软件生命周期参与。SWE-bench将仓库级辅助框定为问题到补丁的任务[5]，而较新的基准如SWE-Lancer[364]和SWE-Bench Pro[365]评估更长时间范围、经济意义上重要的软件交付物，跨越多个文件并需要专业工程努力。相关基准如Terminal-Bench[366]和AppWorld[367]进一步反映了向交互式环境的同一转变，智能体必须通过命令、工具和可执行应用状态来操作[368,369]。在部署中，这一趋势表现为智能体在持久工程工作流内工作，而非静态仓库快照，包括拉取请求审查、CI/CD反馈和生产问题解决[370,371]。在生产规模上，LingmaAgent报告显示，阿里云自主部署的问题解决智能体完全自主解决16.9%的内部问题，另有43.3%需要人工干预[372,373]。这表明编码助手正在成为工作流参与者，而不仅仅是补丁生成器。

**多智能体编码辅助与共享仓库：** 在频谱的高端，编码辅助日益采取多智能体形式，规划者、编码者、测试者和审查者角色在共享仓库上操作。ChatDev[330]、MetaGPT[55]、CodeAgent[185]和METAL[374]展示了角色专门化与共享可执行制品相结合如何实现单智能体难以在长周期内维持的协调模式。仓库及其测试和执行轨迹既成为通信媒介又成为收敛目标，直接实例化了§4的共享程序世界。然而，并发编辑会静默地使其他智能体持有的假设失效，暴露了世界状态同步挑战。尽管存在这些挑战，多智能体设置代表了最强大的编码支架范式之一，因为角色专门化、共享制品和可执行反馈的结合实现了超越单一模型架构能力的编码能力。这些系统的成功进一步加强了代码即智能体支架的论点：最有效的编码助手不是更好的模型，而是更好的支架——管理仓库上下文、工具、反馈和协作角色的系统。

### 5.1.2 GUI/OS智能体作为程序世界

GUI和操作系统智能体构成第二个主要领域，其中代码即智能体支架范式自然适用。在这种环境中，智能体通过操作界面元素、软件应用程序、命令行工具和Web服务与数字世界交互。与文本代码不同，GUI/OS环境的状态是部分可观察和多模态的，通常混合像素级和结构化的表示。动作空间通常是离散但组合性的，通过点击、输入、键盘快捷键、shell命令和API调用来表达。这一领域日益采用的策略是将GUI/OS交互映射到可执行代码上，将界面状态、执行历史和反馈信号统一到程序化支架中。

当前系统显示了多种代码即支架的策略。最全面的方法之一是Cradle[403]，它利用可执行代码动作、自我反思、技能库和记忆来在稀疏环境中实现跨游戏和软件应用的通用交互。通过将感知环境状态编译为代码生成的条件，它使用代码作为各种环境的行动统一接口。OS-Copilot[402]和Executable Code Actions[222]进一步证明，在FRIDAY框架中，可执行Python代码优于结构化API、按钮点击和纯文本命令，因为代码抽象共享的运行时库、条件分支和迭代循环，有效地编码了复杂任务策略。TheAgentCompany[404]扩展了这种基于执行的交互，通过创建涉及内部工具、网页和通信的内部业务环境，智能体通过编码生成和工具使用执行全方位的工作相关任务。类似地，AutoGLM[419]使用代码生成作为网络浏览、表单填写和应用程序管理的主要范式，利用结构化代码来序列化GUI交互，从而更稳健地处理导航、数据提取和多步骤工作流。

除了这些通用的基于代码的框架，已经开发了专门的系统来利用代码作为处理特定GUI/OS挑战的支架。SeeClick[405]和其他视觉接地工作[406,407,408,409,410,411,412]将视觉表示与程序化行动生成桥接，确保GUI交互反映正确的屏幕语义。通过将视觉语义与代码生成相结合，这些系统使用代码作为接地动作空间。UI-Voyager[35]和[418]将自演化方法引入GUI智能体，使系统能够积累和细化技能库以进行更有效的引导和检索。与此同时，OS-Atlas[407]和UI-TARS[411]通过将GUI状态映射到标准化的行动空间进一步系统化了GUI交互的代码表示，使得智能体可以生成可执行的交互序列。基于Web的智能体如WebVoyager[395]和WebArena[60]也通过将网络交互抽象为可执行的代码动作来操作，这些动作可以针对实时网站或模拟环境执行和验证。这些方法共同将GUI/OS交互建模为在可观察接口状态上的代码生成问题，显式地或在DOM树、可访问性API和渲染截图中接地。这使代码即支架视角自然延伸，因为执行既是行动又是验证：智能体编写被解释器直接执行的交互，然后通过后续状态检查验证结果。

评估此领域的框架也拥抱基于代码的交互。AndroidWorld[391]、OSWorld[368,396]、Windows Agent Arena[392]和VisualWebArena[423]等基准要求智能体生成可执行动作，这些动作在模拟或真实环境中运行并根据状态转换进行验证。终端智能体Bench[366]、AgentBench[12]、InterCode[11]及其他[393,424,394,395,425]强调了代码执行作为评估核心部分的强调。工作流导向的基准如WorkArena[398]和Spider2-V[413]进一步要求智能体通过可执行动作导航复杂的工作相关界面，在这些场景中，代码被用作行动支架。在基准之外，还有许多基础设施工作，包括用于沙箱的E2B[252]、用于可复现评估的SWE-ReX[266]以及用于执行反馈的Code2World[38,426]，它们共同构成了使GUI/OS智能体可执行的系统层。

工业系统也反映了这一转变。Anthropic的Computer Use[427]赋予Claude直接操纵计算机界面的能力——移动光标、点击按钮和输入文本——通过将屏幕截图映射到可执行动作。相比传统方法，这种基于视觉的交互不需要平台特定的API。类似地，OpenAI的Operator[428]和Google DeepMind的Project Mariner[429]使用类似的基于视觉的界面控制。这些平台实现了代码即支架方法的基本原则：智能体通过与可执行环境的程序化交互来推理、行动和接收反馈。代码不再是目标；它是智能体与数字世界之间的支架。

### 5.1.3 自主具身智能体

具身智能体代表代码作为行动和环境的支架自然适用的第三个领域。在这种设置中，智能体必须通过感知和行动与物理或模拟世界交互。代码即支架的方法通过将高层意图转化为在物理约束下验证的可执行技能，弥合了语言模型推理和物理世界之间的差距。

具身智能体中的代码即支架通过多种互补策略实现。在策略生成方面，Code-as-Policies（CaP）[10]使用LLM生成的可执行Python程序作为机器人策略。与之前的端到端学习系统不同，CaP利用代码的结构化性质——循环、条件语句和数学运算——生成组合和适应性强的机器人控制程序。RoboCodeX[33]通过多模态、树结构代码生成扩展了这一概念，支持导航和操作任务。Code-BT[34]将LLM生成的代码编译为行为树，实现了受约束的执行和迭代运行时反馈。GenSwarm[118]进一步将程序化控制扩展到多智能体机器人系统，协调跨多个具身智能体的策略生成和部署。在技能获取方面，Voyager[32]通过自动课程和持续扩展的可执行技能库为Minecraft中的开放式任务确立了终身代码智能体的范式。Lyra[121]将人类修正编码为可重用的可执行技能和检索增强记忆。ViReSkill[122]将视觉接地重新规划与技能记忆缓存相结合，以在环境失败下保持稳定交互。Ski11sCrafter[123]面对可执行操作技能的持续积累，通过持续语言条件操作结构解决灾难性遗忘问题。在可行性接地方面，SayCan[9]通过将语言规划与接地技能执行相结合，在具身约束下连接抽象推理和物理行动。KnowNo[110]引入基于一致预测的不确定性感知控制。SkillVLA[111]将问题扩展到组合具身环境中的结构化技能重用。BOSS[112]通过语言引导的实践合成新技能链，随时间扩展行动空间。

评估方法同样反映了代码执行的基础性。具身智能体评估通常依赖于模拟器，模拟器本身就是可执行环境，智能体代码在其中运行和验证。具身AI的基准通常涉及可执行任务，其中代码生成、执行和反馈构成评估循环。即使用于评估的智能体系统（如PALM-E[436]、Gemini Robotics[437]和RT-2）是端到端的，最近的设计选择使用可执行代码作为中间表示和运行时接口，重申了代码作为行动和环境建模的支架的观点。

共同主题是将具身智能体系统组织为三个层次：底层通过使用代码作为技能库实现持久的交互界面；中间层通过在环境反馈中生成策略代码桥接感知和行动；顶层嵌入高层规划和接地，有时使用代码进行物理世界建模和假设检验。这些系统展示了代码即支架视角如何为具身AI提供统一的架构模式。

### 5.1.4 科学发现的智能体作为程序世界

科学发现代表代码即最纯粹形式的支架领域：代码不仅仅是智能体产生的东西，而是假设、实验和发现发生的介质。科学工作流本质上是可执行的——它们涉及模拟、数据处理、统计分析和可视化——使得代码成为支架接口的自然基础。在大语言模型时代之前，科学软件就已经将代码作为操作基板；LLM智能体现在通过动态生成、执行和修改科学代码来加速这一过程。

在生物医学科学中，BioMiN[64]通过结合动态代码生成与模块化工具集来协调假设检验工作流。它展示了LLM智能体如何通过使用代码序列化假设，然后执行程序化实验来弥合生物医学知识与可测试预测之间的差距。虚拟实验室（Virtual Lab）[440]通过协调基于LLM的科学家团队来设计蛋白质结合物，将科学的协作方面直接实例化为一个多智能体代码生成问题。A-Lab[452]通过将代码生成与机器人实验执行相结合，将这一概念扩展到自我驱动实验室，展示了可执行代码如何连接数字规划和物理验证。在化学领域，ChemCrow[61]和Coscientist[62]代表了一种程序化科学发现的方法，其中智能体通过Python代码生成和API调用的循环迭代来推理化学程序、计划合成路线和执行实验协议。CodeKea[64]进一步通过增加文件系统状态管理和长周期上下文规划来扩展代码介导科学方法的方法。在量子化学中，El Agente[449]自动化自洽场计算和结构优化，编写在量子化学软件上执行的可执行脚本。在一般科学发现中，AI Scientist[63]和AI Scientist-v2[438]通过自动化假设生成、实验设计、数据分析、论文写作和迭代同行评审的完整循环，将代码支架推向自动化。其工作流的核心是代码生成和执行：用代码编写假设，通过可执行模拟进行测试。AIDE[448]将代码空间中的科学发现建模为搜索问题，通过迭代代码生成和执行来在机器学习中优化性能指标，使用代码作为发现介质。AlphaEvolve[451]将自动算法发现的边界推向了算法优化的科学，通过代码生成和基于执行的反馈来演化科学和算法代码。MatPilot[454]和[447]进一步突出了AI科学家的不同角色，前者专注于AI与人类的协作代码生成，后者提供LLM科学智能体的全面分类。

在科学发现中，代码即支架方法引入了一个关键约束：代码必须产生科学上有意义的结果，而不仅仅是文本连贯的输出。这建立了与其他领域不同的验证标准：科学发现中的代码可执行性带来了客观性的负担——结果必须可复现、现实世界可操作，并符合科学标准。如PaperQA[450]和MLE-bench[456]等工作集中于使用代码生成来查询、综合和生成关于复杂问题的科学信息，强调了代码在科学推理中的核心地位。随着这些系统成熟，它们正在将代码支架从简单的工具使用演变为使科学方法自动化的核心基板：提出假设、以代码形式设计实验、通过执行测试预测、分析结果，并通过后续代码修改迭代改进。

### 5.1.5 智能体个性化

个性化提出了一个独特的代码支架挑战：推荐和适应系统必须表示用户状态、交互历史和随时间演化的偏好，同时将推荐策略接地到可执行代码。代码即支架的方法通过使用代码作为用户建模、策略适应和反馈集成的接口来应对这一挑战。

在用户建模方面，ReMind[465]将代码生成集成到推荐循环中，其中LLM生成表示用户交互序列的程序化推理路径。Agent[466]将规划与执行动作分析相结合的混合系统，利用代码生成来构建和更新用户档案。A-mem[469]和MemRec[471]引入基于代码的记忆机制，通过可执行的偏好结构维护持久的用户状态。在策略适应方面，RecoWorld[464]通过可执行模拟环境为推荐策略迭代优化提供测试平台。iAgent[468]实现了一个代码介导的交互层，过滤和调整推荐输出以确保用户控制。用户模拟器[472,473]采用代码生成来创建交互式用户代理，用于训练和评估推荐系统。长期记忆系统如Mem0[470]和[204]提供可执行组件，用于跨会话持久存储用户上下文。对话推荐系统[474]利用代码生成来跟踪交互历史和动态调整建议。

在代码即支架框架下，个性化系统利用代码作为用户状态和行为动态的可执行表示。这使推荐策略能够通过可执行流水线进行测试、验证和迭代改进，而非停留在静态、不可执行的配置文件中。代码的执行性质在推荐上下文中特别有价值，其中策略必须通过真实用户交互或模拟环境进行客观评估。通过将用户建模和策略适应表达为代码，这些系统受益于可执行反馈循环：推荐可以被运行、评估和迭代优化，使得个性化支架能够适应不断变化的用户偏好和上下文。此方法可应用于各种环境，包括网络服务、内容平台和交互式系统，其中代码作为动态表示用户状态和优化个性化策略的灵活基板。

## 5.2 开放问题

在本小节中，我们概述了基于代码的智能体支架在未来几年面临的最重要开放问题。这些挑战横跨评估、验证、安全、协调和多模态交互——共同决定了代码即智能体支架能否从研究原型发展为可靠的生产系统。

### 5.2.1 支架级评估与预言机充分性

代码即支架系统的评估提出了两个核心挑战。首先，大多数现有基准在最终任务成功（如测试通过率）上评估智能体，但这不足以衡量支架在现实条件下支持长期操作的能力。需要更丰富的度量来捕捉：**鲁棒性**（智能体是否优雅处理缺失上下文、错误或边界条件）、**验证强度**（如测试覆盖、预言机多样性和误接受率）、**恢复能力**（智能体能否在无效行动后诊断和修复失败）、**状态一致性**（内存、仓库状态、执行轨迹和智能体信念是否保持同步）、**安全合规**（权限、沙箱和人工批准关卡是否被尊重）以及**可回放性**（完整轨迹能否从日志和制品重建和审计）[475]。这一议程中的一个核心瓶颈是**预言机充分性**：评估器是否捕获了意图任务而非仅仅是狭窄的可执行代理。开放问题不仅仅是构建更难的基准，而是将代码智能体支架作为可执行运行时系统进行评估。

### 5.2.2 超越可执行反馈的语义验证

预言机充分性尤其具有挑战性，因为执行反馈虽然对以代码为中心的智能体至关重要，但可能产生虚假的正确性感觉：代码可以运行，轨迹可以检查，测试可以验证，失败可以反馈回修改中。然而，执行只与其附着的预言机一样可靠。单元测试可能不完整，静态分析器可能过度近似，GUI检查器可能遗漏不可接受的中间行动，科学脚本可能编码无效假设，机器人模拟器可能隐藏物理风险。结果，支架可能恰恰因为拥有可执行反馈而变得过度自信：智能体看到绿色测试，但绿色测试并非完整规范。

缺失的核心抽象是具有显式范围的验证栈。未来的支架不应将通过/失败视为单个终端信号，而应组合多个验证制品：单元测试、集成测试、基于属性的测试、模糊器、静态分析器、类型检查器、安全扫描器、运行时监控器、覆盖报告、形式规范、基于模型的批判和人工审查。每个制品应声明它验证什么、不能验证什么以及它提供什么置信度。这对于自修复和自演化支架尤为重要：如果验证器薄弱，智能体将学会针对错误信号优化。一个有希望的方向是使每个被接受的动作携带证据包，包含已运行的检查、保留的假设、未测试的区域和剩余风险。在此视角下，验证不是最终关卡；它是智能体、支架和环境之间演化的、可检查的契约。

其他有希望的方向包括反馈校准、独立验证、蜕变测试、差异测试、基于属性的测试生成、执行轨迹总结和不确定性感知的评判者[476,477,478]。可靠反馈也应基于其类型以不同方式路由：编译器错误可能触发局部语法修复，测试失败可能触发行为诊断，覆盖缺口可能触发测试生成，不一致的审查者评论可能触发仲裁。更广泛的目标是构建不仅仅是反应性的，而且是认知上 aware的反馈循环：支架应知道信号何时足够强以采取行动，何时薄弱，以及何时需要额外证据。

### 5.2.3 无回归的自演化支架

大多数当前支架是手工设计的：开发者选择规划循环、内存格式、工具集、权限规则、调试过程和智能体拓扑。然而，随着任务变得更长和更多样化，固定支架可能是次优的。一个在竞赛编程中表现良好的支架可能在仓库修复中失败；一个为GUI导航调优的支架可能对科学工作流效率低下；一个在一种任务分布上成功的多智能体拓扑可能在另一种上浪费计算。这表明未来系统应将支架本身视为可编程组件，可以适应新环境，而非围绕基础模型的固定包装器。

自动支架演化已经在进行中。AutoHarness合成约束无效动作的代码支架[14]，MetaHarness在支架代码上搜索[13]，Agentic Harness Engineering从可观察性信号演化支架组件[281]，相关方法通过反思、搜索或执行反馈优化提示、上下文和工作流[18,312,17]。这些系统指向一个更广泛的范式，其中总体优化过程分析运行时反馈（如计算成本、决策路径、工具使用轨迹、内存压力和特定失败案例）并提出对支架本身的修改。此类修改可能重组子智能体间的通信、调整内存分配、修改检索或验证策略，或更改执行反馈在系统中路由的方式。因此，"自动支架演化"本身不是开放问题。更困难的问题是支架能否在不过度拟合、削弱安全性、增加成本、隐藏失败或在罕见但重要的任务上回归的情况下自我改进。

核心洞察是支架变异应像对安全关键运行时的代码更改一样处理。每个提议的编辑应携带变更契约：哪个组件被修改、针对哪个失败模式、预测什么改进、必须保留哪些不变量、什么评估可以证伪它以及如何回滚。这对于支架变更影响未来智能体行为分布尤为重要。新的检索策略可能提高基准准确度同时增加幻觉证据；新的工具模式可能减少令牌成本同时削弱权限边界；新的验证器可能通过接受欠指定解决方案来提高通过率。未来工作应开发携带证据的支架演化、保留的回归套件、安全不变量、金丝雀部署、回滚语义以及支架编辑为何有所帮助的因果证据。目标不是频繁变化的支架，而是仅在能证明变更合理性时才变化的支架。一个实际的研究议程包括：定义支架组件的变异算子；构建遥测标准；跨不同任务评估演化后的支架；在演化过程中强制执行安全不变量；以及将支架的改进与基础模型的改进分离。

### 5.2.4 事务性共享程序状态与语义冲突解决

从单智能体扩展到多智能体系统将代码库转变为共享支架基板。规划者、编码者、测试者、审查者、安全智能体和人类都可能读取和修改重叠的制品。前面的章节显示许多系统仍然依赖顺序交接、共享日志或仅文件状态，而较新的系统引入黑板、仓库记忆、执行反馈和显式信念状态同步[330,55,50,250,347]。开放问题在于同步本身不提供事务语义或假设级一致性：这些机制通常同步制品但不同步假设。一个智能体可能从旧的仓库快照规划，另一个可能测试较新的补丁，第三个可能记住过时的不变量，人类审查者可能引入未传播到系统其余部分的新约束。

缺失的抽象是**事务性共享程序状态**。智能体不应仅仅将消息附加到公共日志；每个行动应声明其读集、写集、假设、版本依赖、验证器义务和冲突策略。冲突不仅应在文件差异级别检测，还应在计划、测试、检索证据、权限、内存条目和潜在用户需求级别检测。未来的支架需要语义性而非纯文本性的冲突解决机制，包括语义合并、回滚、依赖感知锁定、信念状态协调、冲突解释和合并后重新验证。经典版本控制、数据库、CRDT和构建系统提供了有用的类比，但智能体系统添加了传统工具看不见的冲突：不兼容的计划、过时的记忆、重复的子任务、不一致的工具权威和对用户目标的分歧解释。一个关键的研究挑战是确定何时可以自动解决冲突以及何时需要外部判断。此类机制还需要超越合并正确性的度量，包括合并成功率、语义回归率、回滚频率、冲突复发率和人工干预成本。

### 5.2.5 人在环安全与问责作为支架状态

随着代码即智能体支架系统被用于日益重要的场景，安全不能委派给基础模型或仅编码为自然语言指令。在软件部署、网络安全、金融、医疗、科学实验、企业自动化和具身控制等关键领域，智能体行动可能影响生产系统、私人数据、外部用户、物理设备或机构合规性。支架因此需要不仅作为上下文管理器或工具执行器运作，还要作为模型意图和现实世界后果之间的**安全调节器**。它应按风险分类提议的行动、强制权限层级、拒绝违反硬约束的行动，并要求对不可逆或外部有影响的转换进行人工批准。例如，当智能体请求凭据、修改安全关键代码、访问用户数据、部署服务、发布金融或医疗建议或控制物理设备时，支架应能覆盖基础模型并暂停自主性，直到做出人工决策[52,263,119]。

未来的支架需要显式的治理机制，在模型意图和环境行动之间进行中介。一个有用的设计模式是多层权限模型。在最低层，智能体可以读取文件、检查日志和运行静态分析。在更高层，它们可以编辑本地文件、执行沙箱代码、访问网络、调用外部API、修改共享仓库或影响生产系统。每层应指定其允许的行动、约束、审计日志、回滚机制和高风险操作的人在环关卡。此类治理还必须是上下文敏感的。同一命令在一次性沙箱中可能安全，但在生产仓库中不安全；同一网络请求在文档检索期间可能无害，但在传输本地状态时危险。因此，权限不仅应依赖于工具身份，还应依赖于参数、环境状态、数据敏感性和预期副作用。开放问题包括策略规范、副作用预测、沙箱逃逸预防、秘密处理、安全工具模式、可逆执行以及衡量自主性和安全性之间的权衡。

这种安全角色也改变了人类反馈的表示方式。人在环控制不应仅作为偶尔的提示中断出现；它应成为持久的**支架状态**。每个批准、拒绝、策略例外或审查者修正都应更新支架的权限规则、升级策略、验证标准和未来内存检索。同样，高风险的批准应是可审计的状态转换：提议了什么行动、展示了什么证据、暴露了什么风险、谁批准或拒绝了、以及之后什么责任边界发生了变化。开放问题是设计能够决定何时自主性适当以及何时人类判断是必需的支架。在此视角下，可靠的代码即智能体支架系统不仅需要可执行代码和可验证反馈，还需要**可执行的问责**：在行动到达现实世界之前过滤、否决、升级和记录智能体行动的安全层。

### 5.2.6 多模态代码支架系统

大多数代码智能体支架仍然围绕文本状态设计：提示、文件、日志、工具输出、测试和执行轨迹。然而，许多新兴的智能体系统在关键状态是多模态的环境中操作。GUI智能体观察屏幕截图、可访问性树和渲染的界面状态；具身智能体依赖以自我为中心的图像、深度、力、触觉信号、对象姿态以及模拟器或机器人状态；科学智能体检查图表、显微镜图像、分子结构和实验读数。在这些设置中，支架不能再将感知视为模型的被动输入。它必须管理多模态观察作为持久的、可查询的和可验证的状态。

一个核心挑战是多模态上下文压缩。视觉观察庞大、冗余且通常仅部分与任务相关。一个GUI截图可能包含数百个元素，而只有一个按钮重要；一个具身轨迹可能包含数千帧，而只有少数揭示任务关键的对象关系、接触事件或失败原因。未来的支架需要保留任务相关视觉证据而不仅仅是减少令牌成本的压缩机制。这建议了一个多级内存设计：原始图像或帧作为不可变证据存储；对象、区域、元素和姿态级注释提供结构化的中间状态；紧凑的文本或符号摘要仅暴露技能检索和规划所需的信息。开放问题是决定什么多模态信息应保留、抽象、遗忘或提升到长期记忆中，特别是当后续失败揭示早期视觉或物理细节很重要时。

视觉接地引入了第二个挑战：将观察与行动对齐。在文本中心的支架中，行动通常可以对照文件、命令或测试结果进行检查。在视觉环境中，智能体必须将语言目标映射到图像区域、界面元素、对象、坐标、姿态和可执行行动。一个GUI智能体必须知道计划的点击对应于正确的渲染按钮；一个具身智能体必须知道抓取命令在当前摄像头视图和物理配置下针对预期对象。这需要连接感知、行动和验证的支架级接地契约。每个行动不仅应携带自然语言理由，还应携带它所依赖的证据的接地引用，如边界框、对象标识符、UI元素、帧索引、区域特征、对象位置或方向。执行后，支架应验证预期的接地状态是否按预期变化，而非仅依赖模型的自我报告。

在多模态设置中，可靠反馈也更难。文本错误消息或单元测试失败提供显式信号，但视觉和物理反馈通常是隐式的、延迟的或模糊的。一个按钮可能看似被点击而未触发正确的状态转换；一个机器人可能看似持有一个对象而抓取不稳定；一个图表可能看似支持某个结论而其轴刻度改变了解释。未来的支架因此需要多模态验证栈，结合视觉状态检查、对象跟踪、OCR或UI树检查、模拟器状态、物理传感器、触觉反馈和任务特定验证器。更重要的是，每个反馈信号应暴露其范围和不确定性。例如，边界框检测器验证定位但不验证任务完成；模拟器状态验证对象位置但不验证物理鲁棒性；OCR结果验证可见文本但不验证语义正确性。这也要求世界建模和行动建模之间更紧密的集成：支架应预测行动后视觉或物理世界预期如何变化，将该预测与观察结果进行比较，并使用不匹配来诊断失败。在具身和机器人设置中，这种预测误差信号对于恢复尤为重要，因为失败可能源于遮挡、滑移、碰撞、不可达姿态或违反前提条件，而非显式错误消息。将多模态反馈视为校准的证据而非二元成功信号，对于安全的长周期自主性至关重要。

多模态内存还应支持技能演化。在视觉中心领域如GUI控制和具身操作中，可重用技能不能仅表示为文本或代码片段。一个有用的技能通常耦合多模态前提条件、可执行行动模式和预期后置条件：智能体在行动前应看到或感知到什么，应执行什么程序、UI命令或运动原语，以及应发生什么视觉、物理或状态变化。例如，一个GUI技能可能编码如何从截图中定位设置菜单、点击正确区域以及验证新面板出现。一个具身技能可能编码如何识别可抓取对象、选择接近姿态、执行原语控制器以及通过视觉、力或触觉反馈确认对象已进入夹具。此类技能应从成功的轨迹、失败的尝试和人工修正中演化，同时保留其接地证据。支架因此必须决定视觉行动模式何时可重用、应如何抽象存储以及如何跨布局、视点、具身、传感器或任务进行适应。

### 5.2.7 迈向支架工程科学

综合来看，这些开放问题表明代码即支架研究正在向更广泛的支架工程科学迈进。研究的中心对象不再是模型或生成的程序，而是完整的闭环系统：上下文、内存、工具、执行、反馈、安全、协调和评估。进展将需要暴露长周期失败的基准、使轨迹可审计的遥测、隔离支架组件的度量以及允许智能体在持久程序世界中安全操作的设计原则。

最重要的未来系统很可能是那些结合四个属性的系统。首先，它们将是**可执行的**，将决策建立在代码、工具、测试和环境之上。其次，它们将是**可检查的**，暴露计划、状态、来源和失败原因。第三，它们将是**有状态的**，跨长轨迹和多个智能体保留任务相关信息。第四，它们将是**受治理的**，确保自主性受权限、验证和问责的约束。这些属性定义了可靠的长周期智能体AI的下一个前沿。
