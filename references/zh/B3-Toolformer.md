## 摘要

语言模型（LM）展现出从少量示例或文本指令就能解决新任务的卓越能力，尤其是在大规模时。然而，矛盾的是，它们在基本功能（如算术运算或事实查找）上却表现挣扎，而这些领域恰恰是更简单、更小的模型所擅长的。在本文中，我们展示了语言模型可以通过简单的API自学使用外部工具，从而实现两全其美。我们引入了Toolformer，这是一个经过训练的模型，能够决定调用哪些API、何时调用、传递什么参数，以及如何最好地将结果融入未来的token预测中。这一切都是以自监督的方式完成的，仅需为每个API提供少量示范。我们整合了一系列工具，包括计算器、问答系统、搜索引擎、翻译系统和日历。Toolformer在多种下游任务中取得了显著改进的零样本性能，通常可与更大的模型竞争，且不牺牲其核心的语言建模能力。

## 1 引言

大型语言模型在各种自然语言处理任务中取得了令人印象深刻的零样本和少样本结果（Brown et al., 2020; Chowdhery et al., 2022 等），并展现出多种涌现能力（Wei et al., 2022）。然而，所有这些模型都存在几个固有的局限性，这些局限最多只能通过进一步扩展来部分解决。这些局限包括：无法获取关于近期事件的最新信息（Komeili et al., 2022）以及相关的事实幻觉倾向（Maynez et al., 2020; Ji et al., 2022）；理解低资源语言的困难（Lin et al., 2021）；缺乏进行精确计算的数学技能（Patel et al., 2021）；以及对时间流逝的无知（Dhingra et al., 2022）。

克服当今语言模型这些局限的一种简单方法是赋予它们使用外部工具的能力，例如搜索引擎、计算器或日历。然而，现有方法要么依赖大量的人工标注（Komeili et al., 2022; Thoppilan et al., 2022），要么将工具使用限制在特定任务的设置中（例如，Gao et al., 2022; Parisi et al., 2022），这阻碍了在语言模型中更广泛地采用工具使用。

因此，我们提出了Toolformer，这是一个以新颖方式学习使用工具的模型，它满足以下期望：

- **工具的使用应以自监督的方式学习，无需大量人工标注。** 这一点之所以重要，不仅是因为此类标注的成本，还因为人类认为有用的东西可能与模型认为有用的东西不同。
- **LM不应失去其通用性，且应能自行决定何时以及如何使用哪种工具。** 与现有方法相比，这使得工具的使用更加全面，且不局限于特定任务。

我们实现这些目标的方法基于近期利用大型语言模型通过上下文学习（Brown et al., 2020）从头生成整个数据集的思想（Schick and Schütze, 2021b; Honovich et al., 2022; Wang et al., 2022）：仅需少量人工编写的关于如何使用API的示例，我们就让语言模型用一个庞大的语言建模数据集标注潜在的API调用。然后，我们使用自监督损失来确定哪些API调用真正有助于模型预测未来的token。最后，我们在模型认为有用的API调用上对语言模型本身进行微调。如图1所示，通过这种简单的方法，语言模型可以学会控制各种工具，并自行选择何时以及如何使用哪种工具。

由于我们的方法与所使用的数据集无关，我们可以将其应用于最初用于预训练模型的同一数据集。这确保了模型不会失去其任何通用性和语言建模能力。我们在各种不同的下游任务上进行了实验，结果表明，在学习使用工具后，基于预训练GPT-J模型（Wang and Komatsuzaki, 2021）（6.7B参数）的Toolformer在零样本结果上取得了显著提升，在各种任务上明显超过了大得多的GPT-3模型（Brown et al., 2020）以及其他几个基线模型。

## 2 方法

我们的目标是使语言模型M能够通过API调用的方式使用不同的工具。我们要求每个API的输入和输出都可以表示为文本序列。这允许将API调用无缝插入任何给定的文本中，使用特殊token来标记每次此类调用的开始和结束。

我们将每个API调用表示为一个元组 c = (a_c, i_c)，其中 a_c 是API的名称，i_c 是对应的输入。给定一个API调用 c 及其对应的结果 r，我们将不包含和包含其结果的线性化序列分别表示为：

e(c) = <API>a_c(i_c) </API>
e(c, r) = <API>a_c(i_c)→r </API>

其中"<API>"、"</API>"和"→"是特殊token¹。图1展示了一些插入到文本序列中的线性化API调用示例。

给定一个由纯文本组成的数据集 C = {x¹, ..., x^|C|}，我们首先将该数据集转换为一个用API调用增强的数据集 C*。这分为三个步骤，如图2所示：首先，我们利用M的上下文学习能力来采样大量潜在的API调用。然后，我们执行这些API调用，最后检查获得的响应是否有助于预测未来的token；这被用作过滤标准。过滤后，我们合并不同工具的API调用，得到增强数据集C*，并对M本身在该数据集上进行微调。以下将更详细地描述每个步骤。

> **注1：** 在实践中，我们分别使用token序列" ["、"]"和"->"来表示"<API>"、"</API>"和"→"。这使得我们的方法无需修改现有LM的词表即可工作。为便于阅读，在本节中我们仍将它们称为"<API>"、"</API>"和"→"。

**图2：** 我们方法的关键步骤，以问答工具为例说明：给定输入文本 x，我们首先采样一个位置 i 和相应的API调用候选 c¹_i, c²_i, ..., c^k_i。然后我们执行这些API调用，并过滤掉所有不能降低下一个token上损失L_i的调用。所有剩余的API调用都与原始文本交错，形成新的文本 x*。

### 采样API调用

对于每个API，我们编写一个提示 P(x)，鼓励LM用API调用标注示例 x = x₁, ..., xₙ。图3展示了一个用于问答工具的提示示例；所有使用的提示见附录A.2。设 p_M(z_{n+1} | z₁, ..., z_n) 为M将token z_{n+1} 作为序列 z₁, ..., z_n 的延续所分配的概率。我们首先通过计算每个 i ∈ {1, ..., n} 的概率来采样最多k个候选位置用于进行API调用：

p_i = p_M(<API> | P(x), x_{1:i-1})

这是M在位置 i 开始一个API调用的概率。给定一个采样阈值 τ_s，我们保留所有位置 I = {i | p_i > τ_s}；如果这样的位置超过k个，我们只保留前k个。

对于每个位置 i ∈ I，我们然后以序列 [P(x), x₁, ..., x_{i-1}, <API>] 为前缀，以 </API> 为序列结束token，从M中采样获得最多m个API调用 c¹_i, ..., c^m_i²。

> **注2：** 我们丢弃所有M没有生成</API> token的示例。

### 执行API调用

作为下一步，我们执行M生成的所有API调用，以获得相应的结果。如何执行完全取决于API本身——例如，可能涉及调用另一个神经网络、执行Python脚本或使用检索系统对大型语料库进行搜索。每个API调用 c_i 的响应需要是一个单一的文本序列 r_i。

### 过滤API调用

设 i 为序列 x = x₁, ..., x_n 中API调用 c_i 的位置，r_i 为来自API的响应。进一步，给定一个权重序列 (w_t | t ∈ N)，定义：

L_i(z) = -Σ_{j=i}^{n} w_{j-i} · log p_M(x_j | z, x_{1:j-1})

为M在 token x_i, ..., x_n 上的加权交叉熵损失，前提是模型以 z 为前缀。

我们比较该损失的两种不同实例化：

L^+_i = L_i(e(c_i, r_i))
L^-_i = min(L_i(ε), L_i(e(c_i, ε)))

其中 ε 表示空序列。前者是在将API调用及其结果作为前缀提供给M时，对所有 token x_i, ..., x_n 的加权损失³；后者是不做API调用与进行API调用但不提供响应所获损失的最小值。直观地说，如果同时提供API调用的输入和输出使得模型更容易预测未来的token，相比于根本不接收API调用或只接收其输入，那么这个API调用对M是有帮助的。给定一个过滤阈值 τ_f，我们因此只保留满足以下条件的API调用：

L^-_i - L^+_i ≥ τ_f

即，添加API调用及其结果与不做任何API调用或未获得结果相比，至少减少了 τ_f 的损失。

> **注3：** 我们将 e(c_i, r_i) 作为前缀提供，而不是在位置 i 插入，因为M尚未在任何包含API调用的示例上进行微调，所以将其插入 x 中间会中断流程且与预训练语料库中的模式不一致，从而损害困惑度。

### 模型微调

在为所有API进行采样和过滤调用后，我们最终合并剩余的API调用，并将它们与原始输入交错。也就是说，对于一个输入文本 x = x₁, ..., x_n，其在位置 i 处有对应的API调用和结果 (c_i, r_i)，我们构造新的序列 x* = x_{1:i-1}, e(c_i, r_i), x_{i:n}；对于具有多个API调用的文本，我们类似处理。对所有 x ∈ C 执行此操作，得到用API调用增强的新数据集 C*。我们使用这个新数据集对M进行微调，使用标准的语言建模目标。关键的是，除了插入的API调用之外，增强数据集 C* 包含与原始数据集 C 完全相同的文本。因此，在 C* 上微调M使其暴露的内容与在 C 上微调相同。此外，由于API调用恰好插入在那些帮助M预测未来token的位置，并使用那些确切的输入，因此在 C* 上微调使得语言模型能够完全基于自身的反馈来决定何时以及如何使用哪种工具。

### 推理

在微调后使用M生成文本时，我们执行常规解码，直到M生成"→" token，表明它接下来期望得到某个API调用的响应。此时，我们中断解码过程，调用相应的API获取响应，然后在插入响应和</API> token后继续解码过程。

## 3 工具

我们探索了多种工具来解决常规LM的不同缺陷。我们对这些工具施加的唯一限制是：（i）它们的输入和输出都可以表示为文本序列，以及（ii）我们可以获得一些关于其预期用途的示范。具体来说，我们探索了以下五种工具：问答系统、维基百科搜索引擎、计算器、日历和机器翻译系统。表1展示了与每种工具相关的API的可能调用和返回字符串的示例。我们将在下面简要讨论所有工具；更多细节见附录A。

### 问答（Question Answering）

我们的第一个工具是一个基于另一个LM的问答系统，可以回答简单的事实性问题。具体来说，我们使用Atlas（Izacard et al., 2022），这是一个在Natural Questions上微调过的检索增强型LM（Kwiatkowski et al., 2019）。

### 计算器（Calculator）

作为第二个工具，我们使用一个可以执行简单数值计算的计算器；我们只支持四种基本算术运算。结果总是四舍五入到两位小数。

### 维基百科搜索（Wikipedia Search）

我们的第三个工具是一个搜索引擎，给定搜索词，返回来自维基百科的短文本片段。与我们的问答工具相比，该搜索使模型能够获得关于某个主题的更全面信息，但需要模型自己提取相关部分。作为我们的搜索引擎，我们使用BM25检索器（Robertson et al., 1995; Baeza-Yates et al., 1999），它索引了来自KILT的维基百科转储（Petroni et al., 2021）。

### 机器翻译系统（Machine Translation System）

我们的第四个工具是一个基于LM的机器翻译系统，可以将短语从任何语言翻译成英语。更具体地说，我们使用600M参数的NLLB（Costa-jussà et al., 2022）作为我们的多语言机器翻译模型，该模型适用于200种语言（包括低资源语言）。源语言使用fastText分类器自动检测（Joulin et al., 2016），而目标语言始终设置为英语。

### 日历（Calendar）

我们的最后一个工具是一个日历API，当被查询时，返回当前日期而不需要任何输入。这为需要一定时间意识的预测提供了时间上下文。

## 4 实验

我们研究我们的方法是否能使模型在没有进一步监督的情况下使用工具，并自行决定何时以及如何调用哪些可用工具。为了测试这一点，我们选择了多种下游任务，在这些任务中我们假设至少有一个考虑中的工具是有用的，并在零样本设置中评估性能（第4.2节）。除此之外，我们还确保我们的方法不会损害模型的核心语言建模能力；我们通过查看两个语言建模数据集上的困惑度来验证这一点（第4.3节）。最后，我们研究了学习使用工具的能力如何受模型大小的影响（第4.4节）。

### 4.1 实验设置

**数据集生成** 在我们的所有实验中，我们使用CCNet的一个子集（Wenzek et al., 2020）作为我们的语言建模数据集 C，并使用GPT-J（Wang and Komatsuzaki, 2021）作为我们的语言模型M。为了降低用API调用标注 C 的计算成本，我们为某些API定义了启发式规则，以获取 C 的一个子集，使得该子集中的API调用比平均文本更可能有用。例如，对于计算器工具，我们只考虑包含至少三个数字的文本。所用启发式规则的详细信息见附录A。为了从 C 获得 C*，我们执行第2节中描述的所有步骤，并额外过滤掉所有在过滤步骤中所有API调用都被消除的示例⁴。对于加权函数，我们使用：

w_t = \~w_t / Σ_{s∈N} \~w_s，其中 \~w_t = max(0, 1 - 0.2·t)

以确保API调用发生在API提供的信息对模型真正有帮助的位置附近。阈值 τ_s 和 τ_f 为每种工具单独选择，以确保足够大的示例数量；详见附录A。表2显示了我们的最终数据集的有关统计数据。

> **注4：** 虽然这种过滤改变了训练示例的分布，但我们假设剩余示例与原始分布足够接近，使得M的语言建模能力不受影响。这一假设在第4.3节中得到经验验证。

|---|---|---|---|
| 问答 | 51,987 | 18,526 | 5,135 |

**模型微调** 我们在 C* 上微调M，批大小为128，学习率为1·10⁻⁵，在前10%的训练步骤中进行线性预热。微调过程的详细信息见附录B。

**基线模型** 在本节的后续部分，我们主要比较以下模型：

- **GPT-J**：一个没有经过任何微调的常规GPT-J模型。
- **GPT-J + CC**：在 C（我们的CCNet子集，没有任何API调用）上微调的GPT-J。

对于大多数任务，我们还与OPT (66B)（Zhang et al., 2022）和GPT-3 (175B)（Brown et al., 2020）进行比较，这两个模型分别比我们的其他基线模型大约10倍和25倍。

### 4.2 下游任务

我们在各种下游任务上评估所有模型。在所有情况下，我们都考虑提示式零样本设置——即，模型被指示用自然语言解决每个任务，但我们不提供任何上下文示例。这与先前关于工具使用的工作（例如，Gao et al., 2022; Parisi et al., 2022）形成对比，在这些工作中，模型会获得数据集特定的示例，展示如何使用工具来解决具体任务。我们选择了更具挑战性的零样本设置，因为我们想看看Toolformer是否能在用户没有预先指定应该以何种方式使用哪些工具来解决特定问题的情况下正常工作。

我们使用标准贪心解码，但对Toolformer做了一处修改：我们不仅当<API>是最可能的token时允许模型开始API调用，而且只要它是k个最可能的token之一就允许。对于 k=1，这对应于常规贪心解码；我们改用 k=10 以增加模型利用其可访问API的倾向。同时，我们在每个输入中最多允许一个API调用，以确保模型不会陷入不断调用API而不产生任何实际输出的循环。这些修改的效果在第5节中探讨。

#### 4.2.1 LAMA

我们在LAMA基准测试的SQuAD、Google-RE和T-REx子集上评估我们的模型（Petroni et al., 2019）。对于每个子集，任务是用缺失的事实（例如，日期或地点）完成一个简短陈述。由于LAMA最初是为评估掩码语言模型而设计的（例如，Devlin et al., 2019），我们过滤掉掩码token不是最终token的示例，以便剩余的示例可以以从左到右的方式处理。为了考虑不同的分词方式以及未告知模型只需要一个单词所带来的额外复杂性，我们使用比精确匹配略微宽松的评估标准，只需检查正确的单词是否在模型预测的前五个单词内。由于LAMA基于直接来自维基百科的陈述，我们阻止Toolformer使用维基百科搜索API，以避免赋予它不公平的优势。

所有模型的结果见表3。所有不使用工具的GPT-J模型实现了相似的性能。关键的是，Toolformer明显优于这些基线模型，分别比最佳基线提高了11.7、5.2和18.6个百分点。它还明显优于OPT (66B)和GPT-3 (175B)，尽管这两个模型都大得多。这是因为模型在几乎所有情况下（98.1%）都独立决定向问答工具询问所需信息；只有极少数示例（0.7%）使用了不同的工具，或者根本没有使用工具（1.2%）。

#### 4.2.4 多语言问答

我们在MLQA（Lewis et al., 2019）上评估Toolformer和所有基线模型，这是一个多语言问答基准测试。每个问题的上下文段落以英文提供，而问题可以是阿拉伯语、德语、西班牙语、印地语、越南语或简体中文。为了解决该任务，模型需要理解段落和问题，因此可能受益于将问题翻译成英语。我们的评估指标是模型的生成（最多10个单词）包含正确答案的百分比。

结果见表6。使用API调用持续提高了Toolformer在所有语言上的表现，表明它已经学会了使用机器翻译工具。根据语言的不同，该工具在63.8%到94.9%的示例中被使用；唯一的例外是印地语，机器翻译工具仅在其中7.3%的案例中被使用。然而，Toolformer并不一致地优于原始GPT-J。这主要是因为对于某些语言，在CCNet上微调会降低性能；这可能是由于与GPT-J原始预训练数据相比存在分布偏移。

OPT和GPT-3在所有语言上的表现出奇地差，主要是因为它们尽管被指示用英语回答，却未能提供英文答案。GPT-J没有遭受此问题的一个潜在原因是，它在比OPT和GPT-3更多的多语言数据上进行了训练，包括EuroParl语料库（Koehn, 2005; Gao et al., 2020）。作为上界，我们还在MLQA的一个变体上评估了GPT-J和GPT-3，其中上下文和问题都以英文提供。在这种设置下，GPT-3的表现优于所有其他模型，支持了我们的假设，即它在MLQA上表现不佳是由于任务的多语言方面。

#### 4.2.5 时间数据集

为了研究日历API的效用，我们在TEMP LAMA（Dhingra et al., 2022）和我们称为DATESET的新数据集上评估所有模型。TEMP LAMA是一个基于Wikidata构建的数据集，包含关于随时间变化的事实的完形查询（例如，"克里斯蒂亚诺·罗纳尔多效力于___"）以及2010年至2020年间每年的正确答案。DATESET（在附录D中描述）也是通过一系列模板生成的，但使用随机日期/持续时间的组合填充（例如，"30天前是星期几？"）。关键在于，回答这些问题需要知道当前日期。

对于这两项任务，我们使用与原始LAMA数据集相同的评估方法。

表7所示的结果表明，Toolformer在TEMP LAMA和DATESET上都优于所有基线。然而，进一步检查发现，TEMP LAMA上的改进不能归因于日历工具（该工具仅用于0.2%的示例），而主要归因于维基百科搜索和问答工具——这是Toolformer调用最多的工具。考虑到TEMP LAMA中的命名实体通常非常具体和罕见，即使知道确切日期也无济于事，这是合理的。对于这个数据集，最佳行动方案——首先查询日历API获取当前日期，然后使用该日期查询问答系统——不仅受到我们每个示例最多使用一个API调用的限制所禁止，而且鉴于Toolformer训练数据中的所有API调用都是独立采样的，这对它来说也很难学习。

另一方面，对于DATESET，Toolformer相比其他模型的显著改进可以完全归功于日历工具，它在54.8%的示例中使用了该工具。

除了验证在各项下游任务上的性能提升外，我们还想确保Toolformer的语言建模性能不会因为使用API调用的微调而下降。为此，我们在两个语言建模数据集上评估了我们的模型：WikiText（Merity et al., 2017）和来自CCNet（Wenzek et al., 2020）的10,000个随机选择的文档子集（这些文档在训练中未使用）。各种模型的困惑度见表8。正如预期，在CCNet上进行微调略微提高了在不同CCNet子集上的性能，但略微降低了在WikiText上的性能，这大概是因为GPT-J的原始预训练数据与我们的CCNet随机选择子集相比更接近WikiText。

然而，最重要的是，在禁用API调用进行推理时，在 C*（我们的用API调用标注的数据集）上进行训练相比在 C 上进行训练并没有导致困惑度增加⁸。

## 6 相关工作

**语言模型预训练** 有各种方法在预训练期间用某种形式的额外文本信息来增强语言模型，包括各种形式的元数据（Keskar et al., 2019）、HTML标签（Aghajanyan et al., 2021）、维基百科标记（Schick et al., 2022），或从信息检索系统获得的相关文本（Guu et al., 2020; Borgeaud et al., 2021; Izacard et al., 2022）。对于所有这些方法，无论是否有帮助，都会始终提供额外信息。相比之下，Toolformer自学如何明确地请求正确的信息。

**工具使用** 有几种方法旨在赋予LM使用外部工具的能力，例如搜索引擎（Komeili et al., 2022; Thoppilan et al., 2022; Lazaridou et al., 2022; Shuster et al., 2022; Yao et al., 2022）、网络浏览器（Nakano et al., 2021）、计算器（Cobbe et al., 2021; Thoppilan et al., 2022）、翻译系统（Thoppilan et al., 2022）和Python解释器（Gao et al., 2022）。这些模型学习使用工具的方式大致可以分为两种方法：要么依赖大量的人工监督（Komeili et al., 2022; Nakano et al., 2021; Thoppilan et al., 2022），要么通过在针对特定任务量身定制的少样本设置中提示语言模型来工作，其中事先知道需要使用哪些工具（Gao et al., 2022; Lazaridou et al., 2022; Yao et al., 2022）。相比之下，Toolformer的自监督性质使其能够学习如何以及何时使用工具，而无需提供展示如何使用工具的特定任务示例。与我们工作最接近的可能是TALM（Parisi et al., 2022），该方法使用类似的自监督目标来教模型使用计算器和搜索引擎，但仅在模型为下游任务进行微调的设置中进行探索。

**自举（Bootstrapping）** 使用自训练和自举技术来改进模型的思想已在各种背景下得到研究，从词义消歧（Yarowsky, 1995）、关系抽取（Brin, 1999; Agichtein and Gravano, 2000）、句法分析（McClosky et al., 2006; Reichart and Rappoport, 2007）、序列生成（He et al., 2020）、少样本文本分类（Schick and Schütze, 2021a）和检索（Izacard and Grave, 2021）到推理（Zelikman et al., 2022）。与这些方法的思路类似，Toolformer在经过基于困惑度的过滤步骤后在其自身预测上进行训练。

---

## 7 局限性

虽然我们的方法使LM能够以自监督的方式学习如何使用多种工具，但我们的方法在当前形式下所能实现的目标存在一些明显的限制。其中一个限制是Toolformer无法以链式方式使用工具（即，将一个工具的输出用作另一个工具的输入）。这是因为每种工具的API调用是独立生成的；因此，微调数据集中没有链式工具使用的示例。我们当前的方法也不允许LM以交互方式使用工具——特别是对于像搜索引擎这样的工具，它们可能返回数百个不同的结果，使LM能够浏览这些结果或改进其搜索查询（类似于Nakano et al. (2021) 的精神）对于某些应用可能至关重要。

除此之外，我们发现使用Toolformer训练的模型在决定是否调用API时通常对其输入的确切措辞很敏感；考虑到已知LM在零样本和少样本设置中都对其提供的提示非常敏感，这也许并不令人意外（Jiang et al., 2020; Schick and Schütze, 2021a）。根据工具的不同，我们的方法也非常样本低效；例如，处理超过一百万个文档只产生几千个对计算器API有用的调用示例。解决这个问题的一个潜在方法是迭代地应用我们的方法，类似于相关的自举方法中的做法（Schick and Schütze, 2021a; Izacard and Grave, 2021; Parisi et al., 2022）。最后，在决定是否进行API调用时，Toolformer目前没有考虑进行API调用所产生的依赖于工具的计算成本。

---

## 8 结论

我们引入了Toolformer，这是一种语言模型，它通过简单的API调用以自监督的方式学习如何使用不同的工具，如搜索引擎、计算器和翻译系统。这是通过在大量采样得到的API调用上进行微调来实现的，这些API调用根据它们是否降低未来token上的困惑度进行过滤。Toolformer显著提高了6.7B参数GPT-J模型的零样本性能，使其甚至能在各种不同的下游任务上超越大得多的GPT-3模型。

## 附录A：API详情

在采样和过滤API调用时，默认情况下我们使用 τ_s = 0.05 和 τ_f = 1.0——即，我们只在<API> token概率至少为5%的位置进行API调用，并且如果API调用将损失降低至少1.0，则保留它们。我们只保留前k=5个这样的位置，并为文本中识别出的每个位置采样最多m=5个API调用。由于下面描述的启发式过滤，我们仅为CCNet的一个小子集生成计算器和机器翻译系统的API调用；为补偿这一点，我们为这些工具设置 τ_s = 0.0、k=20 和 m=10。由于生成的API调用集仍然相对较小，我们额外设置 τ_f = 0.5。

### A.1 实现

**问答** 我们使用Izacard等人（2022）的Atlas模型（在Natural Questions上微调）作为我们的问答系统。为创建 C*，我们使用Atlas-large，以便高效处理数百万个API调用；在推理过程中，我们使用更大的Atlas-xxl模型。

**计算器** 我们的计算器基于一个简单的Python脚本，只支持运算符"+"、"-"、"*"和"/"。对于语法无效的方程，它不返回任何结果。对于采样API调用，我们对CCNet子集应用启发式过滤，只处理满足以下条件的文档：（i）在100个token的窗口内包含至少三个数字，其中一个数字是对另外两个进行数学运算的结果，（ii）包含序列"="、"equals"、"equal to"、"total of"、"average of"后跟一个数字，或（iii）包含至少三个数字；对于仅符合最后一个条件的文本，我们只保留1%的随机子集。

**日历** 为创建我们的数据集 C*，我们假设在这样的情况下日历日期应该是文档创建的日期。我们通过从URL中提取日期（如果存在）来近似这一点。我们过滤掉无法提取日期的文本，大约保留18%的文档。

**机器翻译** 对于训练和推理，我们都使用600M参数的NLLB（Costa-jussà et al., 2022）作为我们的机器翻译（MT）模型。源语言使用fastText分类器自动检测（Joulin et al., 2016），而目标语言始终设置为英语。由于CCNet数据集大部分是英语，我们在生成API调用之前过滤掉仅包含英语文本的部分。更具体地说，我们只保留那些包含非英语文本块且前后为英语文本的段落。我们使用大小为10个token的文本块。为确定中间文本块是否使用非英语语言，我们再次使用fastText分类器，置信度大于0.8。我们还过滤掉仅包含数字或特殊符号的任何文本块。这种过滤机制使我们能够更高效地生成数据，将API调用的生成集中在MT工具可能有用的地方。在生成MT API调用后，我们还从训练集中移除那些MT工具的输入出现在API调用之后但在它之前未出现的情况。虽然在数据生成过程中模型可以向前查看以生成API调用，但在推理时这是不可能的，因此我们希望劝阻模型不要在这种情况下调用API。
