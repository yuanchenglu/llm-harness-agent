## 摘要

基于 LLM 的智能 Agent 面临着重大的部署挑战，特别是在资源管理方面。允许 Agent 无限制地访问 LLM 或工具资源，会导致资源分配和利用的低效，甚至可能有害。此外，当前 Agent 设计中缺乏适当的调度和资源管理机制，阻碍了并发处理，并限制了整体系统效率。为了解决这些挑战，本文在管理基于 LLM 的 Agent 的背景下，提出了 AIOS（基于 LLM 的 AI Agent 操作系统）的架构。该架构引入了一种新颖的设计，通过将 Agent 应用的资源（如 LLM 和工具）隔离到 AIOS 内核中，来为基于 LLM 的 Agent 提供服务。AIOS 内核为运行时 Agent 提供基础服务（如调度、上下文管理、内存管理、存储管理、访问控制）。为增强可用性，AIOS 还包含 AIOS SDK，一套设计用于利用 AIOS 内核功能的全面 API。实验结果证明，使用 AIOS 可以将不同 Agent 框架构建的 Agent 执行速度提升高达 **2.1 倍**。源代码可在 https://github.com/agiresearch/AIOS 获取。

## 1. 引言

在自主 Agent 领域，研究工作（Wooldridge & Jennings, 1995; Jennings et al., 1998; Bresciani et al., 2004）致力于开发能够感知环境、理解指令、做出决策、采取行动并从反馈中学习的 Agent。大语言模型（LLM）（Achiam et al., 2023; Touvron et al., 2023a; Team et al., 2023）的出现为 Agent 开发带来了新的可能性（Ge et al., 2023a）。当前的 LLM 在理解指令（Ouyang et al., 2022; Chung et al., 2022; Touvron et al., 2023b; Geng et al., 2022）、推理和解决问题（Kojima et al., 2022; Nijkamp et al., 2022; Taylor et al., 2022; Hao et al., 2023; Kim et al., 2023）、以及与人类用户（Ross et al., 2023）和外部环境（Driess et al., 2023; Brohan et al., 2023）交互方面展现了强大的能力。基于这些强大的 LLM，新兴的 LLM-based Agent（Ge et al., 2023a; Yao et al., 2023; Shinn et al., 2023; Deng et al., 2023; Packer et al., 2023; Wu et al., 2024）在从虚拟助手到更复杂的推理和问题解决系统等多样化环境中表现出强大的任务完成能力。

图 1 展示了基于 LLM 的 Agent 执行真实世界任务的一个示例：旅行 Agent 处理一个行程组织请求。该 Agent 方法性地将请求分解为可执行的步骤——根据用户偏好预订航班、预订住宿、处理付款、更新日历。在整个执行过程中，Agent 展现了源自其 LLM 基础的推理和决策能力，这将其区别于受限于预定义功能或工作流的传统应用程序。实现这个旅行场景需要 Agent 无缝集成 LLM 相关服务（偏好检索、API 选择、响应生成）与传统的 OS 服务（磁盘访问、软件执行）。

**当前的 Agent 框架存在关键的设计缺陷**，它们允许 Agent 直接访问系统级资源（如 LLM 和工具）（Qin et al., 2024），这损害了资源优化并引入了潜在的漏洞。没有适当的调度，Agent 可能独占资源，例如向 LLM 大量发送请求而其他 Agent 在等待。缺乏有效的资源管理机制显著损害了系统效率。在并发条件下，现有框架（如 AutoGen、LangChain）对 LLM 调用采用低效的试错方法：提示词被转换为张量并加载到 GPU 内存中，直到 CUDA 内存限制触发异常，强制释放张量并需要多次重试——当众多 Agent 竞争有限的 LLM 资源时，这大幅降低了系统吞吐量。

为了缓解上述限制，我们引入了 **AIOS**，一个旨在更高效地为基于 LLM 的 Agent 提供服务的架构。我们的贡献如下：

- **新的 Agent 服务架构。** 我们引入 AIOS，一种新颖的基于 LLM 的 Agent 服务架构。AIOS 将 Agent 应用和 Agent 资源（如 LLM 和工具）划分为不同的层——即应用层和内核层。这种分离有助于系统化的资源管理、效率优化和安全性增强。
- **AIOS 内核设计与实现。** AIOS 的核心是我们设计并实现的 AIOS 内核，它封装了资源管理抽象。在该内核中，Agent 查询被分解为子执行单元（即 AIOS 系统调用）以促进并行化。我们设计了 Agent 调度器来编排跨模块的系统调用执行，而内存管理器、存储管理器、工具管理器和 LLM 核心则负责处理分派的系统调用。我们还设计了上下文管理器来处理上下文中断，以及访问管理器来验证 Agent 操作，以确保 AIOS 内核的可靠性。
- **AIOS SDK 开发。** 我们开发了 AIOS SDK，它提供了内核功能的更高级抽象，使开发者能够专注于应用逻辑和高级功能，而不必被复杂的内核细节所困扰。
- **实证结果。** 我们对使用各种 Agent 框架开发的 Agent 进行了广泛的 AIOS 评估。结果表明，AIOS 能够在标准 Agent 基准测试中保持 Agent 的性能，甚至能够在涉及工具调用的基准测试中增强性能。此外，AIOS 显著提高了执行效率，为不同框架的 Agent 实现了高达 **2.1 倍**的执行速度提升。这些实验结果强调了 AIOS 在优化 Agent 性能和执行速度方面的有效性。

## 2. AIOS 的架构

如图 2 所示，AIOS 架构分为三个不同的层：应用层、内核层和硬件层。这种分层设计旨在建立系统内清晰的关注点分离。高层应用通过明确定义的接口（如软件开发工具包 SDK 和系统调用）与底层交互，抽象了底层的复杂性。

**应用层。** 应用层利用 AIOS SDK，为请求 AIOS 内的系统资源提供接口。这种设计使 Agent 从资源管理中解脱出来，同时通过阻止直接资源操作来强制执行隔离。AIOS SDK 既支持原生 Agent 开发，也支持从多种框架适配的非原生 Agent，包括 ReAct（Yao et al., 2023）、Reflexion（Shinn et al., 2023）、AutoGen（Wu et al., 2023）、Open-Interpreter（Lucas, 2024）和 MetaGPT（Hong et al., 2023）。非原生 Agent 通过适配器函数与 AIOS 内核资源交互，而原生开发则通过调用系统调用的预定义 API 简化。这种抽象使开发者可以专注于 Agent 逻辑而非资源管理细节。

**内核层。** 内核层集成了两个组件：用于非 LLM 计算任务的传统 OS 内核和我们创新的 AIOS 内核。在 AIOS 内核中，专用模块通过系统调用处理 Agent 请求。调度器使用先进的调度策略（详见第 3.3 节）将这些调用分派到适当的模块。我们设计了一个统一接口，将 LLM 封装为核心（类似于 CPU 核心），支持集成各种不同的 LLM 端点。为支持 LLM 上下文切换，我们实现了一个具有快照和恢复能力的上下文管理器（第 3.4 节）。为高效处理 Agent 数据，我们开发了用于运行时操作的内存管理器（第 3.5 节）和用于持久化存储的存储管理器（第 3.6 节）。此外，工具管理器加载工具并解决 AIOS SDK 支持工具的调用冲突（第 3.7 节），而访问管理器实现访问控制和用户干预协议（第 3.8 节）。

**硬件层。** 硬件层由系统的物理组件组成，如 CPU、GPU、内存、磁盘和外围设备。硬件层不是本工作的主要关注点——AIOS 内核不直接与硬件交互，而是依赖 OS 系统调用来访问硬件层中的物理资源。

## 3. AIOS 内核

在本节中，我们从 AIOS 内核的概述开始，重点说明每个模块如何与其他模块协作以支持集成功能。接着，我们深入探讨每个模块的设计和实现，讨论它们在整体 AIOS 架构中的角色和贡献。

### 3.1 模块间的关系和连接

在 AIOS 内核中，Agent 查询被分解为分类的系统调用（LLM 处理、内存访问、存储操作、工具使用），如图 3 所示，完整的系统调用目录见附录 A.1。每个系统调用都是线程绑定的，由调度器进行分派，调度器集中管理所有模块的队列。系统调用根据其属性集路由到相应的模块队列，每个模块监控其指定的队列以获取已调度的调用。上下文管理器将在 LLM 核心内部被触发来处理上下文中断，而不是通过调度器被调度。

### 3.2 LLM 核心

由于 LLM 有多种部署选项——例如使用哪个 LLM、LLM 托管在云端还是本地设备上、LLM 需要什么硬件条件、使用哪个推理框架——我们将采用不同部署选项的每个 LLM 实例封装为一个核心，类似于传统操作系统中的 CPU 核心。这种设计允许我们将每个 LLM 实例视为一个专用的处理单元，增强了 AIOS 架构内的模块化和可扩展性。为适应不同的 LLM 实例，我们为每个 LLM 实例引入一个包装器，并在该包装器内设计专门的统一系统调用用于 LLM 推理。通过将 LLM 实例抽象为核心并实现标准化的系统调用，AIOS 提供了一种灵活的方式来集成不同部署选项下的 LLM 实例，这归功于 LLM 核心的模块化设计。LLM 核心的详细信息见附录 A.2。

### 3.3 调度器

我们将所有队列集中在调度器模块中，而不是分发到各个处理模块，从而隔离了请求管理职责，使每个模块能够仅专注于执行。这种集中化简化了跨模块的任务协调，并提供了统一的调度框架。为管理系统调用，我们实现了两种经典算法：**先进先出（FIFO）**，按到达顺序处理调用，但可能增加后续请求的等待时间；以及**轮转调度（Round Robin, RR）**，以时间分片方式循环处理调用，实现更均衡的资源分配。RR 策略得到我们的 LLM 推理上下文中断机制的支持，详见附录 A.3。

### 3.4 上下文管理器

LLM 推理时间通过长时间运行的系统调用独占资源而成为瓶颈。我们的上下文中断机制通过任务中断和恢复（快照和恢复操作）来解决这一问题。上下文管理器设计了两种方法：**基于文本的方法**（用于没有 logits 访问权限的闭源 LLM，保存解码输出）和**基于 logits 的方法**（保留中间搜索树结构，用于更细粒度的状态恢复）。

基于 logits 的方法如图 4 所示。使用束搜索（常见的 LLM 技术，Touvron et al., 2023b; Jiang et al., 2023; Biderman et al., 2023），简单起见束宽设为 1，我们来演示这个过程：当处理"确定航班 UA057 目的地是否会下雨"时，LLM 评估每一步的候选 token。如果在生成过程中被调度器挂起，上下文管理器会快照中间结果。恢复时，它重新加载此快照，从挂起点继续执行，达到最终答案"Search weather in Paris"，而无需重新开始计算。

### 3.5 内存管理器

与处理物理 RAM 的传统 OS 内存管理器不同，AIOS 的内存管理器解决的是 Agent 运行时的交互历史（Lerman & Galstyan, 2003; Zhang et al., 2024），包括对话日志和工具调用结果。它管理内存结构、分配、读写操作、删除和更新。Agent 内存默认驻留在 RAM 中，但当分配的空间接近容量时，管理器实现 RAM 和磁盘之间的内存交换。当 Agent 的内存使用超过其块限制（例如，分配的 80%）时，内存管理器启动 **K-最近最少使用（LRU-K）驱逐策略**，通过存储管理器（详详见第 3.6 节）将数据从 RAM 传输到磁盘。LRU-K 优先在 RAM 中保留最近至少被访问 K 次的项，同时将较少频繁访问的项移到磁盘。这在卸放不频繁访问的数据的同时确保需要时仍可检索，平衡了内存效率。内存管理器的详细实现见附录 A.5。

### 3.6 存储管理器

存储管理器处理 Agent 的持久化数据存储，包括 Agent 运行所需的文件或知识库，以及需要持久化存储的 Agent 记忆。在 Agent 运行时，当 Agent 的内存使用超过分配的限制时，内存管理器调用存储管理器将数据交换到磁盘中。具体来说，存储管理器根据从内存管理器传递的 Agent ID 读写数据。除了内存管理器，Agent 本身在运行时也可能请求在磁盘上读写数据，这些 Agent 请求也由存储管理器处理。具体来说，Agent 调用 SDK 中的存储 API，该 API 进一步转换为存储相关的系统调用，并由调度器放入存储队列。存储管理器随后处理队列中的请求以满足 Agent 的需求。存储管理器使用本地文件和向量数据库（如 chromadb）实现。存储管理器的实现细节见附录 A.6。

### 3.7 工具管理器

AIOS 内核中的工具管理器负责管理 AIOS SDK 支持的广泛 API 工具套件。

**标准化工具加载。** 管理器采用标准化接口统一处理各种工具，同时在执行前进行参数验证以防止工具崩溃。当按名称调用时，工具管理器动态加载工具实例，包括可执行文件初始化和依赖项验证。

**工具调用冲突解决。** 对于具有并行访问约束的工具，系统利用哈希表监控实时实例数量。请求处理涉及对使用限制和并行限制进行哈希表验证；当检测到冲突时，系统跳过冲突请求，继续处理队列中的后续请求，直到找到一个无冲突的候选项。实现细节见附录 A.7。

### 3.8 访问管理器

AIOS 内核中的访问管理器提供以下两个关键功能。

**访问控制。** 访问管理器通过实现基于权限的访问控制机制来规范跨 Agent 的数据读写操作。它将每个 Agent 分配给一个特定的权限组，并通过将 Agent ID 映射到其对应权限组的哈希表架构来强制执行权限。访问请求在执行前根据此权限结构进行验证，确保 Agent 只能访问其共享权限域内的其他 Agent 的资源。

**用户干预。** 为降低不可逆操作的风险（删除、覆盖、权限修改），提供了一个用户干预界面供用户确认。这要求在对文件或工具进行具有潜在破坏性的操作之前，必须获得用户的明确验证。实现细节见附录 A.8。

### 3.9 AIOS SDK

我们设计了 AIOS SDK 来简化在 AIOS 架构上开发和集成 Agent 的过程。该 SDK 不仅使开发者能够构建与 AIOS 内核中的核心函数交互的 Agent，还抽象了复杂的系统调用，使开发者能够专注于 Agent 的内部工作流。

**工具集成。** 为支持多样化的 Agent 功能，AIOS SDK 集成了来自各种平台的大量工具，并支持不同情况的输入-输出模态。这些集成工具的详细信息见附录 B.3。

**与 AIOS 内核的交互接口。** 为便利使用 AIOS 内核中 AIOS 系统调用提供的功能，SDK 定义了不同的 API 函数，Agent 可以使用这些函数来调用系统调用并请求资源。

**Agent 框架适配器。** 为支持使用各种 Agent 创建框架构建的 Agent，如 AutoGen（Wu et al., 2023）、Open-Interpreter（Lucas, 2024）和 MetaGPT（Hong et al., 2023），AIOS SDK 为这些框架提供了适配器。这些适配器定位前述框架中的核心函数，并将其重定向到 AIOS 中的函数。这种适配允许来自不同框架的 Agent 在 AIOS 环境中运行，而无需修改 Agent 代码。关于每种 Agent 框架的核心函数和具体适配的进一步细节见附录 B.5。

## 4. 评估

在本节中，我们进行实验来回答以下研究问题：
- **RQ1**：当同时运行多个 Agent 实例时，AIOS 能否在标准基准上保持甚至增强 Agent 的性能？
- **RQ2**：在为大量使用不同 Agent 框架构建的 Agent 提供服务时，AIOS 在优化系统执行吞吐量和减少响应延迟方面的效果如何？
- **RQ3**：当并发运行的 Agent 数量增加时，AIOS 的可扩展性如何？

### 4.1 设置

**模型。** 我们使用 GPT-4o-mini（Achiam et al., 2023）作为闭源 API，并在实验中分别使用两个开源 LLM 作为 LLM 核心：Llama-3.1-8b（Dubey et al., 2024）和 Mistral-7b（Jiang et al., 2023）。开源模型均使用指令微调版本，我们使用 float16 精度。

**硬件。** 我们的实验在配备 NVIDIA RTX A5000 GPU（24GB）的 Ubuntu 22.04 机器上进行。所有实验使用单个 A5000 GPU。

**Agent 框架。** 我们通过运行使用各种流行的 Agent 框架构建的 Agent 进行评估：ReAct（Yao et al., 2023）、Reflexion（Shinn et al., 2023）、AutoGen（Wu et al., 2023）、Open-Interpreter（Lucas, 2024）和 MetaGPT（Hong et al., 2023）。这些 Agent 框架的详细信息见附录 B.5。

**工作负载。** 我们在资源受限的场景下进行评估，即 Agent 并发运行，部署的单个 LLM 一次只能处理一个提示请求。为创建这些并发条件，我们将最大工作线程数默认设置为 250，即最多 250 个 Agent 可以同时并发运行。增加 Agent 数量的影响将在第 4.4 节中分析。默认情况下，我们使用 RR 作为 AIOS 运行 Agent 的调度策略。使用其他策略（即 FIFO）的影响见附录 D。

### 4.2 Agent 性能（RQ1）

为评估使用 AIOS 对 Agent 在标准基准上性能的影响，我们采用四个 Agent 基准——HumanEval（Chen et al., 2021a）、MINT（代码子集）（Wang et al., 2023b）、GAIA（Mialon et al., 2023）和 SWE-Bench-Lite（Jimenez et al., 2024）——分别在无 AIOS 和有 AIOS 的情况下运行 Agent。我们使用成功率（SR%）作为指标，与原始基准一致，并使用 GPT-4o-mini 作为 LLM 核心运行所有 Agent。在所有实验中将 GPT-4o-mini 的温度设为 1.0。基准测试设置和配置的详细描述见附录 C。

| ReAct w/ AIOS | 50.6 | 30.1 | 7.3 | 4.3 |
| Reflexion w/o AIOS | 50.6 | 32.4 | 6.7 | 4.7 |
| Reflexion w/ AIOS | 51.8 | 33.8 | 7.8 | 5.1 |
| AutoGen w/o AIOS | 87.8 | 42.5 | 7.3 | 4.3 |
| AutoGen w/ AIOS | 87.8 | 42.5 | 9.7 | 4.3 |
| Open-Interpreter w/o AIOS | 85.4 | 45.9 | — | 4.7 |

如表 1 所示，集成 AIOS 在标准基准上持续保持了 Agent 性能。在某些情况下，AIOS 还可以促进 Agent 性能提升。例如，在代码生成基准（如 MINT、HumanEval 和 SWE-Bench-Lite）中，AIOS 通过提示增强来提升 Agent 性能，该增强在 LLM 包装器内嵌入了更具结构化的输入和输出格式的系统提示词。这些增强的提示词为 LLM 提供了更多上下文和响应生成的结构性指导。在工具调用基准（如 GAIA）中，Agent 性能甚至通过以下两种机制得到提升：(1) 通过结构化正则表达式进行执行前参数验证，在执行前检查工具调用的格式；(2) 冲突解决哈希表以缓解并发访问问题。

### 4.3 效率分析（RQ2）

在我们的效率实验中，我们使用两个关键指标评估系统性能：**吞吐量**和**延迟**。吞吐量通过计算每秒执行的 AIOS 系统调用数来衡量，指示系统并行处理多个请求的能力。延迟则通过 Agent 经历的平均等待时间来衡量——从查询提交到响应完成的时间，反映系统的响应能力。为确保受控且一致的测试环境，我们使用两个本地托管的开源模型（Llama-3.1-8b 和 Mistral-7b）进行这些评估。本地托管这些模型减少了因网络相关延迟问题导致的 LLM API 响应时间的潜在波动。

如图 6a 和图 7a 所示，结果表明 AIOS 在不同 Agent 框架上实现了显著更高的吞吐量，在 Llama-3.1-8b 上使用 Reflexion-based Agent 时吞吐量提升达 2.1 倍。这一改善归因于 AIOS 内核中采用的调度——通过避免将无法加载到 GPU 上执行的提示词进行不必要的试错尝试，防止了低效率。在延迟方面，如图 6b 和图 7b 所示，Agent 的平均等待时间也大幅减少。这种减少突显了 AIOS 在服务基于 LLM 的 Agent 方面的高效性。

### 4.4 可扩展性分析（RQ3）

我们通过将活跃 Agent 从 250 递增到 2000 来评估 AIOS 的可扩展性，使用 Llama-3.1-8b 和 Mistral-7b 模型在 HumanEval 基准上进行测试。我们复制了 HumanEval 的 164 个样本以匹配 Agent 数量，实现了大规模并发执行。如图 8 所示，AIOS 维持了整体执行时间和平均 Agent 等待时间与 Agent 数量之间的大致线性关系。这表明 AIOS 即使在不断增长的需求下也能高效处理工作负载。相比之下，无 AIOS 和有 AIOS 之间的执行和等待时间差距随着 Agent 数量增加而扩大。这种不断扩大的差异突显了 AIOS 在高并发工作负载下的可扩展性。

## 5. 相关工作

**操作系统的演化** 从基础批处理系统（IBM, 2010）发展到高级进程管理，包括分时共享（Ritchie & Thompson, 1974）和多任务处理（Hoare, 1974; Engler et al., 1995），支持复杂的任务处理。发展朝着模块化架构推进，具有用于进程调度（Liu & Layland, 1973; Dijkstra, 2002）、内存管理（Denning, 1968; Daley & Dennis, 1968）和文件系统操作（Rosenblum & Ousterhout, 1992; McKusick et al., 1984）的专用组件，提高了系统效率。图形用户界面（GUI）在 Macintosh、Windows 和 GNOME 中的引入增强了用户交互。当前，AI 模型（特别是 LLM）正在从应用层迁移到系统层，为跨应用程序提供标准化服务。

**大语言模型 Agent** 用于解决复杂的规划和推理任务（Xie et al., 2024; Ge et al., 2023a）。单 Agent 可以与数字环境或物理环境交互，可能调用 API（Ge et al., 2023a; Schick et al., 2023; Yao & Narasimhan, 2023; Parisi et al., 2022; Tang et al., 2023; Xie et al., 2024）、浏览网站（Nakano et al., 2022; Deng et al., 2023; Wu et al., 2024）、或执行代码（Zhang et al., 2023; Yang et al.），而物理环境中的 Agent 可能操作物体（Brohan et al., 2023; Fan et al., 2022; Wang et al., 2023a）、进行实验（Boiko et al., 2023; Bran et al., 2023）、或做出可执行的决策（Huang et al., 2022; Xiang et al., 2023）。

**基于 LLM 的多智能体系统（MAS）** 利用多个 Agent 之间的交互来解决问题。多个 Agent 之间的关系可以是合作的（Wang et al., 2023c; Mandi et al., 2023）、竞争的（Chan et al., 2023; Du et al., 2023）、或合作与竞争的混合（Ge et al., 2023b）。在合作的多智能体系统中，每个 Agent 接收并评估其他 Agent 提供的信息，从而共同解决复杂任务，例如角色扮演（Li et al., 2023; Chen et al., 2023; Zhu et al., 2023）、社会模拟（Park et al., 2023）和软件开发（Hong et al., 2023; Qian et al., 2023; Wu et al., 2023; Josifoski et al., 2023）。

## 6. 结论与未来工作

本文介绍了 AIOS，一个通过创新内核为基于 LLM 的 Agent 提供服务的架构，该内核将 LLM 服务和资源从 Agent 应用中隔离出来。补充性的 AIOS SDK 使 Agent 应用能够高效利用内核功能。实验验证确认 AIOS 在标准基准上保持或增强了 Agent 性能，同时显著加速执行时间、提高系统吞吐量，并在不断增加并发 Agent 负载时展现可扩展性。我们预期这项工作将促进未来的创新，以完善和扩展该架构，满足基于 LLM 的 Agent 开发和部署中不断变化的需求。
